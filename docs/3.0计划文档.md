## 0. 执行摘要

3.0 阶段不是继续围绕 PatchCore 增加专用逻辑，也不是试图一次性支持世界上所有科研环境。它要完成的是一组通用、可替换、可扩展的工程能力：

```text
用户目标 + 论文/PDF/TXT + README + GitHub 仓库 + 代码 + 主机事实
                         ↓
                  LLM 生成结构化计划
                         ↓
            AutoAD Core 做安全与权限校验
                         ↓
             Environment Builder 构建环境
                         ↓
               Verifier 确定性验证
                         ↓
              失败结果反馈给 LLM 修订
                         ↓
               Controlled Runner 执行实验
                         ↓
          Metrics / Validity / Reproducibility
                         ↓
                 可追溯阶段报告
```

3.0 完成后，系统必须具备以下能力：

1. LLM 可以把对任意科研项目环境的判断输出为通用 `EnvironmentPlan`；
2. Builder 可以在安全边界内创建或验证隔离环境；
3. Verifier 可以证明环境是否真正可用，而不是让 LLM 主观宣布成功；
4. 失败后可形成结构化诊断，允许 LLM 修订计划；
5. 系统可以执行一个真实、受控、可复现的实验；
6. 两次运行可以比较环境、仓库、数据、命令、资产和指标；
7. 所有关键事实都以 artifact、SHA256、事件和日志保存；
8. PatchCore 仅作为一个 CUDA 异常检测回归案例，不进入通用核心模型；
9. 完成统一 fixture、CI、文档和阶段封板；
10. 3.0 结束后停止扩充底层基础设施，进入真实 Repository Reader。

## 1. 3.0 阶段定位

### 1.1 3.0 要解决的问题

AutoAD-Researcher 当前已经拥有 run、artifact、事件、schema、Harness 和基础 preflight，但这些能力还不足以完成真实科研闭环。3.0 要补齐以下断点：

```text
LLM 的环境判断无法直接安全执行
环境创建过程缺少统一状态和证据
环境成功与否缺少确定性验证
失败后缺少结构化修订入口
模型权重等外部资产没有统一管理
实验命令、日志和指标缺少完整闭环
单次成功不能证明可复现
固定 benchmark 逻辑容易污染通用架构
```

### 1.2 3.0 不解决的问题

本阶段明确不做：

- 不实现全领域、全语言、全包管理器支持；
- 不建立硬编码的 PyTorch/CUDA 版本数据库；
- 不为每个科研仓库编写专用 Builder；
- 不让 LLM 获得无限制 shell；
- 不自动安装 GPU 驱动、修改内核或执行系统级高风险操作；
- 不实现无限重试和无限自主修复；
- 不完成真实 Repository Reader、Paper Reader 或 Idea Generator；
- 不实现大规模多 Agent 自组织；
- 不引入 Temporal、PostgreSQL、对象存储等重型基础设施；
- 不要求 CI 真实安装 CUDA 环境或运行 GPU benchmark。

### 1.3 阶段退出定义

只有同时满足以下条件，3.0 才能结束：

## 4. 统一设计原则

### 4.1 LLM 与 Core 的职责边界

LLM 负责：

- 阅读材料；
- 推断环境类型；
- 选择 Python、包管理器、依赖版本和安装策略；
- 识别源码中未声明依赖；
- 决定是否复用已有环境；
- 设计验证步骤；
- 根据错误修订计划。

AutoAD Core 负责：

- 校验计划格式；
- 限制路径和权限；
- 控制网络阶段；
- 执行结构化命令；
- 保存 stdout、stderr、退出码和时间；
- 生成环境快照；
- 确定性判断验证是否通过；
- 阻止越权、污染和虚假成功。

### 4.2 不建立环境知识数据库

Core 不维护：

- GPU 型号到 CUDA 版本映射；
- PyTorch 与 torchvision 版本矩阵；
- 各仓库安装指令库；
- 各论文代码专属依赖修复规则。

这些信息由 LLM 基于材料和必要时的官方文档推理。

### 4.3 Adapter 而不是项目专用 Builder

Builder 使用 adapter：

```text
UvVenvAdapter
PipVenvAdapter
ExistingPythonAdapter
CondaAdapter（本阶段可完成合同和测试，真实运行按机器条件）
```

未来可以增加：

```text
DockerAdapter
RenvAdapter
NodeAdapter
JuliaAdapter
```

无需修改 EnvironmentPlan 主合同。

### 4.4 Artifact-first

任何成功或失败都必须产生 artifact：

```text
plan.json
plan_policy_report.json
step_results.json
stdout/stderr logs
environment_snapshot.json
validation_report.json
build_result.json
revision.json（如有）
```

### 4.5 有限修复

环境修复最多允许：

```text
初始计划 1 次
修订计划最多 2 次
```

超过限制后停止并生成失败报告，不允许无限重试。

### 5.2.1 EnvironmentPlan

```python
class EnvironmentPlan(BaseModel):
    schema_version: Literal[1]
    plan_id: str
    run_id: str
    revision: int
    parent_plan_id: str | None

    target: EnvironmentTarget
    evidence: list[EvidenceReference]
    assumptions: list[PlanAssumption]
    build_steps: list[CommandStep]
    validation_steps: list[ValidationStep]
    permissions: EnvironmentPermissions

    created_by: Literal["llm", "fixture", "user"]
```

### 5.2.2 EnvironmentTarget

```python
class EnvironmentTarget(BaseModel):
    kind: Literal[
        "python_uv_venv",
        "python_pip_venv",
        "existing_python",
        "conda",
    ]
    environment_path: str | None
    runtime_requirements: dict[str, str]
    repository_path: str | None
```

`runtime_requirements` 允许 LLM表达：

```yaml
python: "3.11"
platform: linux_x86_64
accelerator: cuda
```

Core 不解释 accelerator 兼容性，只把它作为后续验证目标。

### 5.2.3 EvidenceReference

```python
class EvidenceReference(BaseModel):
    source_type: Literal[
        "user",
        "repository",
        "document",
        "source_code",
        "host",
        "previous_error",
    ]
    path_or_id: str
    claim: str
    sha256: str | None
```

要求每个计划至少有一个 evidence。真实 LLM 计划不得只给命令，不给依据。

### 5.2.4 PlanAssumption

```python
class PlanAssumption(BaseModel):
    assumption_id: str
    statement: str
    risk: Literal["low", "medium", "high"]
    validation_id: str | None
```

高风险 assumption 必须有验证项或人工审批。

### 5.2.5 CommandStep

```python
class CommandStep(BaseModel):
    step_id: str
    program: str
    args: list[str]
    cwd: str
    environment: dict[str, str]
    timeout_seconds: int
    network: bool
    modifies_repository: bool
    requires_approval: bool
```

强制：

```text
shell=False
program 不能为空
args 不能包含 NUL
cwd 必须是安全相对路径或由 workspace resolver 解析
环境变量只允许白名单
```

### 5.2.6 ValidationStep

第一版支持：

```text

file_exists

gpu_compute
project_smoke
```

模型：

```python
class ValidationStep(BaseModel):
    validation_id: str
    kind: ValidationKind
    parameters: dict[str, Any]
    required: bool
    timeout_seconds: int
    network: Literal[False]
```

验证默认禁止联网。

### 5.2.7 EnvironmentPermissions

```python
class EnvironmentPermissions(BaseModel):
    network_during_build: bool
    network_during_validation: Literal[False]
    allow_system_package_install: bool
    allow_repository_modification: bool
    allow_global_environment_mutation: bool
    max_revision_count: int
```

MVP 默认：

```yaml
allow_system_package_install: false
allow_repository_modification: false
allow_global_environment_mutation: false
max_revision_count: 2
```

## 5.3 Policy Gate

`validate_environment_plan_policy()` 负责：

- 环境目录位于 `workspace/envs`；
- repo 位于 `workspace/repos`；
- output 位于 `runs/{run_id}`；
- 禁止 `sudo`、`su`、shell pipe、重定向和 `curl | bash`；
- 禁止写入 `/usr`、`/opt`、用户 HOME 配置；
- 禁止删除工作区外路径；
- system package install 必须审批；
- 修改 repo 必须审批；
- build 与 validation 网络权限一致；
- step_id 和 validation_id 唯一；
- 至少一个 required validation。

## 5.4 LLM 输出接入

3.0 只完成 LLM 输出合同、fixture 和受控消费链路，不接入真实 Repository Reader，也不要求真实模型自动生成计划。真实的“读取任意仓库后自动生成 EnvironmentPlan”属于 Step 3.1。3.0 只提供：

```text
load_environment_plan()
validate_environment_plan()
write_environment_plan()
```

以及三份 fixture：

```text
fixtures/environment_plans/python_cpu_uv.yaml
fixtures/environment_plans/python_cuda_uv.yaml
fixtures/environment_plans/existing_python.yaml
```

PatchCore 只可作为 `python_cuda_uv.yaml` 的实例内容之一。

## 7.2 Verifier 注册表

```python
    "runtime_version": RuntimeVersionVerifier,
    "package_inventory": PackageInventoryVerifier,
    "python_import": PythonImportVerifier,
    "command": CommandVerifier,
    "file_exists": FileExistsVerifier,
    "repository_clean": RepositoryCleanVerifier,
    "gpu_available": GpuAvailableVerifier,
    "gpu_compute": GpuComputeVerifier,
    "project_smoke": ProjectSmokeVerifier,
}
```

### 通用验证

- Runtime 版本；
- Distribution inventory；
- 模块 import；
- CLI `--help`；
- 文件存在；
- repo 前后 clean；
- 指定命令 exit code；
- 最小项目 smoke。

### GPU 验证

GPU verifier 只执行 plan 声明的验证：

```text
设备可见
目标 index 存在
最小 tensor 运算
可选 runtime version 输出
```

它不决定应该使用哪个 CUDA 版本。

## 7.3 ValidationReport

```python
class ValidationReport(BaseModel):
    status: Literal["passed", "failed"]
    results: list[ValidationResult]
    required_passed: int
    required_total: int
    report_sha256: str
```

单项结果：

```python
class ValidationResult(BaseModel):
    validation_id: str
    kind: str
    status: Literal["passed", "failed", "skipped"]
    code: str
    message: str
    observed: dict[str, Any]
    artifact_paths: list[str]
```

`message` 不包含密钥和本机私密路径。

## 7.4 LLM 修订输入

系统生成：

```json
{
  "original_plan": "environment/plan.json",
  "failed_step": "install_dependencies",
  "failure_code": "ENV_COMMAND_FAILED",
  "stderr_summary": "No matching distribution found...",
  "failed_validations": ["project_import"],
  "suggested_evidence": [
    "repository/pyproject.toml",
    "repository/src/model.py",
    "environment/logs/install.stderr.log"
  ]
}
```

注意：`suggested_evidence` 只是提示 LLM 查看什么，不给具体修复答案。

## 7.5 EnvironmentPlanRevision

```python
class EnvironmentPlanRevision(BaseModel):
    schema_version: Literal[1]
    parent_plan_id: str
    revision: int
    reason: str
    evidence: list[EvidenceReference]
    replacement_plan: EnvironmentPlan
```

不做复杂 patch DSL。修订直接生成完整新 plan，并保留父子关系。

## 7.6 修订循环控制

```text
revision 0 → build
失败 → revision 1 → build
失败 → revision 2 → build
再次失败 → stop
```

每次必须创建新的 build 目录，不覆盖旧证据。

## 7.7 三类真实/半真实 fixture

### Fixture A：CPU Python 项目

- 小型本地 package；
- uv 或 pip；
- 无 GPU；

- CI 只运行 schema 和 fake runner；
- 验证依赖、CUDA compute 和项目 import；
- 权重不在此工作包加载。

### Fixture C：Existing Environment

- 指定当前测试 Python；
- 禁止安装；
- 只做 runtime、package、import 验证；
- 证明 Builder 不强制重建所有环境。

可选 Fixture D：Conda 命令生成/本机 smoke。

# 9. 工作包 3.0D-2：Controlled Experiment Runner

## 9.1 目标

```text

```

所有输入都通过 SHA 引用。

## 9.3 ExperimentCommandPlan

复用或扩展已有 `BenchmarkCommandSpec`，但名称逐渐通用化：

```python
class ExperimentCommandPlan(BaseModel):
    schema_version: Literal[1]
    command_id: str
    program: str
    args: list[str]
    cwd: str
    environment: dict[str, str]
    timeout_seconds: int
    network: Literal[False]
    expected_outputs: list[str]
```

不使用 shell 字符串。

## 9.4 Runner 行为

执行前：

- preflight 全通过；
- repo clean；
- environment SHA 匹配；
- dataset manifest SHA 匹配；
- asset manifest SHA 匹配；
- command SHA 固定；
- output dir 不存在；
- execution network=false。

执行中：

- stdout/stderr 流式写文件；
- 记录开始和结束；
- 资源超时；
- 可选 GPU telemetry；
- 不覆盖旧 attempt；
- 不修改 protected paths。

执行后：

- exit code；
- repo fingerprint before/after；
- output manifest；
- raw metrics source；
- execution result。

## 9.5 失败分类

```text
preflight_failed
execution_failed
metric_parse_failed
invalid_repository_mutation
success
```

不得把 metric parse failure 伪装为 execution failure，也不得因无指标而宣布实验成功。

## 9.6 测试

- 输出目录已存在；
- network=true 拒绝；
- command hash 不一致；
- timeout；
- 非零退出；
- repo mutation；
- protected file mutation；
- stdout/stderr 保存；

只提交精简证据到 Git。

## 18.4 回归原则

每发现一个真实 bug，增加一个最小 fixture。禁止只修代码不加回归测试。

# 19. 风险与控制

## 风险 1：EnvironmentPlan 变成过度复杂 DSL

控制：

- 第一版只支持四种 target；
- validation kind 控制在九类；
- 修订生成完整 plan，不设计复杂 patch 语言；
- 不增加项目专属字段。

## 风险 2：Builder 开始自行做技术决策

控制：

- Adapter 只能翻译 plan；
- 版本和 index 必须来自 plan；
- Builder 不自动升级、降级或换源；
- 任何 fallback 由新 revision plan 表达。

## 风险 3：LLM 直接输出危险命令

控制：

- 结构化 CommandStep；
- Policy Gate；
- shell=False；
- 高风险动作审批；
- workspace 边界。

## 风险 4：PatchCore 再次污染通用设计

控制：

- 核心模型测试不引用 PatchCore；
- 至少两个非 PatchCore fixture；
- PatchCore 代码仅存在于 `configs/benchmarks` 和 `evidence` 实例层。

## 风险 5：3.0 无止境扩张

控制：

- 以本文件退出条件为唯一范围；
- 3.0E 后停止新增底层抽象；
- Docker、R、Node、Julia、Temporal 推迟；
- 进入 3.1 后由真实 Reader 需求驱动后续扩展。

# 21. 3.0 完成后的正式边界

3.0 完成后，AutoAD-Researcher 应被描述为：

> 已具备通用环境计划、受控环境构建、确定性环境验证、外部资产准备、不可变实验执行、真实指标解析、科研有效性检查和双跑复现能力的科研闭环底座。

但仍不应被描述为：

- 能自动理解任意仓库；
- 能自动提出可靠科研创新；
- 能全自动修改任意代码；
- 能支持所有语言和计算平台；
- 能无人监督地完成科研。

3.0 的直接输出是为 Step 3.1 提供可靠执行底座。

# 22. Step 3.1 入口条件

完成 3.0 后，下一阶段正式开始：

```text
Step 3.1：真实 Repository Reader
```

Repository Reader 的输入：

- 用户仓库 URL 或本地路径；
- 用户目标；
- 可选论文/PDF/TXT；
- 主机能力；
- AutoAD 安全策略。

Repository Reader 的输出将直接服务于本阶段建立的能力：

```text
repository_summary.json
entrypoints.json
dependency_evidence.json
modifiable_paths.json
evaluation_contract.json
environment_context.json
```

随后由 LLM 生成 `EnvironmentPlan`，而不是把仓库内容硬编码进 Builder。

# 23. 最终执行原则

整个 3.0 阶段必须坚持：

```text
LLM 负责推理，不负责宣告事实
Plan 负责表达，不负责执行
Builder 负责执行，不负责决策
Verifier 负责证明，不负责修复
Revision 负责修正，不允许无限循环
Runner 负责实验，不允许改变科研合同
Artifact 负责记录，不只保存成功
PatchCore 负责回归，不代表产品边界
```

当第 20 节验收矩阵全部通过后，记录：

```text
Step 3.0 accepted and sealed
Sealed commit: <final commit sha>
Next: Step 3.1 Repository Reader
```
