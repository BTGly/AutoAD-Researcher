> 文档状态：修订稿 v1.2（v1.1 major_revision_required，本轮重点修复 P0-1 ~ P0-8）
> 更新日期：2026-06-17
> 阶段名称：**Idea & Transfer Design / 科研 Idea 确认与迁移方案设计**
> 上游输入：`stable research_context.json` + `idea_transfer_handoff.json`（来自 3.2 Context Layer，或经 3.3 ContextPatch 修复后）
> 下游阶段：Step 3.5 Multi-variant Experiment Planner
> 3.4 本身：**不执行代码、不运行实验、不生成文件级 PatchPlan、不重复读取论文/仓库源码**

# 0. 阶段定义

Step 3.4 合并原计划中的两个阶段：
- 原 3.4 Single Idea Confirmation
- 原 3.5 Transferability Judge

合并后的 3.4 完成一件事：

```
从 stable 研究上下文出发
  → 用户确认一个科研 Idea（每个 run 恰好一个）
  → 生成 1–3 个架构级 Variant
  → 对每个 Variant 分别判断迁移兼容性
  → 确定性过滤 non-viable Variant
  → 用户选择 Variant 并显式接受风险
  → 确定性验证
  → 交给 3.5 Experiment Planner
```

## 0.1 核心不变量

```
一个 run → 一个 confirmed Idea
一个 Idea → 1–3 个 implementation Variant
Variant 可以选一个或多个
Idea 不得多选
non-viable → 当前 run 终止，新 Idea 使用 child run
```

## 0.2 职责边界

3.4 做：
- 路由 Idea 来源（用户 / 论文候选）
- 将用户确认的一个 Idea 转为 `IdeaContract`
- 消费 3.1/3.2 产出，缺失时发出 reanalysis request
- 生成架构级 Variant
- 对每个 Variant 分别执行 CompatibilityAnalyzer
- 确定性过滤 non-viable Variant
- 用户选择 Variant + 显式接受风险
- 确定性验证

3.4 不做：
- 重新读取论文 PDF 或解析方法组件（3.2 已做）
- 重新扫描仓库源码或提取 tensor position（3.1 已做）
- 生成文件级 PatchPlan（3.6）
- 生成开放式科研 Idea（超出 MVP）
- 执行代码或运行实验
- 替用户静默确认任何决策

# 1. 核心流程

## 1.1 流程总览

```
              stable research_context.json
              + idea_transfer_handoff.json
              + paper_idea_sources.json          (← 3.2)
              + baseline_architecture_contract.json  (← 3.1，如缺失则发 reanalysis)
                                │
                    ┌───────────┴───────────┐
                    ▼                       ▼
         路径 A: 用户提供 Idea      路径 B: 论文候选
         (source=user_provided)   (source=paper_grounded)
                    │                       │
                    ▼                       ▼
              IdeaSourceRouter        IdeaSourceResolver
           (验证→确认→写入)        (从 3.2 候选列表展示)
                    │                       │
                    └───────────┬───────────┘
                                ▼
                    用户明确确认一个 Idea
                    (confirm / select_one)
                                │
                    无合法 idea source 时
                    → blocked_without_confirmable_idea
                    不静默生成、不降级

                                ▼

                                │
                                ▼

              (paper mechanism ↔ baseline contract 对齐)
                                │
                      baseline_architecture_contract 缺失
                      → repository_reanalysis_request
                      paper evidence 不足
                      → paper_reanalysis_request
                                │
                                ▼
                      VariantGenerator
              (同一 Idea 下 1–3 个架构级 Variant)
                                │
                                ▼
               逐 Variant CompatibilityAnalyzer
          (每个 Variant 独立分析 9 个维度)
                                │
               ┌────────────────┼────────────────┐
               ▼                ▼                ▼
          全部 viable    viable_with        non_viable
                       _conditions       (过滤掉)
               │                │
               │        needs_reanalysis
               │        (过滤掉，发 reanalysis)
               │                │
               └────────────────┘
                        │
                        ▼
                  VariantSelector
         (展示 viable + viable_with_conditions Variants
          用户明确选择 + 显式接受 medium/high risks)
                        │
                        ▼
                DeterministicValidator
           (八类不变量检查)
                        │
                        ▼
       Step 3.5 Multi-variant Experiment Planner
```

## 1.2 关键：Compatibility 以 Variant 为单位

v1.1 的错误是先对整个 Idea 做一次全局兼容性分析，再生成 Variant。这导致：
- Variant B/C（纯 inference 修改）错误继承了 Variant A 的训练约束
- 不同 Variant 的训练/资源/状态兼容性显然不同

v1.2 改为：先对齐架构、生成 Variant，然后对每个 Variant 独立分析兼容性。

## 1.3 上游 contract 缺失时的处理

3.1 当前正式产出中不包含 `baseline_architecture_contract.json`。3.4 需要两种路径：

**路径 1**：3.1 已有 `baseline_architecture_contract.json` → 直接消费。

**路径 2**：3.1 未产出该 artifact → 3.4 发出 `repository_reanalysis_request.json`，请求 3.1 定向补充读取接口、tensor、phase 和 hook。返回后 3.4 重新执行 ArchitectureAligner 及之后的所有步骤。

`baseline_architecture_contract.json` 的 schema（由本次 3.4 定义，但它属于 3.1 的产出）见第 5 节。

# 4. IdeaContractNormalizer（公共）

## 4.1 IdeaContract — 判别联合

source 不同，required 字段不同：

```python
class DerivedClaim(BaseModel):
    """3.4 产生的分析性声明，不是事实。"""
    value: str
    status: Literal["inferred"] = "inferred"
    producer_stage: Literal["3.4"] = "3.4"
    supporting_evidence_ids: list[str]
    assumptions: list[str]

class ConstraintRef(BaseModel):
    description: str
    source: Literal["user_provided", "paper_derived", "baseline_derived"]
    evidence_ids: list[str]

class UserProvidedIdeaContract(BaseModel):
    source: Literal["user_provided"] = "user_provided"
    user_description: str
    user_evidence_ids: list[str]
    mechanism_hypothesis: DerivedClaim
    transfer_relevance: DerivedClaim

class PaperGroundedIdeaContract(BaseModel):
    source: Literal["paper_grounded"] = "paper_grounded"
    paper_idea_source_id: str
    paper_mechanism_summary: str
    paper_evidence_ids: list[str]
    original_mechanism_rationale: DerivedClaim  # 机制在原论文中为什么有效
    transfer_relevance: DerivedClaim  # 为什么对 baseline 有价值

IdeaContractCore = Annotated[
    Union[UserProvidedIdeaContract, PaperGroundedIdeaContract],
    Field(discriminator="source"),
]

class IdeaContract(BaseModel):
    idea_id: str

    # 判别联合
    idea_source: IdeaContractCore

    # 共享
    must_preserve_behaviors: list[ConstraintRef]

    # 确认
    confirmation_status: Literal["confirmed", "pending"]
    confirmed_by_user_at: datetime | None
    confirmation_evidence_id: str | None

    class Config:
        extra = "forbid"
```

关键约束：
- `source=user_provided` 时不需要 `mechanism_why_in_paper`；由 `mechanism_hypothesis` 代替
- 所有 DerivedClaim.status 必须为 `inferred`，producer_stage 必须为 `3.4`
- `must_preserve_behaviors` 中每个条目必须有 `source` 和 `evidence_ids`

### 对齐后的路由

| 有 entry 状态 | 处理 |
|-------------|------|
| INSUFFICIENT_PAPER_EVIDENCE | paper_reanalysis_request |
| INSUFFICIENT_REPOSITORY_EVIDENCE | repository_reanalysis_request |

```

→ 3.4 重新从 ArchitectureAligner 开始
→ 所有下游 artifact（alignment / variants / analysis / selection / handoff）失效重算
```

# 7. VariantGenerator（公共）

## 7.1 位置变化

v1.2 将 VariantGenerator 移到 CompatibilityAnalyzer 之前。先产生 Variant，再逐 Variant 分析。

## 7.2 Variant 模型（架构级）

```python
class InterfaceChangeDescription(BaseModel):

    variant_id: str

    adapter_required: bool

    regime_change: RegimeChange | None

    # 状态/记忆变更
    state_change: StateChangeDescription | None

    # 约束
    new_dependencies: list[str]

    # 预期和风险
    expected_behavior_rationale: str
    risk_level: Literal["low", "medium", "high"]

```

### 不包含（留给 3.6）

```
files_to_modify / files_to_add / 函数名 / 代码行数 / final_patch.diff
```

### 关键变化

- 不再使用单一的 `target_hook: str`，改为 `hook_bindings: list[HookBinding]`
- 不再包含 `touches_protected_path: bool`（由 DeterministicValidator 从 hook contract 确定性计算）
- 不再强制"A 最激进、C 最保守"

## 7.3 生成规则

```
1. 同一 Idea 下 1–3 个 Variant
2. 每个 Variant 的 hook_bindings 引用 ModificationHook.hook_id
3. Variant 之间的差异维度（至少满足一条）：
   a. 不同的 hook binding 组合
   b. 不同的 adapter 策略
   c. 不同的 regime_change
   d. 不同的 state_change
4. Variant 不自行声明是否触碰 protected path
```

# 8. 逐 Variant CompatibilityAnalyzer（公共）

## 8.1 设计原则

- 不使用伪精确的 1-10 数值评分
- 每个 Variant 独立分析所有 9 个维度
- 最终结论由确定性规则派生
- retry loop 只能用于 schema repair / evidence reference fix / 内部矛盾消除

## 8.2 兼容性维度与状态

```python
class CompatibilityDimension(str, Enum):
    SEMANTIC = "semantic"
    INPUT = "input"
    OUTPUT = "output"
    SHAPE = "shape"
    TRAINING = "training"
    DATA = "data"
    LABEL = "label"
    EVALUATION = "evaluation"
    RESOURCE = "resource"

class CompatibilityStatus(str, Enum):
    COMPATIBLE = "compatible"
    COMPATIBLE_WITH_ADAPTER = "compatible_with_adapter"
    REQUIRES_REGIME_CHANGE = "requires_regime_change"
    INCOMPATIBLE = "incompatible"
    INSUFFICIENT_EVIDENCE = "insufficient_evidence"
```

### 维度判定模型

```python
class DimensionJudgment(BaseModel):
    variant_id: str
    dimension: CompatibilityDimension
    status: CompatibilityStatus
    blocking: bool
    reasoning: str

    # 证据
    idea_contract_evidence_ids: list[str]
    paper_evidence_ids: list[str]
    repository_evidence_ids: list[str]

    # 非 compatible 时
    assumptions: list[str]
    required_changes: list[str]
    required_adapter: str | None
    risk: Literal["low", "medium", "high"] | None
```

## 8.3 每 Variant 整体状态的确定性派生

```python
CORE_DIMENSIONS = {
    CompatibilityDimension.INPUT,
    CompatibilityDimension.OUTPUT,
    CompatibilityDimension.SHAPE,
    CompatibilityDimension.TRAINING,
}

def derive_variant_status(
    judgments: list[DimensionJudgment],
    constraints: list[TransferConstraint],
) -> TransferStatus:
    # 规则 1: 核心维度 insufficient_evidence → needs_reanalysis（优先于一切）
    if any(
        j.status == CompatibilityStatus.INSUFFICIENT_EVIDENCE
        and j.dimension in CORE_DIMENSIONS
        for j in judgments
    ):
        return TransferStatus.NEEDS_REANALYSIS

    # 规则 2: 任何维度 incompatible → non_viable
    if any(j.status == CompatibilityStatus.INCOMPATIBLE for j in judgments):
        return TransferStatus.NON_VIABLE

    # 规则 3: requires_regime_change 但违反用户已确认约束 → non_viable
    if any(
        j.status == CompatibilityStatus.REQUIRES_REGIME_CHANGE
        and violates_confirmed_constraint(j, constraints)
        for j in judgments
    ):
        return TransferStatus.NON_VIABLE

    # 规则 4: 有 adapter / regime change / 非核心 insufficient_evidence → viable_with_conditions
    if any(
        j.status in {
            CompatibilityStatus.COMPATIBLE_WITH_ADAPTER,
            CompatibilityStatus.REQUIRES_REGIME_CHANGE,
            CompatibilityStatus.INSUFFICIENT_EVIDENCE,
        }
        for j in judgments
    ):
        return TransferStatus.VIABLE_WITH_CONDITIONS

    # 规则 5: 全部 compatible → viable
    return TransferStatus.VIABLE
```

### Validator 强制规则

```
INCOMPATIBLE → blocking 必须为 true
核心维度 INSUFFICIENT_EVIDENCE → variant 整体 needs_reanalysis
非核心 INSUFFICIENT_EVIDENCE → 必须在 UnresolvedDimension 中被分类处理
```

## 8.4 输出结构

```python
class VariantTransferAnalysis(BaseModel):
    variant_id: str
    dimensions: list[DimensionJudgment]
    overall_status: TransferStatus
    constraints: list[TransferConstraint]
    unresolved_dimensions: list[UnresolvedDimension]

class IdeaTransferAnalysis(BaseModel):
    idea_id: str
    variant_analyses: dict[str, VariantTransferAnalysis]
    viable_variant_ids: list[str]
    conditional_variant_ids: list[str]
    non_viable_variant_ids: list[str]
    needs_reanalysis_variant_ids: list[str]
```

### Retry Loop 的合法用途

```
允许:
- schema 缺失字段 → 补充
- judgment 内部矛盾 → 修正
- evidence_ids 引用不存在 → 修正

不允许:
- 对同一事实重新判断
- 把 INCOMPATIBLE 改成 COMPATIBLE
- 超过 2 次重试
```

# 9. 确定性过滤

CompatibilityAnalyzer 完成后，过滤 Variant：

```
viable → 直接展示给用户
viable_with_conditions → 展示给用户（附带条件说明）
non_viable → 过滤，不展示，记录 rejected_variants
needs_reanalysis → 过滤，发出 reanalysis request
```

non_viable 后：
- 当前 run 的目标 Idea 无法实施
- 不回到 Idea 选择（违反 one-run-one-Idea）
- run 终止为 non_viable
- 如果用户希望尝试新 Idea → 发出 `spawn_child_run_request.json`

```
  插入 hook: embedding_before
  训练兼容性: compatible
  风险: low

以下 Variant 因兼容性不通过已被过滤:
  -（当前无限）

请选择你想实现的 Variant（可以多选）。
```

## 10.2 风险接受

`risk_level ∈ {medium, high}` 的 Variant 在被选中时需要用户显式接受：

```
Variant A 被标记为 medium 风险：需要新增拟合阶段。

是否接受此风险？
A. 接受 —— Variant A 进入实现队列
B. 拒绝 —— 但 Variant A 不进入队列
C. 了解更多 —— 查看 risk detail
```

接受的 risk 必须记录为 `AcceptedRisk`：

```python
class AcceptedRisk(BaseModel):
    risk_id: str
    variant_id: str
    dimension: CompatibilityDimension
    description: str
    severity: Literal["low", "medium", "high"]

    accepted_by_user: bool
    user_decision_evidence_id: str | None  # UserInputEvidenceRef，必须非空
    accepted_at: datetime | None

    evidence_ids: list[str]
```

Validator 强制：`severity ∈ {medium, high}` 且 `accepted_by_user=true` 时 `user_decision_evidence_id` 必须非空。

## 10.3 用户沉默

```
可以推荐（recommended_variant_ids）
不能自动选择（不能写入 selected_variants）
不能把"你看着办"视为无限授权
沉默 → 保持 blocked，不进入下一步
```

# 11. UnresolvedDimension 分类

## 11.1 分类规则（确定性）

```python
class ResolutionClass(str, Enum):
    DESIGN_BLOCKING = "design_blocking"
    EXPERIMENT_RESOLVABLE = "experiment_resolvable"
    NONBLOCKING_WARNING = "nonblocking_warning"

class UnresolvedDimension(BaseModel):
    variant_id: str
    dimension: CompatibilityDimension
    status: CompatibilityStatus
    classification: ResolutionClass
    resolution_reason: str

    # experiment_resolvable 必须提供
    verification_target: str | None
    acceptance_criterion: str | None

    evidence_ids: list[str]
    classified_by_rule_id: str
```

### 分类规则表

```
INPUT/OUTPUT/SHAPE 核心接口未知 → needs_reanalysis（不进入此表）
INPUT/OUTPUT/SHAPE 已知但兼容性不确定 → design_blocking
LABEL 冲突且 baseline 无对应标签 → design_blocking
TRAINING requires_regime_change 且违反用户约束 → design_blocking
资源消耗未知但可通过 smoke benchmark 测量 → experiment_resolvable
非关键日志格式差异 → nonblocking_warning
```

`experiment_resolvable` 必须提供 `verification_target` 和 `acceptance_criterion`，否则 3.5 无法将其转为实验验证步骤。

design_blocking 的 unresolved dimension 不得进入 3.5 handoff。

# 12. DeterministicValidator

## 12.1 八类不变量

### 12.1.1 Idea 不变量

```
1. 一个 run 恰好一个 confirmed Idea
2. Idea 有明确 source evidence（UserInputEvidenceRef 或 PaperEvidenceRef）
3. confirmation_status = confirmed
4. source=paper_grounded 时 paper_evidence_ids 全部在 Evidence Index 中存在
```

### 12.1.2 Baseline 不变量

```
1. baseline_architecture_contract 来自固定 repository_source_id 和 commit
2. 所有 ModificationHook.hook_id 唯一
3. 所有 TensorSpec.evidence_ids 在 Evidence Index 中存在
4. 所有 ExecutionPhaseContract.phase 唯一
```

### 12.1.3 Variant 不变量

```
1. variant 数量 ∈ [1, 3]
2. 所有 variant 的 idea_id 指向同一个 IdeaContract
3. 每个 HookBinding.hook_id 存在于 baseline_architecture_contract.modifiable_hooks
4. 不包含文件级修改（files_to_modify、new_components 等字段）
5. Variant 之间至少满足一条差异维度规则
```

### 12.1.4 Variant-Hook 不变量

```
1. 每个 HookBinding 引用的 ModificationHook:
   a. allowed_for_transfer_design=true → binding 合法
   b. path_classification=protected_candidate → binding 必须标记为 blocked
   c. module_path 必须在 3.1 的 modifiable_paths.json 中存在
2. protected path 触碰由 hook contract 确定性计算，不由 Variant 自报
```

### 12.1.5 Compatibility 不变量

```
1. 每个 variant 有完整的 9 维 DimensionJudgment
2. INCOMPATIBLE → blocking 必须为 true
3. 核心维度 INSUFFICIENT_EVIDENCE → variant overall_status=needs_reanalysis
4. overall_status 严格由 derive_variant_status() 派生
5. non_viable variant 不进入 VariantSelector
```

### 12.1.6 Dimension Judgment 不变量

```
1. blocking=true 且 status ∈ {INCOMPATIBLE, INSUFFICIENT_EVIDENCE} → 必须有 reasoning
2. COMPATIBLE 不需要 required_changes 和 required_adapter
3. COMPATIBLE_WITH_ADAPTER → required_adapter 非空
```

### 12.1.7 Risk 不变量

```
1. AcceptedRisk.severity ∈ {medium, high} + accepted_by_user=true → user_decision_evidence_id 非空
2. 被选中的 Variant 的所有 AcceptedRisk 必须有用户证据
3. risk_level ∈ {medium, high} 的 Variant 被选中时，对应 AcceptedRisk 必须存在
```

### 12.1.8 Handoff 不变量

```
1. source_context_id/version/SHA 匹配上游
2. idea_contract_sha256 匹配
3. variant_selection_sha256 匹配
4. 不含 design_blocking unresolved dimension
5. 不含 non_viable variant
6. next_stage=3.5_multi_variant_experiment_planner
```

## 12.2 Validator 行为

```
确定性代码，不发起 LLM call
验证失败 → 报告具体违规项
schema error → 可由 llm_call:artifact_repair 修复（不是 llm_call:transfer_validator）
修复后重新验证
```

Permission Profile 中删除 `llm_call:transfer_validator`，替换为 `llm_call:artifact_repair`。

# 13. Resume Fingerprint

当上游 artifact 变化时，下游必须失效重算：

```python
class TransferResumeFingerprint(BaseModel):
    research_context_sha256: str
    baseline_architecture_contract_sha256: str
    paper_idea_sources_sha256: str
    method_components_sha256: str
    idea_contract_sha256: str
    skill_sha256: str
    policy_sha256: str
```

变化追踪：

```
research_context 变化 → 全部失效
baseline_architecture_contract 变化 → alignment + variants + analysis + selection 失效
paper_idea_sources 变化 → idea_contract + alignment + ... 失效
method_components 变化 → alignment + ... 失效
idea_contract 变化 → variants + analysis + selection 失效
```

# 15. 状态机

```
idle
  → waiting_for_baseline_contract     (3.1 未产出 baseline_architecture_contract)
  → waiting_for_idea_source            (无合法 idea source)
  → idea_source_routed                 (路径 A)
  → candidates_resolved                (路径 B)
  → reanalysis_requested               (PAPER 或 REPOSITORY)

idea_source_routed / candidates_resolved
  → waiting_for_idea_confirmation

waiting_for_idea_confirmation
  → idea_confirmed
  → blocked_no_confirmable_idea

idea_confirmed
  → idea_contract_normalized
  → architecture_aligned
  → variants_generated
  → compatibility_analyzed

compatibility_analyzed
  → variants_filtered                  (过滤 non_viable / needs_reanalysis)
  → all_non_viable                     (当前 run 终止)

variants_filtered
  → waiting_for_variant_selection

waiting_for_variant_selection
  → variants_selected
  → blocked_no_variant_selected

variants_selected
  → risks_pending                      (有 medium/high risk 未被接受)
  → risks_accepted

risks_accepted
  → validated
  → validation_failed

validated
  → handoff_generated
  → ready_for_3_5

waiting_for_baseline_contract
  → reanalysis_requested → architecture_aligned  (3.1 返回后重新开始)

all_non_viable
  → run_terminated_non_viable
  → spawn_child_run_request (如果用户希望尝试新 Idea)
```

# 21. 测试计划

## 21.1 Contract Tests

```
IdeaContract（判别联合：user_provided vs paper_grounded 字段差异）
TensorSpec / ModificationHook / BaselineArchitectureContract
DimensionJudgment / VariantTransferAnalysis / IdeaTransferAnalysis
UnresolvedDimension（分类规则）
AcceptedRisk（mandatory user evidence field）
```

## 21.2 Routing Tests

```
路径 A: user_provided → IdeaSourceRouter
路径 A: paper_grounded → IdeaSourceRouter
路径 A: method 不存在 → typo 检查 → reanalysis 或确认为原创
路径 B: paper_idea_sources 非空 → select_one
路径 B: 无 paper_idea_sources → blocked
```

## 21.3 Reanalysis Tests

```
baseline_architecture_contract 缺失 → repository_reanalysis_request
paper evidence 不足 → paper_reanalysis_request
reanalysis 返回后 contract SHA 变化 → alignment + variants + analysis 失效重算
```

## 21.4 Compatibility 逐 Variant Tests

```
两个 Variant 有不同 training compatibility
Variant A 被正确标记为 requires_regime_change
Variant B 被正确标记为 compatible
INCOMPATIBLE + blocking=true → non_viable
核心 INSUFFICIENT_EVIDENCE → needs_reanalysis（优先于 non_viable）
INCOMPATIBLE + blocking=false → Validator 拒绝
LLM retry 不改变 INCOMPATIBLE 判定
```

## 21.5 Variant-Hook 不变量 Tests

```
LLM 自报 touches_protected_path=false 但 hook 属于 protected → Validator 拒绝
allowed_for_transfer_design=false → Validator 标记 blocked
hook_id 不存在于 contract → Validator 拒绝
```

## 21.6 Risk 不变量 Tests

```
medium risk accepted → user_decision_evidence_id 非空 → 通过
medium risk accepted → user_decision_evidence_id 为空 → Validator 拒绝
选中 Variant 的 AcceptedRisk 缺失 → Validator 拒绝
```

## 21.7 Unresolved Dimension Tests

```
design_blocking dimension 进入 3.5 handoff → Validator 拒绝
experiment_resolvable 缺少 verification_target → 拒绝
unresolved blocker 被 LLM 标成 experiment_resolvable → Validator 拒绝
```

## 21.8 One-Run-One-Idea Tests

```
non_viable → 当前 run 终止
non_viable 后 IdeaContract 有两个 confirmed idea → Validator 拒绝
spawn_child_run_request 格式正确
```

## 21.9 Security Tests

```
DeterministicValidator 不发起任何 LLM call
用户输入不改变 paper evidence
prompt injection 不改变 policy
```
