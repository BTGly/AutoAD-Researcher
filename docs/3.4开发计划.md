# AutoAD-Researcher Step 3.4 开发计划：Idea & Transfer Design

> 文档状态：初稿 v1
> 更新日期：2026-06-17
> 阶段名称：**Idea & Transfer Design / 科研 Idea 确认与迁移方案设计**
> 前置阶段：Step 3.3 Context Repair & Clarification Addendum（输出 `research_context.json` + `idea_transfer_handoff.json`）
> 下一阶段：Step 3.5 Multi-variant Experiment Planner
> 操作范围：**从确认一个科研 Idea 到生成 1-3 个实现 Variant，含用户选择和可行性判断**
> 核心原则：**任何 Variant 都有可追溯的论文证据和 baseline contract 分析。用户不知道做什么时，系统从论文中 extract 候选 Idea 供选择。**

---

# 0. 阶段定义

Step 3.4 合并原计划中的两个阶段：
- 原 3.4 Single Idea Confirmation
- 原 3.5 Transferability Judge

合并后的 3.4 完成一件事：

```
research_context.json
    ↓
确认一个科研 Idea（用户提供 / 论文 extract / 系统生成）
    ↓
论文机制提取 + baseline 架构对齐
    ↓
逐项兼容性检查（semantic / tensor / training / data / label / eval / resource）
    ↓
生成 1–3 个同一 Idea 下的实现 Variant
    ↓
用户确认一个或多个 Variant
    ↓
输出给 3.5 Experiment Planner
```

# 1. 核心流程

## 1.1 流程总览

```
                          research_context.json
                                   │
                   ┌───────────────┼───────────────┐
                   ▼               ▼               ▼
            用户明确指定      论文中有候选      用户无 Idea
            IdeaSource       IdeaSource       idea_source=null
                │               │               │
                ▼               ▼               ▼
           IdeaRouter     IdeaExtractor    IdeaGenerator
          (直接使用)     (从论文提取)     (从论文+baseline
                                          生成候选 Idea)
                │               │               │
                └───────────────┼───────────────┘
                                ▼
                      SingleIdeaConfirm
                   (用户从中确认一个)
                                │
                                ▼
                    MechanismExtractor
                  (提取论文核心机制)
                                │
                                ▼
                   ArchitectureAligner
               (机制 ↔ baseline 架构对齐)
                                │
                                ▼
                    CompatibilityJudge
             (语义 / tensor / training / data /
               label / eval / resource 逐项检查)
                                │
                                ▼
                    VariantGenerator
               (生成 1–3 个实现 Variant)
                                │
                                ▼
                    VariantSelector
               (用户选择一个或多个)
                                │
                                ▼
                idea_transfer_design_handoff.json
                         → Step 3.5
```

## 1.2 三条路径的入口

| 路径 | 触发条件 | 过程 | 用户参与 |
|------|---------|------|---------|
| **A: 用户明确指定** | 用户提供了论文中具体要迁移的 method | IdeaRouter 直接确认 → 跳过 extract/generate → 进入 MechanismExtractor | 确认即可 |
| **B: 论文有候选** | 论文中提到了多个方法/模块，用户没指定要哪一个 | IdeaExtractor 从论文 structure 中提取所有候选 → 展示给用户 | 选择一个 |
| **C: 用户无 Idea** | 用户只有论文和 baseline，不知道该迁移什么 | IdeaGenerator 从论文+baseline 中生成 3-5 个候选 Idea → 展示给用户 | 选择一个或多个 |

# 2. 路径 A：用户明确指定 Idea

## 2.1 触发条件

`idea_transfer_handoff.json` 中 `eligible_idea_sources` 非空，且用户已在 3.3 中确认了目标 method。

或者用户在 3.4 入口直接指定：

```
"把论文里的 Cross-Scale Attention 模块迁移到 PatchCore"
```

## 2.2 IdeaRouter 做什么

```
1. 读取 research_context.json，定位用户指定的 method
2. 验证该 method 确实在论文中存在（paper evidence 交叉验证）
3. 如果论文中存在 → 直接确认，进入 MechanismExtractor
4. 如果论文中不存在 or 不明确 → 降级到路径 C（生成候选）
```

## 2.3 输出（路径 A）

```json
{
  "route": "user_specified",
  "confirmed_idea": {
    "idea_id": "...",
    "source": "user_provided",
    "paper_evidence_ids": ["ev_paper_method_003"],
    "mechanism_label": "Cross-Scale Attention"
  }
}
```

# 3. 路径 B：论文中有候选 Idea

## 3.1 触发条件

`research_context.json` 中有 `method_components`（来自 3.2 Paper Intelligence），但用户没有指定具体要哪个。

## 3.2 IdeaExtractor 做什么

从 3.2 已产出的 `method_components.json` 中提取每个 component 的：

```
1. 名称和论文中的位置（section/page）
2. 一句话机制描述
3. 在论文架构图中的位置（backbone head / neck / post-process / loss）
4. 论文声称的贡献程度（main contribution / auxiliary / ablation only）
5. 输入输出 tensor shape（如果论文给出了）
6. 是否涉及训练参数更新
```

### 核心 Prompt 逻辑

参考 AI-Scientist `generate_ideas.py` 和 `perform_review.py`：

```
SYSTEM: 你是一个论文精读专家。给定一篇已解析为结构化 JSON 的论文，
请从 method_components 中提取每个组件的迁移信息。

对每个 CandidateMethod：
- mechanism_summary: 一句话说清楚这个模块在做什么
- role_in_paper: main_contribution / auxiliary / ablation_only
- input_shape: 如果论文给了，写出；如果没给，标记 unknown
- output_shape: 同上
- has_trainable_params: true / false / unknown
- dependencies_on_other_modules: 是否依赖论文中其他组件
- potential_baseline_hooks: 你推测这个模块可以插入 baseline 的哪个位置
  (backbone_after / embedding_before / post_processing / loss / ...)
- transfer_feasibility_hint: 初步评估——likely_direct / likely_needs_adapter / uncertain
```

输出格式参考 AI-Scientist 的 review JSON——每个 candidate 独立打分：

```json
{
  "candidate_methods": [
    {
      "method_id": "cross_scale_attention",
      "method_name": "Cross-Scale Attention",
      "paper_location": "Section 3.2, Fig. 3",
      "mechanism_summary": "Multi-scale feature fusion via learnable attention weights...",
      "role_in_paper": "main_contribution",
      "input_shape": "BxCxHxW (multi-scale)",
      "output_shape": "BxCxHxW (fused)",
      "has_trainable_params": true,
      "dependencies_on_other_modules": ["backbone feature extractor"],
      "potential_baseline_hooks": ["backbone_after", "embedding_before"],
      "transfer_feasibility_hint": "likely_direct",
      "transfer_feasibility_reason": "Operates on feature maps, compatible with PatchCore's feature extraction stage"
    }
  ],
  "extraction_confidence": "high",
  "paper_evidence_ids": ["ev_page_5_eq_3", "ev_fig_3"]
}
```

## 3.3 用户交互

```
我已经从论文中提取了以下可迁移的候选模块：

1. Cross-Scale Attention (Section 3.2, Fig. 3)
   核心机制：多尺度特征通过可学习 attention 权重融合
   推测插入位置：backbone 输出后、embedding 前
   初步可行性：likely_direct

2. Adaptive Memory Bank (Section 3.3)
   核心机制：在线更新 memory bank，不需要重新计算 coreset
   推测插入位置：替换现有 memory bank
   初步可行性：likely_needs_adapter（需要兼容现有 inference 流程）

3. Asymmetric Distance (Section 4.1)
   核心机制：训练阶段使用不对称距离函数
   推测插入位置：anomaly_score 计算阶段
   初步可行性：likely_direct

你想迁移哪个模块？（可以多选）
```

选项带 consequence：

| 选项 | 后果 |
|------|------|
| 1 Cross-Scale Attention | 可能需要在 backbone 和 embedding 之间插入新模块，需轻微修改 forward |
| 2 Adaptive Memory Bank | 需要替换 memory bank 逻辑，inference 流程可能需要 adapter |
| 3 Asymmetric Distance | 只修改距离计算，影响范围最小 |

# 4. 路径 C：用户无 Idea（系统生成）

## 4.1 触发条件

- 用户提供了论文和 baseline，但没有指定想迁移什么
- 或者用户在 3.3 中明确说"你先看看有什么值得做的"
- 或者路径 A 降级（用户指定的 method 在论文中找不到）

## 4.2 IdeaGenerator 做什么

这是 3.4 中最复杂的路径。本质上是 **AI-Scientist `generate_ideas.py` 的逆向版本**——不是从零生成 idea，而是从已有论文中 **发掘** paper→baseline 的迁移 idea。

### 输入

```
1. paper_summary.json（3.2 产出）
2. paper_method_components.json（3.2 产出）
3. baseline_contract.json（3.4 自身提取）
4. known_transfer_patterns.json（历史成功/失败迁移记录）
5. 用户偏好（3.3 产出）
```

### baseline_contract.json 提取

在生成 Idea 之前，必须先从 baseline 代码中提取 contract。参考 anomalib 源码分析：

```python
# 从 baseline 代码中自动提取
class BaselineContract(BaseModel):
    model_name: str
    training: Literal["no_gradient", "gradient_based", "mixed"]
    forward_input_shape: str  # e.g. "[B, 3, 224, 224]"
    forward_output_type: str  # e.g. "InferenceBatch"
    critical_middle_tensors: list[MiddleTensor]
    shared_components: list[str]  # e.g. ["TimmFeatureExtractor", "KCenterGreedy"]
    modifiable_hooks: list[ModificationHook]

class MiddleTensor(BaseModel):
    name: str
    shape: str
    location: str  # e.g. "after backbone, before pooling"
    source_file: str

class ModificationHook(BaseModel):
    hook_name: str  # e.g. "backbone_after", "embedding_before"
    description: str
    typical_modification: str
    risk: Literal["low", "medium", "high"]
```

例如 PatchCore 的 contract：

```json
{
  "model_name": "PatchCore",
  "training": "no_gradient",
  "forward_input_shape": "[B, 3, 224, 224]",

  "critical_middle_tensors": [
    {"name": "features", "shape": "[B, C, H, W]", "location": "backbone output, before memory_bank lookup"},
    {"name": "memory_bank", "shape": "[N_coreset, C]", "location": "after KCenterGreedy subsampling"},
    {"name": "distances", "shape": "[B, H*W, N_coreset]", "location": "nearest neighbor search"}
  ],
  "shared_components": ["TimmFeatureExtractor", "KCenterGreedy"],
  "modifiable_hooks": [
    {"hook_name": "backbone_after", "description": "after feature extraction, before coreset subsampling", "risk": "medium"},
    {"hook_name": "embedding_before", "description": "before memory bank lookup in forward", "risk": "low"},
    {"hook_name": "distance_computation", "description": "nearest neighbor distance calculation", "risk": "low"},
    {"hook_name": "anomaly_map_generation", "description": "post-processing distance → anomaly_map", "risk": "low"},
    {"hook_name": "backbone_replacement", "description": "replace feature extractor", "risk": "high"},

```

### 核心 Prompt

参考 AI-Scientist `generate_ideas.py` 的 idea_first_prompt 结构 + AutoScientists Step 1b2 的归纳推理：

```
SYSTEM: 你是一个科研迁移专家。你的任务是从一篇论文中发掘可以迁移到指定 baseline 的科研 Idea。

你拥有：
1. 论文摘要和方法组件列表
2. baseline 的架构 contract（哪些位置可以修改、风险如何）
3. 历史上类似迁移的成功/失败记录

你需要生成 3-5 个候选 Idea。

对每个候选，要求：
- 论文中的哪一段机制
- 这个机制为什么在它的原始场景中有效（inductive reasoning，参考 AutoScientists Step 1b2）
- baseline 中对应的修改位置（使用 modifiable_hooks 中的 hook_name）
- 修改的预期效果（而不是保证效果）
- 风险等级

不要：
- 提议改动 paper evidence 不支持的机制
- 推荐修改 high risk hook 除非 medium/low 全部不适用
- 生成论文中根本没有的东西
```

### 用户交互（路径 C）

```
你的论文和 baseline 已经分析完毕。我从论文中发掘了以下可以迁移的候选 Idea：

1. Multi-scale Feature Aggregation (来源：Section 3.2)
   机制：训练一个轻量 attention 模块融合多层 backbone 特征
   插入位置：backbone 输出后、embedding 前
   为什么可能有效：PatchCore 当前只用单尺度 patch，multi-scale 已知对多尺度异常有效
   风险：medium（需要训练一个小模块）
   历史成功率：80%（5 次类似 attention 迁移中 4 次成功）
   [灵感度：★★★☆☆]

2. Adaptive Coreset Selection (来源：Section 3.3)
   机制：不随机选 coreset，用 uncertainty 加权
   插入位置：替换 KCenterGreedy
   为什么可能有效：coreset 质量直接影响 memory bank 代表性
   风险：low（只替换算法，不改变接口）
   历史成功率：暂无记录（新方向）
   [灵感度：★★★★☆]

3. Contrastive Feature Learning (来源：Section 4.2)
   机制：训练阶段引入 contrastive loss 使正常样本特征更紧凑
   插入位置：训练阶段（PatchCore 当前无训练）
   为什么可能有效：特征更紧凑 → 异常更易检测
   风险：high（需要引入训练阶段，改变 PatchCore 基本范式）
   历史成功率：33%（3 次中 1 次成功）
   [灵感度：★★☆☆☆]

4. Top-K Anomaly Pooling (来源：Section 5.1)
   机制：只取距离最大的 top-K 个 patch 计算 anomaly score
   插入位置：distance computation 后，anomaly map 生成前
   为什么可能有效：减少 background noise 对评分的影响
   风险：low（只修改后处理）
   [灵感度：★★★★★]

你想尝试哪几个？（可以多选）
```

每个候选附带：
- 论文证据来源
- 为什么可能有效（inductive reasoning）
- insertion point（使用 hook_name）
- 风险等级
- 历史成功率（如果有）
- inspiration_score（帮助排序）

# 5. MechanismExtractor（公共能力）

无论哪个路径，确认 Idea 后都要进入此步骤。

## 5.1 做什么

```
1. 从 paper_summary.json 和 method_components.json 中定位目标 method
2. 提取该 method 的完整信息，写入 idea_contract.json
3. 标注证据来源
```

## 5.2 idea_contract.json

```python
class IdeaContract(BaseModel):
    idea_id: str
    source: Literal["user_specified", "paper_extracted", "system_generated"]
    route: Literal["A_user_specified", "B_paper_candidate", "C_system_generated"]

    # 核心信息
    mechanism_label: str
    mechanism_description: str
    mechanism_why: str  # 为什么它在原始论文中有效（AutoScientists Step 1b2 风格）
    problem_solved: str
    must_preserve_behaviors: list[str]

    # 论文来源
    paper_location: str  # section / page / figure
    paper_evidence_ids: list[str]

    # 用户确认
    user_confirmation: Literal["confirmed", "user_selected_from_candidates", "pending"] | None

    class Config:
        extra = "forbid"
```

### mechanism_why 字段

参考 AutoScientists Step 1b2 的"Post-KEEP Inductive Reasoning"：

不能只写"作者提出了一个 attention 模块"——必须解释**为什么这个模块在原始场景下有效**：

```
好的 mechanism_why:
"Cross-scale attention fuses fine-to-coarse features that are
independently extracted by the backbone. In the original paper's
segmentation task, this bridges the gap between high-resolution
localization and semantic context. The property that matters for
transfer is 'multi-scale feature correspondence' — something
PatchCore currently lacks because it operates on single-scale
patches from fixed backbone layers."

差的 mechanism_why:
"The paper proposes an attention module that improves performance."
```

# 6. ArchitectureAligner（公共能力）

## 6.1 做什么

```
1. 读取 idea_contract.json 中的机制描述
2. 读取 baseline_contract.json 中的架构拓扑
3. 产出一张"对齐表"——论文中每个组件的输入/输出对应 baseline 的哪个 tensor
4. 标注匹配关系和缺口
```

## 6.2 对齐表格式

```json
{
  "alignment_table": [
    {
      "paper_component": "Cross-Scale Attention input",
      "paper_tensor_shape": "[B, C1, H1, W1] (from layer2) + [B, C2, H2, W2] (from layer3)",
      "baseline_equivalent": "features after backbone layers ['layer2', 'layer3']",
      "baseline_tensor_shape": "[B, 512, 28, 28] + [B, 1024, 14, 14] (WRN50)",
      "match_status": "compatible",
      "note": "PatchCore already extracts from both layers — shapes match"
    },
    {
      "paper_component": "Cross-Scale Attention output",
      "paper_tensor_shape": "[B, C_fused, H, W]",
      "baseline_equivalent": "features before pooling → memory_bank",
      "baseline_tensor_shape": "[B, C, H_patches, W_patches]",
      "match_status": "compatible_with_adapter",
      "note": "需要将 attention 输出 reshape 为 patch grid 格式"
    },
    {
      "paper_component": "learnable query/key/value weights",
      "paper_tensor_shape": "[d, d] 矩阵",
      "baseline_equivalent": "<无对应>",
      "baseline_tensor_shape": null,
      "match_status": "incompatible",
      "note": "PatchCore 无训练阶段，需要新增训练步骤来学习这些参数"
    }
  ]
}
```

## 6.3 对齐后的后续路由

| 对齐结果 | 含义 | 下一步 |
|---------|------|--------|
| 全部 `compatible` | 论文机制可以直接插入 | CompatibilityJudge（快速确认） |
| 有 `compatible_with_adapter` | 需要 adapter/bridge | CompatibilityJudge（重点检查 adapter 可行性） |
| 有 `incompatible` | 存在根本性冲突 | CompatibilityJudge → 可能标记为 non_viable |
| 有 `unknown` | paper evidence 信息不足 | 标记 `insufficient_evidence`，可选向 3.2 请求 reanalysis |

# 7. CompatibilityJudge（公共能力）

## 7.1 做什么

逐项判断兼容性，参考 AI-Scientist `perform_review.py` 的多维度评分模式 + SWE-Agent `reviewer.py` 的 retry loop。

## 7.2 兼容性维度

```python
class CompatibilityDimension(str, Enum):
    SEMANTIC = "semantic"
    INPUT = "input"
    OUTPUT = "output"
    SHAPE = "shape"
    TRAINING = "training"
    DATA = "data"
    LABEL = "label"
    EVALUATION = "evaluation"
    RESOURCE = "resource"

class CompatibilityStatus(str, Enum):
    COMPATIBLE = "compatible"
    COMPATIBLE_WITH_ADAPTER = "compatible_with_adapter"
    INCOMPATIBLE = "incompatible"
    INSUFFICIENT_EVIDENCE = "insufficient_evidence"

class DimensionJudgment(BaseModel):
    dimension: CompatibilityDimension
    status: CompatibilityStatus
    score: int  # 1-10，severity/confidence 综合得分
    reasoning: str
    evidence_ids: list[str]
    adapter_requirement: str | None  # 如果 compatible_with_adapter，需要什么 adapter
    blocker: bool  # 是否构成阻塞
```

### 7.2.1 Semantic Compatibility（语义兼容性）

```
问题：论文机制解决的问题，在异常检测 baseline 中是否存在对应问题？

判断逻辑：
- baseline 中是否存在与论文对应的"gap"
- 如果论文解决的是分割中的信息丢失问题，AD 的 anomaly map 生成也有类似问题 → compatible
- 如果论文解决的是时序建模问题，AD baseline 是纯图像模型 → incompatible

证据来源：
- paper_summary.json 的 claims
- baseline_contract.json 的 known_limitations
- 历史 transfer success/failure 记录
```

### 7.2.2 Input Compatibility（输入兼容性）

```
问题：论文机制的输入类型和 shape 是否在 baseline 中存在对应？

检查项：
- tensor dtype（float32/float16）
- channel 维度对齐（论文假设 RGB，baseline 也是 RGB → compatible）
- 是否需要预处理（论文需要归一化到[-1,1]，baseline 是 ImageNet 归一化 → compatible_with_adapter）

注意：不是说"形状一模一样"——而是说 baseline 中是否存在对应的信息流。
```

### 7.2.3 Output Compatibility（输出兼容性）

```
问题：论文机制产生的输出能否被 baseline 下游消费？

检查项：
- 输出 tensor 是否有下游消费者（例如 PatchCore 的 features 进入 memory_bank）
- 如果论文机制输出的是 probability map，但 baseline 下游期望 feature vector → compatible_with_adapter
- 如果论文机制可能破坏 InferenceBatch 的 pred_score 计算 → blocking
```

### 7.2.4 Shape Compatibility（形状兼容性）

```
问题：tensor shape 在插入点是否匹配？

检查项：
- 论文中 tensor 的维度语义（B=128 → baseline 用 B=32 可以吗？）
- spatial resolution（paper 用 256×256，baseline 用 224×224 → adapter 做 resize）
- channel dimension 的对齐方式

这是一个确定性检查（re 正则匹配 shape 字符串 + 维度语义分析），
不需要 LLM 做算术。
```

### 7.2.5 Training Compatibility（训练兼容性）

```
问题：论文机制是否需要训练？baseline 是否支持训练？

判断逻辑：
- PatchCore/PaDiM：无训练阶段（no_gradient），只做 inference
- FastFlow：有训练阶段（flow 需要 fit）
- 如果论文机制需要训练参数，目标 baseline 是 PatchCore → incompatible
  除非 variant 将其改为 frozen 或 lightweight fine-tuning
```

### 7.2.6 Data Compatibility（数据兼容性）

```
问题：论文使用的数据集假设是否与 baseline 冲突？

检查项：
- 论文是否需要 labeled anomaly（异常检测通常只使用正常样本训练）
- 是否需要 paired data（论文用了 stereo pairs → 不兼容）
- 是否需要特定 domain 的数据（医学影像 → 但 baseline 在工业检测上）
- 是否需要超大数据量（论文训练了 1M 样本 → 资源检查）
```

### 7.2.7 Label Compatibility（标签兼容性）

```
问题：论文是否需要 baseline 数据集中不存在的标签？

检查项：
- 论文需要 pixel-level anomaly segmentation mask → MVTec AD 提供了，compatible
- 论文需要 bounding box → MVTec AD 不提供，incompatible
- 论文完全不需要标签（unsupervised）→ 所有 AD baseline 都支持
```

### 7.2.8 Evaluation Compatibility（评价兼容性）

```
问题：引入论文机制后，评价协议是否仍然有效？

检查项：
- protected path（3.3 中确认不可修改的 eval script）
- 论文是否要求特殊的 eval protocol（例如需要 retrain N 次取平均 → 超出资源预算）
- baseline 的 eval 流程有没有被破坏（例如论文机制生成额外输出，导致 eval script 报错）
```

### 7.2.9 Resource Compatibility（资源兼容性）

```
问题：引入论文机制后，资源需求是否在用户预算范围内？

检查项：
- GPU memory（新增模块是否超出显存）
- 训练时间（如果需要新增训练步骤）
- 存储（是否需要存储额外中间结果）
- 参考 AutoSOTA 的教训：绝大多数有效优化是小改动，不是架构重写
```

## 7.3 Judge 工作模式

参考 SWE-Agent `ScoreRetryLoop` + AI-Scientist 的 ensemble meta-review：

```python
def judge_compatibility(
    idea_contract: IdeaContract,
    alignment_table: list[AlignmentEntry],
    baseline_contract: BaselineContract,
    protected_constraints: list[ProtectedConstraint],
) -> TransferAnalysis:
    # 1. 每维度独立评分（1-10）
    judgments = []
    for dim in CompatibilityDimension:
        judgment = judge_single_dimension(dim, ...)
        judgments.append(judgment)

    # 2. 如果存在 incompatible，标记 blocker
    blockers = [j for j in judgments if j.status == CompatibilityStatus.INCOMPATIBLE and j.blocker]

    # 3. 如果 blocker > 0，进入 retry loop（最多 2 次）
    #    —— 让 LLM 反思是否可以设计 adapter 绕过
    if blockers:
        judgments = retry_with_reflection(judgments, blockers, max_attempts=2)

    # 4. 计算 overall score
    #    全部 compatible → 10
    #    全部 compatible/compatible_with_adapter → 7+ 
    #    有 incompatible（非 blocker）→ 4-6
    #    有 incompatible（blocker）→ 1-3

    # 5. 如果 overall < 4 且用户确认了该 Idea → 标记 non_viable 并建议换方向
```

### retry loop 逻辑

```
第 1 轮：直接判断
第 2 轮：对每个 incompatible 维度，
  "假设你可以添加一个 adapter/bridge 模块，
   能否在不改变 protected path 的前提下绕过这个不兼容？
   如果能，给出 adapter 的设计思路。"
第 3 轮（如果仍有个别维度 incompatible）：
  最终裁决——标记为 incompatible，在 transfer_constraints.json 中记录原因
```

# 8. VariantGenerator（公共能力）

## 8.1 做什么

```
输入：idea_contract.json + transfer_analysis.json + baseline_contract.json
输出：同一 Idea 下的 1-3 个实现 Variant
```

**关键设计原则（来自 AutoSOTA 的教训）：Variant 是同一 Idea 的不同实现方式，不是不同的科研方向。**

## 8.2 Variant 的差异维度

同一个 Idea（Cross-Scale Attention）可以有以下 Variant：

| Variant | 插入位置 | 差异 |
|---------|---------|------|
| A | backbone 输出后、memory bank 构建前 | 在 pooling 前融合多尺度特征 |
| B | memory bank 构建后、embedding 查找前 | 在特征向量上做 attention，而不是 feature map |
| C | 只修改 query feature，不改 memory bank | 最保守——只动 inference 过程 |

## 8.3 Variant 模型

```python
class ImplementationVariant(BaseModel):
    variant_id: str
    variant_label: str  # e.g. "Variant A: Pre-memory-bank fusion"
    idea_id: str

    # 插入点
    insertion_hook: str  # 使用 baseline_contract.modifiable_hooks 中的 hook_name
    insertion_before: str  # 在哪个 component 之前
    insertion_after: str   # 在哪个 component 之后

    # 修改范围
    files_to_modify: list[str]
    files_to_add: list[str]
    new_components: list[NewComponentSpec]
    modified_components: list[ModifiedComponentSpec]

    # 复杂度
    lines_changed_estimate: int
    new_dependencies: list[str]
    breaks_protected_path: bool
    requires_training: bool

    # 预期
    expected_improvement_rationale: str
    risk_level: Literal["low", "medium", "high"]
    fallback_behavior: str  # 失败了会怎样

    # 证据
    paper_evidence_ids: list[str]
    transfer_analysis_ref: str
```

## 8.4 Variant 生成的约束

参考 AutoSOTA 的经验——有效优化通常是小改动：

```
1. 必须使用 baseline_contract.modifiable_hooks 中定义的 hook
2. 不能修改 protected path（evaluation script、CLI interface）
3. 必须保留 InferenceBatch 输出格式
4. 如果 baseline 是 no_gradient 类型，variant C 应该完全不需要训练
5. 优先 medium/low risk hook，除非用户声明接受 high risk
6. 每个 variant 之间应有明确的"差异维度"——如果两个 variant 太相似，合并为一个
```

## 8.5 Variant 生成 Prompt 逻辑

参考 AI-Scientist `perform_experiments.py` 的 coder_prompt + SWE-Agent patch_formatter：

```
SYSTEM: 你是一个代码迁移专家。你有以下信息：
1. Idea 的完整描述和论文证据
2. Baseline 的架构 contract 和 modifiable hooks
3. 兼容性分析结果（哪些维度 compatible，哪些需要 adapter）

你需要为同一个 Idea 生成 1-3 个实现 Variant。

规则：
- 每个 Variant 使用不同的 insertion hook
- Variant C 应该是最保守的（最小改动）
- Variant A 应该是最激进的（最大潜力）
- 输出每个 Variant 的文件级修改计划（不生成完整代码）
- 标注 expected improvement rationale
- 标注风险

不要：
- 建议修改 baseline 的 protected path
- 生成两个几乎相同的 Variant
- 建议论文中不存在的机制
```

# 12. 用户交互协议

## 12.1 Turn Structure

每次向用户输出：

```
1. 当前阶段摘要
2. 关键判断及理由
3. 最多 1 个需要用户做的选择
4. 选项及后果
5. 默认建议（如果有）
```

## 12.2 确认类型

- `confirm` — 用户确认系统理解
- `select_one` — 用户从候选中选择一个
- `select_multi` — 用户从候选中选择多个
- `revise` — 用户修改系统理解
- `reject` — 用户拒绝当前方向，回到候选列表
- `switch_idea` — 用户换一个 Idea
- `generate_more` — 用户要求生成更多候选

## 12.3 不该问用户的问题

```
❌ "你觉得这个机制的 tensor shape 兼容吗？"
   → 这是系统的技术判断，不是用户偏好

❌ "baseline 的第 153 行代码在做什么？"
   → 这是 3.1 的技术分析

❌ "参数 λ 应该设为 0.5 还是 1.0？"
   → 这是 3.5 Experiment Planner 的调参问题

✅ "论文中有 3 个候选模块，你想迁移哪个？"
   → 用户提供方向

✅ "Variant A 需要训练、B 不需要，你允许引入训练步骤吗？"
   → 用户提供约束

✅ "这个 Idea 的兼容性评分只有 4/10，要继续还是换一个？"
   → 用户做 risk/reward 决策
```

## 12.4 当用户沉默

```
- 路径 B（论文有候选）：默认推荐 role=main_contribution 且 feasibility_hint=likely_direct 的候选
- 路径 C（无 Idea）：默认推荐 inspiration_score 最高的 3 个
- Variant 选择：默认选择 risk=low 的 variant（最保守方案）
```

```
Variant C 是 risk=low
不生成相同插入点的两个 variant
不修改 protected path
```

## 16.7 Security Tests

```
用户输入不在 evidence 之外改变 contract
prompt injection 不改变 system policy
generated idea 不包含论文中不存在的 method
```

# 17. 风险与控制

## 风险 1：路径 C 生成不靠谱的 Idea

控制：
```
- 每个候选必须引用 paper evidence
- inspiration_score < 3 的不展示给用户
- 历史 transfer 成功率提供给用户作为参考
```

## 风险 2：CompatibilityJudge 误判

控制：
```
- retry loop 给 LLM 第二次机会
- 历史 transfer 记录用于校准（cold axis → 不要直接判 incompatible）
- insufficient_evidence 优于强行判 compatible/incompatible
```

## 风险 3：阶段膨胀——3.4 变成新的独立大阶段

控制：
```
- 复用 3.2 的 paper_summary 和 method_components，不重复读论文
- 复用 3.1 的 repository_summary，不重复读仓库
- baseline_contract 是一次性提取，cache 后复用
- 路径 A（用户指定）可以跳过 IdeaExtractor 和 IdeaGenerator
```

## 风险 4：用户选择 paralysis（路径 C 太多候选）

控制：
```
- 最多展示 5 个候选
- 排序：inspiration_score × history_success_rate
- 默认推荐前 3 个
- 用户说"都可以"就选 top 3
```

# 18. 与上下游的接口

## 18.1 上游（3.3 → 3.4）

3.3 输出的 `idea_transfer_handoff.json` 进入 3.4。3.4 需要：

```
- task_goal: 用户想做什么
- eligible_idea_sources: method_components 引用（路径 B 的输入）
- user_preferences: 用户已确认的偏好和约束
- protected_constraints: 不能改的文件/接口
- known_transfer_constraints: 3.3 已识别的已知约束
```

## 18.2 下游（3.4 → 3.5）

3.5 从 `idea_transfer_design_handoff.json` 中得到：

```
- confirmed_idea: 确认的 Idea 合约
- transfer_analysis: 兼容性全貌（3.5 规划实验时参考）
- selected_variants: 用户选择的 Variant（3.5 为每个 variant 生成实验计划）
- accepted_risks: 用户接受的已知风险
- unresolved_dimensions: 3.5 需要在实验设计中处理的未解决问题
```

3.5 不需要重新确认 Idea 或重新判断兼容性——除非实际生成 patch 时发现新的技术冲突。

# 21. 最终阶段定义

Step 3.4 完成后可以声明：

> AutoAD-Researcher 能够在确认研究上下文后，从论文中提取或生成候选科研 Idea，逐维度判断其向异常检测 baseline 的迁移兼容性，并为每个确认的 Idea 生成 1-3 个实现 Variant 供用户选择。系统支持三条路径：用户明确指定、从论文候选中选择、或由系统根据论文和 baseline 自动生成候选。

不能声明：

```
已生成可运行代码 patch（那是 3.5 的工作）
已执行实验（那是 3.6+ 的工作）
已验证迁移效果
已自动选择最优 variant
```
