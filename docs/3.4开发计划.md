# AutoAD-Researcher Step 3.4 开发计划：Idea & Transfer Design

> 文档状态：修订稿 v1.1（v1 review_failed，本轮重写）
> 更新日期：2026-06-17
> 阶段名称：**Idea & Transfer Design / 科研 Idea 确认与迁移方案设计**
> 上游输入：`stable research_context.json` + `idea_transfer_handoff.json`（来自 3.2 Context Layer，或经 3.3 ContextPatch 修复后）
> 下游阶段：Step 3.5 Multi-variant Experiment Planner
> 3.4 本身：**不执行代码、不运行实验、不生成文件级 PatchPlan、不重复读取论文/仓库源码**

---

# 0. 阶段定义

Step 3.4 合并原计划中的两个阶段：
- 原 3.4 Single Idea Confirmation
- 原 3.5 Transferability Judge

合并后的 3.4 完成一件事：

```
从 stable 研究上下文出发
  → 用户确认一个科研 Idea（每个 run 恰好一个）
  → 对照 baseline architecture contract 判断迁移兼容性
  → 生成 1–3 个架构级实现 Variant
  → 用户选择 Variant
  → 确定性验证
  → 交给 3.5 Experiment Planner
```

## 0.1 核心不变量

```
一个 run → 一个 confirmed Idea
一个 Idea → 1–3 个 implementation Variant
Variant 可以选一个或多个
Idea 不得多选
```

用户希望尝试多个 Idea 时，应为每个 Idea 建立独立 child run，不能放入同一个 `IdeaContract`。

## 0.2 职责边界

3.4 做：
- 路由 Idea 来源（用户 / 论文候选）
- 将用户确认的一个 Idea 转为 `IdeaContract`
- 消费 3.1 的 `baseline_architecture_contract.json` 和 3.2 的 `method_components.json` / `paper_idea_sources.json`
- 逐维度判断兼容性
- 生成架构级 Variant（hook、接口、训练范式、adapter）
- 确定性验证

3.4 不做：
- 重新读取论文 PDF 或解析方法组件（3.2 已做）
- 重新扫描仓库源码或提取 tensor position（3.1 已做）
- 生成文件级 PatchPlan（3.6）
- 生成开放式科研 Idea（超出 MVP）
- 执行代码或运行实验
- 替用户静默确认任何决策

## 1.1 流程总览


## 1.2 路径含义

| 路径 | 触发条件 | 过程 | 用户参与 |
|------|---------|------|---------|
| **A: 用户提供 Idea** | `eligible_idea_sources` 中有 `user_provided` 条目，或用户在入口直接声明 | IdeaSourceRouter 验证 → 记录 UserInputEvidenceRef → 进入 IdeaContractNormalizer | confirm 即可 |
| **B: 论文候选** | `paper_idea_sources` 非空，用户未指定具体 method | IdeaSourceResolver 展示 3.2 已提取的候选列表 → 用户 select_one | 选择一个 |
| **blocked** | 无 user-provided Idea 且 paper_idea_sources 为空或无合法候选 | 不自动生成、不降级到其他路径 | 用户提供方向，或回到 3.2 重新读材料 |

路径 C（开放式 IdeaGenerator）不在 MVP 范围内。未来可独立增加 `Open-ended Idea Generation` 阶段。

## 1.3 折中：paper-grounded transfer hypothesis

当 `paper_idea_sources` 非空但用户不了解论文具体内容时，3.4 可以提供辅助：

```
从 3.2 已读取的 paper_idea_sources 中
  → 对每个 source，从 baseline_architecture_contract 查找可能的 insertion hook
  → 展示为 "论文机制 X 可能对应 baseline hook Y"
  → 但仍然是用户从已知 paper source 中选择
```

这不算开放式生成——机制必须已在论文中存在（由 3.2 的 PaperEvidenceRef 保证）。只是在 baseline 中提出可能的对应位置。

但最终仍只能确认一个，且必须有 `PaperEvidenceRef`。

# 2. 路径 A：用户提供 Idea

## 2.1 触发条件

用户明确声明了想迁移的具体 method。

来源可以是：

1. 3.2/3.3 过程中用户已指定
2. 在 3.4 入口直接输入
3. 用户提出的是原创 idea（不在论文中），但希望以论文中的某个组件为 baseline

## 2.2 IdeaSourceRouter

```
1. 读取用户声明的 idea 标识
2. 判断来源类型：
   a. 在 paper_idea_sources 中有对应条目 → source=paper_grounded, evidence=PaperEvidenceRef
   b. 用户声明为原创，不在论文中 → source=user_provided, evidence=UserInputEvidenceRef
   c. 用户提到的名称在 paper_idea_sources 中不存在
      → 先检查是否是名称 typo（模糊匹配）
      → 可能是 typo → 向用户确认
      → paper evidence 应存在但 3.2 未读到 → paper_reanalysis_request 回到 3.2
      → 用户确认是原创 → source=user_provided
3. 记录 UserInputEvidenceRef
4. 输出 routed_idea_source
```

**不静默降级**：用户说的 method 在论文中找不到时，不自动替换成系统生成的 Idea。路由到 paper_reanalysis_request 或向用户确认。

## 2.3 输出

```json
{
  "route": "A_user_provided",
  "routed_source": {
    "idea_label": "Cross-Scale Attention",
    "source": "paper_grounded",
    "paper_evidence_ids": ["ev_paper_method_cross_scale_attn"],
    "user_evidence_id": "ev_user_specified_method_001"
  }
}
```

# 4. IdeaContractNormalizer（公共）

无论路径 A 还是 B，确认后进入此步骤。

## 4.1 做什么

```
1. 读取用户确认的一个 idea source（来自路径 A 的 routed_source 或路径 B 的 selected_candidate）
2. 从 3.2 的 paper_idea_sources / method_components 中提取该 source 的现有事实
3. 补充迁移语义（mechanism_why_from_paper、transfer_relevance）
4. 生成规范的 IdeaContract
5. 记录用户确认证据
```

## 4.2 IdeaContract

```python
class IdeaContract(BaseModel):
    idea_id: str
    source: Literal["user_provided", "paper_grounded"]

    # 从 3.2 消费的论文事实（如果 source=paper_grounded）
    paper_mechanism_label: str | None
    paper_mechanism_summary: str | None
    paper_location: str | None
    paper_evidence_ids: list[str]  # PaperEvidenceRef

    # 用户提供的信息
    user_description: str | None
    user_evidence_id: str  # UserInputEvidenceRef

    # 3.4 补充的迁移语义（不是论文新事实，是迁移假设）
    mechanism_why_in_paper: str  # 论文机制在原始场景下为什么有效
    transfer_relevance: str  # 为什么这个机制可能对 baseline 有价值

    # 必须保持不变的行为
    must_preserve_behaviors: list[str]

    # 确认状态
    confirmation_status: Literal["confirmed", "pending"]
    confirmed_by_user_at: datetime | None
    confirmation_evidence_id: str | None

    class Config:
        extra = "forbid"
```

### 关键约束

- `paper_evidence_ids` 必须来自 3.2 的 `PaperEvidenceRef`，3.4 不得新增论文事实
- `user_evidence_id` 必须来自 `UserInputEvidenceRef`
- `mechanism_why_in_paper` 和 `transfer_relevance` 是 3.4 的分析，不是事实——二者必须标记为 `producer_stage=3.4` 且 status=`inferred`

## 5.2 baseline_architecture_contract 结构

由 3.1 Repository Intelligence 产出。3.4 只消费，不提取：

```python
class TensorAxis(BaseModel):
    name: str  # "B" | "C" | "H" | "W" | ...
    semantic_role: str  # "batch" | "channel" | "spatial_h" | "spatial_w" | ...
    dynamic: bool  # 是否在运行时变化
    typical_value: int | None

class TensorSpec(BaseModel):
    tensor_name: str
    rank: int | None
    axes: list[TensorAxis]
    dtype: str | None  # "float32" | "float16" | ...
    layout: str | None  # "NCHW" | "NHWC" | ...
    semantic_role: str  # 在 pipeline 中的语义角色
    location_hook: str | None  # 在哪个 hook 处存在
    evidence_ids: list[str]  # RepositoryEvidenceRef

class InterfaceSpec(BaseModel):
    interface_name: str
    tensors: list[TensorSpec]
    is_batched: bool

class ExecutionPhaseContract(BaseModel):
    phase: Literal["fit", "train", "infer", "postprocess", "evaluate"]
    uses_gradient: bool | None
    mutates_model_state: bool | None
    mutates_which_state: list[str]  # e.g. ["memory_bank", "backbone_weights"]
    inputs: list[InterfaceSpec]
    outputs: list[InterfaceSpec]

class ModificationHook(BaseModel):
    hook_name: str
    description: str
    location_in_code: str  # module path, not line number
    pre_tensors: list[str]  # tensor names before this hook
    post_tensors: list[str]  # tensor names after this hook
    risk: Literal["low", "medium", "high"]
    evidence_ids: list[str]

class BaselineArchitectureContract(BaseModel):
    model_name: str
    repository_source_id: str
    repository_commit: str
    backbone: str
    phases: list[ExecutionPhaseContract]
    tensors: list[TensorSpec]
    modifiable_hooks: list[ModificationHook]
    shared_components: list[str]
    evidence_ids: list[str]
```

## 5.3 对齐逻辑

```
1. 读取 IdeaContract 中的方法描述
2. 如果 source=paper_grounded：
   从 3.2 method_components 中获取该方法的 input/output 描述
   如果 3.2 的描述不足以对齐 → insufficient_paper_evidence → paper_reanalysis_request
3. 对照 baseline_architecture_contract：
   a. 每个 hook → 检查该位置的 tensor 语义是否匹配论文机制的语义
   b. 每个 ExecutionPhase → 检查论文机制需要哪个阶段
4. 输出对齐表
```

### 对齐后的路由

| 对齐结果 | 含义 | 后续 |
|---------|------|--------|
| 全部 `compatible` | 可以直接插入 | 进入 CompatibilityAnalyzer |
| 有 `requires_regime_change` | 需要改变执行阶段 | 进入 CompatibilityAnalyzer → 重点标记 |
| 有 `insufficient_paper_evidence` | 3.2 信息不够 | `paper_reanalysis_request` |
| 有 `insufficient_repository_evidence` | 3.1 信息不够 | `repository_reanalysis_request` |

# 6. CompatibilityAnalyzer（公共）

## 6.1 设计原则

- **不使用伪精确的 1-10 数值评分**
- **每个维度使用结构化状态**
- **最终结论由确定性规则派生，不由 LLM "想一个分数"**
- retry loop 只能用于：schema repair、消除表达矛盾、补充缺失 evidence，不能让 LLM 对同一事实"再想一次"直到把 incompatible 改成 compatible

## 6.2 兼容性维度

```python
class CompatibilityDimension(str, Enum):
    SEMANTIC = "semantic"
    INPUT = "input"
    OUTPUT = "output"
    SHAPE = "shape"
    TRAINING = "training"
    DATA = "data"
    LABEL = "label"
    EVALUATION = "evaluation"
    RESOURCE = "resource"

class CompatibilityStatus(str, Enum):
    COMPATIBLE = "compatible"
    COMPATIBLE_WITH_ADAPTER = "compatible_with_adapter"
    REQUIRES_REGIME_CHANGE = "requires_regime_change"
    INCOMPATIBLE = "incompatible"
    INSUFFICIENT_EVIDENCE = "insufficient_evidence"
```

### 6.2.1 各维度检查内容

**Semantic Compatibility**：论文机制解决的问题，在 baseline 中是否存在对应 gap？

**Input Compatibility**：论文机制的输入在 baseline 中是否有语义等价的信息流？
- 不是"形状一模一样"，而是"存在对应的信息流"
- 使用 baseline_architecture_contract 的 TensorSpec.semantic_role 做语义匹配

**Output Compatibility**：论文机制能否产出 baseline 下游期望的接口？
- 检查 ExecutionPhaseContract.outputs

**Shape Compatibility**：使用 TensorSpec 的 axes 做结构化检查
- 匹配 rank、axis semantic_role、dtype
- 不依赖 LLM 解释自由字符串

**Training Compatibility**：论文机制需要的执行阶段在 baseline 中是否存在？
- 使用 ExecutionPhaseContract.phase + uses_gradient + mutates_model_state
- 不再使用模糊的 `training=no_gradient`

**Data Compatibility**：论文的数据假设是否与 baseline 数据域冲突？

**Label Compatibility**：论文是否需要 baseline 数据不存在的标签？

**Evaluation Compatibility**：论文机制是否破坏 baseline 的评价协议？

**Resource Compatibility**：新增开销是否在预算内？

## 6.3 维度判定模型

```python
class DimensionJudgment(BaseModel):
    dimension: CompatibilityDimension
    status: CompatibilityStatus
    blocking: bool
    reasoning: str

    # 证据
    idea_contract_evidence_ids: list[str]  # IdeaContract 中的事实
    paper_evidence_ids: list[str]  # PaperEvidenceRef
    repository_evidence_ids: list[str]  # RepositoryEvidenceRef

    # 当 status 不是 compatible 时
    assumptions: list[str]  # 判断依赖的未验证假设
    required_changes: list[str]  # 需要修改什么
    required_adapter: str | None  # 如果需要 adapter
    risk: Literal["low", "medium", "high"] | None
```

## 6.4 整体结论的确定性派生规则

```python
def derive_overall_status(judgments: list[DimensionJudgment]) -> TransferStatus:
    # 规则 1: 任何 dimension blocking=true → non_viable
    if any(j.blocking for j in judgments):
        return TransferStatus.NON_VIABLE

    # 规则 2: 任何 dimension INSUFFICIENT_EVIDENCE 且影响核心接口 → needs_reanalysis
    core_dims = {CompatibilityDimension.INPUT, CompatibilityDimension.OUTPUT, CompatibilityDimension.TRAINING}
    if any(j.status == CompatibilityStatus.INSUFFICIENT_EVIDENCE and j.dimension in core_dims
           for j in judgments):
        return TransferStatus.NEEDS_REANALYSIS

    # 规则 3: 无 blocker，但有 COMPATIBLE_WITH_ADAPTER 或 REQUIRES_REGIME_CHANGE → viable_with_conditions
    if any(j.status in (CompatibilityStatus.COMPATIBLE_WITH_ADAPTER,
                        CompatibilityStatus.REQUIRES_REGIME_CHANGE)
           for j in judgments):
        return TransferStatus.VIABLE_WITH_CONDITIONS

    # 规则 4: 全部 COMPATIBLE → viable
    return TransferStatus.VIABLE

class TransferStatus(str, Enum):
    VIABLE = "viable"
    VIABLE_WITH_CONDITIONS = "viable_with_conditions"
    NON_VIABLE = "non_viable"
    NEEDS_REANALYSIS = "needs_reanalysis"
```

## 6.5 Retry Loop 的合法用途

```
retry 只允许在以下情况重新要求 LLM 输出：

1. schema 缺失字段 → 补充缺失字段
2. judgment 内部矛盾 → 修正矛盾（例如 reasoning 说 compatible 但 status 是 incompatible）
3. evidence_ids 引用了一个不存在的 evidence → 修正引用

retry 不允许：
- 对同一事实重新判断
- 把 INCOMPATIBLE 改成 COMPATIBLE 通过"再想一想"
- 超过 2 次重试
```

# 7. VariantGenerator（公共）

## 7.1 做什么

```
输入：IdeaContract + transfer_analysis.json + baseline_architecture_contract
输出：同一 Idea 下的 1–3 个架构级 Variant
```

## 7.2 Variant 模型（架构级，不涉及文件/函数名）

```python
class ImplementationVariant(BaseModel):
    variant_id: str
    variant_label: str
    idea_id: str

    # 架构级描述
    target_hook: str  # 对应 baseline_architecture_contract.modifiable_hooks 中的 hook_name

    # 接口变更
    input_interface_change: InterfaceChangeDescription | None
    output_interface_change: InterfaceChangeDescription | None
    adapter_required: bool
    adapter_description: str | None

    # 执行阶段变更
    regime_change: RegimeChange | None

    # 状态/记忆变更
    state_change: StateChangeDescription | None

    # 约束
    touches_protected_path: bool  # 必须为 false
    new_dependencies: list[str]

    # 预期和风险
    expected_behavior_rationale: str
    risk_level: Literal["low", "medium", "high"]
    fallback_behavior: str

    # 证据追溯
    idea_contract_evidence_ids: list[str]
    transfer_analysis_ref: str

class InterfaceChangeDescription(BaseModel):
    before: str
    after: str
    tensors_added: list[str]
    tensors_modified: list[str]

class RegimeChange(BaseModel):
    phases_added: list[str]
    phases_modified: list[str]
    gradient_required: bool
    state_mutation_required: bool

class StateChangeDescription(BaseModel):
    state_name: str
    change_type: Literal["added", "modified", "replaced"]
    description: str
```

### 不包含（留给 3.6 Patch Planner）

```
- files_to_modify / files_to_add
- 具体函数名
- 代码行数估计
- 具体代码步骤
- final_patch.diff
```

## 7.3 Variant 生成规则

```
1. 同一 Idea 下 1–3 个 Variant
2. 每个 Variant 使用 baseline_architecture_contract 中定义的 modifiable_hook
3. Variant 之间的差异维度（至少满足一条）：
   a. 不同的 target_hook
   b. 不同的 adapter 策略
   c. 不同的 regime_change
   d. 不同的 state_change
   e. 不同的参数共享策略
4. 同一 hook 也可以有多个 Variant（通过不同 adapter/regime/state 区分）
5. touches_protected_path 必须为 false
6. 不强制"A 最激进、C 最保守"，不强制 variant_id 携带语义顺序
```

## 7.4 用户交互

```
基于 Idea "Cross-Scale Attention for PatchCore"，设计了 3 个架构级 Variant：

Variant A: Pre-memory-bank multi-scale fusion
  插入 hook: backbone_after
  接口变更: 在 backbone[output] 和 memory_bank[input] 之间插入 fused feature tensor
  需要 adapter: 否
  执行阶段: +fit 阶段（只训练 attention 权重，不动 backbone）
  状态变更: 无
  风险: medium（引入训练阶段）

Variant B: Post-memory-bank feature refinement
  插入 hook: embedding_before
  接口变更: 在 memory_bank[output] 和 distance[input] 之间修改 query tensor
  需要 adapter: 否
  执行阶段: 不新增
  状态变更: 无
  风险: low（纯 inference 修改）

Variant C: Query-only multi-head re-weighting
  插入 hook: embedding_before
  接口变更: 在 query 向量上增加 re-weighting，不修改 memory_bank
  需要 adapter: 否
  执行阶段: 不新增
```

# 9. DeterministicValidator

## 9.1 五类不变量

### 9.1.1 Idea 不变量

```
1. 一个 run 恰好一个 confirmed Idea
2. Idea 有明确 source evidence（UserInputEvidenceRef 或 PaperEvidenceRef）
3. confirmation_status = confirmed
4. confirmed_by_user_at 和 confirmation_evidence_id 存在
5. paper_evidence_ids 中的每个 ID 在 Evidence Index 中存在（如果 source=paper_grounded）
```

### 9.1.2 Baseline 不变量

```
1. baseline_architecture_contract 来自固定 repository_source_id 和 repository_commit
2. 所有 modifiable_hook.hook_name 在 contract 中唯一
3. 所有 TensorSpec 引用的 evidence_ids 在 Evidence Index 中存在
4. 所有 ExecutionPhaseContract 的 phase 唯一
5. baseline_contract_sha256 与 3.1 产出匹配
```

### 9.1.3 Compatibility 不变量

```
1. 所有 9 个 CompatibilityDimension 都有合法 DimensionJudgment
2. 任何 status ∈ {INCOMPATIBLE, INSUFFICIENT_EVIDENCE} 的维度：
   a. 必须有 reasoning
   b. 必须有 blocking 标记
3. blocking=true 的维度不得进入 overall_status=viable 或 viable_with_conditions
4. overall_status 由确定性规则派生，不是 LLM 自由输出
5. insufficient_evidence 且维度为核心接口 → overall_status=needs_reanalysis
```

### 9.1.4 Variant 不变量

```
1. variant 数量 ∈ [1, 3]
2. 所有 variant 的 idea_id 指向同一个 IdeaContract
3. 每个 variant 的 target_hook 存在于 baseline_architecture_contract.modifiable_hooks
4. touches_protected_path 为 false
5. Variant 之间满足至少一条差异维度规则
6. 不包含文件级修改（files_to_modify、files_to_add、new_components 等 3.6 字段）
```

### 9.1.5 Handoff 不变量

```
1. handoff.source_context_id/version/SHA 匹配上游 research_context.json
2. handoff.idea_contract_sha256 匹配 idea_contract.json
3. handoff.variant_selection_sha256 匹配 variant_selection.json
4. handoff 不包含 unresolved blocking dimension
5. 除设计阻塞项外，所有 unresolved dimensions 必须分类为 experiment_resolvable 或 nonblocking_warning
6. next_stage=3.5_multi_variant_experiment_planner
```

## 9.2 unresolved_dimensions 分类

进入 3.5 前，所有 unresolved dimensions 必须分类：

```python
class UnresolvedDimension(BaseModel):
    dimension: CompatibilityDimension
    status: CompatibilityStatus
    classification: Literal[
        "design_blocking",      # 阻塞——必须在 3.4 解决
        "experiment_resolvable", # 可以在 3.5 实验验证中解决
        "nonblocking_warning",   # 已知但不阻塞
    ]
    reason: str

# 规则
# design_blocking → 不能进入 3.5
# experiment_resolvable → 可以进入 3.5（3.5 的 experiment plan 会包含验证步骤）
# nonblocking_warning → 可以进入 3.5（记录在 accepted_risks 中）
```

## 9.3 验证失败行为

```
1. schema error → 修复
2. evidence 缺失 → evidence_recovery（不新增事实，只修正引用）
3. contract 之间不匹配 → 报告具体冲突
4. 用户未确认 → 回到等待状态
```

# 11. 状态机

```
idle
  → waiting_for_idea_source               (无合法 idea source)
  → idea_source_routed                    (路径 A 完成)
  → candidates_resolved                   (路径 B 完成)

idea_source_routed / candidates_resolved
  → waiting_for_idea_confirmation         (等待用户确认 Idea)

waiting_for_idea_confirmation
  → idea_confirmed                        (用户 confirm/select_one)
  → blocked_no_confirmable_idea            (无合法 source 且用户未提供)

idea_confirmed
  → idea_contract_normalized
  → architecture_aligned
  → compatibility_analyzed

compatibility_analyzed
  → non_viable                            (overall_status=non_viable → 回到 idea_confirmed 或 stop)
  → needs_reanalysis                      (→ repository_reanalysis_request / paper_reanalysis_request)
  → variants_generated

variants_generated
  → waiting_for_variant_selection         (等待用户选择 Variant)

waiting_for_variant_selection
  → variants_selected                    (用户 select_multi)
  → blocked_no_variant_selected           (用户沉默，不自动选择)

variants_selected
  → validated
  → validation_failed                     (→ schema repair)

validated
  → handoff_generated
  → ready_for_3_5
```

禁止循环：
- 同一 gap 连续追问超过 2 次
- 总澄清轮数超过 3 轮
- LLM retry 超过 2 次

# 12. 用户交互协议

## 12.1 确认类型

| 操作 | 含义 |
|------|------|
| `confirm` | 用户确认系统理解 |
| `select_one` | 从候选中选一个（Idea） |

| `revise` | 用户修改系统理解 |

## 12.2 不该问用户的问题

```

❌ 要求用户确认 tensor shape
❌ 要求用户读代码
❌ "论文说 X，baseline 是 Y，你觉得兼容吗？"

✅ 论文有 3 个候选模块，你想迁移哪个？（select_one）
```

# 17. 风险与控制

## 风险 1：3.4 再次重复 3.1/3.2 的工作

控制：
- 3.4 只消费 3.1 的 `baseline_architecture_contract.json` 和 3.2 的 `paper_idea_sources.json`
- 不自行读仓库文件、不自行解析论文
- Permission profile 的 deny 项强制此约束

## 风险 2：用户沉默被当作授权

控制：
- `confirmation_status` 和 `selected` 字段必须有 `user_evidence_id`
- 推荐标记与确认状态分离
- 沉默时保持 blocked，不进入下一步

## 风险 3：CompatibilityAnalyzer 分数漂移

控制：
- 不使用 1-10 数值评分
- 使用结构化状态，整体结论由确定性规则派生
- retry loop 只修复 schema/evidence 问题

## 风险 4：3.4 产出被当作代码实现计划

控制：
- Variant 只包含架构级描述
- 不含文件级修改信息
- Handoff 到 3.5（Experiment Planner），代码 patch 留给 3.6

## 风险 5：LLM 声称兼容但未经验证

控制：
- DeterministicValidator 强制执行五类不变量
- 验证失败不能 handoff

# 18. 与上下游的接口

## 18.1 上游

3.4 的输入来自 Context Layer（3.2 Unified Research Context Agent + 可选的 3.3 ContextPatch）：

```
必需：
  research_context.json (stable)
  idea_transfer_handoff.json
  baseline_architecture_contract.json  (← 3.1)
  paper_idea_sources.json              (← 3.2)
  method_components.json               (← 3.2)
```

3.4 依赖的 artifact 名称必须与 3.1/3.2 的产出名称一致。不使用 `paper_method_components.json` 等变体名。

## 18.2 下游（3.4 → 3.5）

3.5 从 `idea_transfer_design_handoff.json` 得到：

- `confirmed_idea`：确认的 Idea 合约
- `transfer_analysis`：兼容性全貌
- `selected_variants`：用户选择的 Variant（架构级）
- `experiment_resolvable_dimensions`：可在实验中验证的未解决问题
- `accepted_risks`：用户接受的已知风险

3.5 不需要重新确认 Idea 或重新判断兼容性。

## 18.3 与 3.6（Patch Planner）的关系

```
3.4 → 架构级 Variant
3.5 → Experiment Plan（共享评价协议）
3.6 → 文件级 Patch Plan（files_to_modify、函数名、代码步骤）
3.7 → Approval + Patch Execution
```

3.4 明确不在 Variant 中包含文件级信息。3.6 是 Patch Planner 的独立阶段。
