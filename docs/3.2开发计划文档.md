# 0. 修订说明

## 0.1 为什么重写

旧版 3.2 将本阶段定义为 `Evidence-based Intent Clarifier`。该编号与当前渐进式纵向闭环不一致。

正式阶段顺序调整为：

```text
Step 3.1 Repository Intelligence
→ Step 3.2 Paper Intelligence
→ Step 3.3 Evidence-based Intent Clarifier
→ Step 3.4 Single Idea Confirmation
→ Step 3.5 Transferability Judge
→ Step 3.6 Experiment Planner
```

原因：

```text
系统必须先读完用户已经提供的论文和仓库材料，
才能知道哪些问题是真正缺失、值得向用户询问的问题。
```

因此：

- 原《3.2 Evidence-based Intent Clarifier 完整开发计划》顺延为 Step 3.3；
- 本文重新定义 Step 3.2；
- 不修改已经封存的历史提交；
- 在后续路线文档中记录阶段编号勘误和新入口关系。

## 0.2 与 3.1 同构但不机械复制

3.2 采用与 3.1 相同的 Agent-Native 基本公式：

```text
Agent 负责理解、导航、冲突解释和综合
Tool 负责执行受控动作
Skill 负责推荐阅读策略和停止条件
Policy 负责不可绕过的安全边界
Evidence Middleware 负责生成可验证引用
Validator 负责确定性兜底
Artifact 负责跨阶段交付
```

但论文与代码仓库不是同一种证据对象。

3.1 的主要证据是：

```text
固定 repository commit
repository-relative path
file SHA256
line range
snippet SHA256
```

3.2 的主要证据是：

```text
固定 PDF SHA256
固定 parser version/profile/model fingerprint
physical page index
section path
block / figure / table ID
content SHA256
```

## 0.3 本轮关键决策

1. Step 3.2 正式名称为 `Paper Intelligence`。
2. MVP 正式输入格式仅支持 PDF。
3. MinerU 是 P0 唯一正式 Paper Parser。
4. MarkItDown 不进入 3.2 PDF 主链。
5. MarkItDown 只作为后续通用文档适配器候选。
6. AlphaXiv 作为交互和导航参考，不作为核心依赖。
7. 用户 PDF 是论文正文事实的最高优先级来源。
8. arXiv HTML / LaTeX source 是可选补充来源。
9. Web 页面不能替代 PDF 支持论文内部精确事实。
10. 不建设向量数据库和全局 RAG 索引。
11. Agent 按 section/page/block 定向阅读，不一次吞入全文。
12. `paper_mentioned` 候选不能自动变成正式 baseline、dataset 或 metric。
13. 论文中的仓库 URL 只能作为 3.1 Discovery / Resolution 的候选输入。
14. Parser 复现与 LLM 语义复现分开验收。
15. 解析失败、页面缺失和证据不足必须显式输出，不生成伪摘要。

# 1. 执行摘要

## 1.1 阶段目标

Step 3.2 建立一个能够处理真实科研论文 PDF 的 Paper Intelligence Agent。

输入：

```text
用户上传的 PDF
可选 arXiv URL / DOI / 项目页
可选用户目标
可选 3.1 Repository Intelligence 结果
```

输出：

```text
论文来源和解析环境证明
页级与章节级结构
论文关键事实及其证据
核心方法和组件
数据与标签假设
训练目标与推理流程
实验设置与主要结果
baseline / dataset / metric 候选
代码、项目页、数据集和权重链接候选
潜在迁移点
论文限制、冲突与未知事项
交给 Step 3.3 的澄清候选
交给 Step 3.4 的论文 Idea 来源候选
```

所有 critical claim 必须绑定可重算的 `PaperEvidenceRef`。

## 1.2 架构公式

```text
Agent 负责：
  论文理解、阅读顺序、定向回读、冲突分析、语义综合

MinerU Tool 负责：
  PDF 解析、阅读顺序、公式、表格、版面、OCR、结构化输出

Paper Tools 负责：
  section/page/block 列举、读取、搜索、figure/table 获取

Skill 负责：
  推荐阅读方法、覆盖清单、候选语义、停止条件

Policy 负责：
  文件范围、网络范围、资源预算、只读和禁止执行

Evidence Middleware 负责：
  PDF/Parser/Page/Section/Block/Content 的证据绑定

Validator 负责：
  Evidence 重算、claim 规则、候选角色和跨 artifact 一致性
```

## 1.3 完整执行链路

```text
用户 PDF / arXiv URL
        ↓
Paper Source Intake
        ↓
PDF Source Attestation
        ↓
MinerU Parse Tool
        ↓
Parser Manifest + Parse Quality Report
        ↓
Paper Structure / Page & Block Store
        ↓
Paper Intelligence Harness
        ↓
自动加载 paper-analysis Skill
        ↓
Agent 按需调用：
  paper_list_sections
  paper_read
  paper_search
  paper_read_figure
  paper_read_table
  web_fetch（可选、deferred）
        ↓
Paper Evidence Index
        ↓
Agent Synthesis
        ↓
Strict Paper Artifacts
        ↓
Paper Evidence Validator
   ↓                 ↓
通过               有界 Repair
   ↓                 ↓
Step 3.3 Clarifier Handoff
Step 3.4 Single Idea Source Handoff
3.1 Repository Resolution Handoff
```

## 1.4 3.2 完成后应具备的能力

1. 对固定 PDF 生成固定来源指纹。
2. 锁定 MinerU 版本、backend、模型和 parser profile。
3. 保留 physical page 与 section/block 映射。
4. 检测空页、扫描页、乱码页、OCR 页和解析失败页。
5. 按章节和问题定向读取，不强制全文一次入模。
6. 提取论文研究问题、方法、组件、目标和假设。
7. 提取 baseline、dataset、metric、repo、asset 候选。
8. 区分 proposed method、used baseline、compared baseline 和 cited-only。
9. 不把 `paper_mentioned` 变成 selected parameter。
10. 所有关键事实具有可重算证据。
11. 表格指标引用具体 table/cell 或 block evidence。
12. 论文内部冲突输出 `conflicting`，不静默选边。
13. PDF 解析失败时不生成伪 paper summary。
14. 向 3.1 提交 repository link candidates，而不自行确认官方仓库。
15. 向 3.3 提交 evidence-backed clarification candidates。
16. 向 3.4 提交一个或多个 paper idea source candidates，但不自动确认 Idea。

# 2. 阶段定位与边界

## 2.1 3.2 解决的问题

3.1 已经回答：

```text
代码仓库是什么？
仓库固定在哪个 commit？
入口、依赖、配置和评价代码在哪里？
```

3.2 需要回答：

```text
论文真正研究什么问题？
论文提出的方法是什么？
方法由哪些组件构成？
依赖哪些数据、标签和训练假设？
实验使用了哪些 baseline、dataset 和 metric？
哪些只是被提到，哪些真正用于实验？
论文声称的贡献和限制是什么？
哪些机制可能成为异常检测迁移 Idea 来源？
论文中的代码、项目页和数据链接是什么？
哪些结论有正文、表格或图证据？
哪些信息缺失、冲突或解析不可靠？
```

## 2.2 3.2 不解决的问题

3.2 严格不做：

```text
不确认最终 baseline
不确认最终 dataset
不确认最终 metric
不确认 Single Idea
不判断 Idea 是否适合异常检测
不生成 ExperimentPlan
不生成 PatchPlan
不修改仓库代码
不安装论文代码依赖
不运行论文代码
不运行 LaTeX 编译
不下载论文数据集或权重
不自动确认论文中的 GitHub URL 是官方仓库
不建立向量数据库
不构建跨论文知识库
不做完整文献综述
不自动生成多个科研方向进行投票
```

这些分别属于：

```text
Step 3.1 Repository Intelligence
Step 3.3 Evidence-based Intent Clarifier
Step 3.4 Single Idea Confirmation
Step 3.5 Transferability Judge
Step 3.6 Experiment Planner
后续 Literature Review / Knowledge Memory
```

## 2.3 正式支持范围

### MVP 深度支持

```text
本地 PDF
arXiv 下载得到的 PDF
原生文本 PDF
扫描 PDF
中英文及混合语言论文
单栏、多栏和常见复杂布局
公式、表格、图和附录
```

### MVP 可选补充

```text
arXiv HTML
论文项目主页
作者主页
公开 supplementary PDF
AlphaXiv 可读取页面
```

### 不进入 3.2 封板

```text
DOCX / PPTX / XLSX 正式支持
EPUB
音频和视频
任意网页作为主论文源
LaTeX source 完整支持
跨论文向量检索
浏览器自动化
付费墙论文自动获取
```

## 2.4 输入来源优先级

```text
1. 用户提供的固定 PDF
2. 用户确认的 arXiv PDF 版本
3. arXiv HTML / LaTeX source
4. 官方论文项目页 / 作者页
5. AlphaXiv 等第三方论文页面
6. 搜索结果摘要
```

只有 1–3 可支持论文内部 critical facts。

4 可支持：

```text
论文—项目页关联
论文—仓库关联候选
论文—数据集页面关联
```

5–6 只作为：

```text
导航线索
关联候选
补充入口
```

不能支持：

```text
精确方法行为
精确实验设置
精确结果数值
论文正式结论
```

## 2.5 与 3.1 的边界

3.2 可以从论文中提取：

```text
GitHub URL
项目主页
代码仓库名称
实现作者或组织
代码可用性声明
```

但只能生成：

```text
repository_link_candidates.json
```

随后交给 3.1：

```text
Discovery
Candidate Verification
Resolution
Acquisition
Attestation
```

3.2 不重复建设：

```text
GitHub metadata 验证
clone
commit pinning
repository attestation
repository source evidence
```

## 2.6 与 3.3 的边界

3.2 输出论文事实和未知项。

3.3 才负责：

```text
将 paper facts + repository facts + 用户输入合并
识别影响任务决策的缺口
只追问真正缺失的问题
形成 clarified task contract
```

3.2 不主动向用户发出任务澄清问卷。

## 2.7 与 3.4 的边界

3.2 可以输出：

```text
PaperIdeaSourceCandidate
```

含义是：

```text
论文中的某个组件或机制可能成为后续 Idea 来源。
```

它不是：

```text
已确认的 SingleIdea
已批准的代码修改方案
已判断可迁移的方法
```

只有用户在 3.4 明确确认后，才能生成：

```text
single_idea.json
idea_confirmation.json
```

# 3. 前置条件、路线勘误与启动模式

## 3.1 必需的 3.0 Tool Foundation

真实 PDF 流程至少依赖：

```text
ToolSpec / ToolResult / ToolContext / ToolRegistry
PermissionEngine
workspace-scoped Filesystem Tools
Skill Loader
Tool Audit Log
Evidence Middleware 基础接口
Model Router
ArtifactStore / EventStore
```

不强制要求通用 ProcessTool 暴露给 Paper Agent。

MinerU 应由：

```text
受控 DocumentParseTool / Parser Provider
```

调用，不允许 Agent 自行拼 MinerU shell 命令。

## 3.2 必需的 3.1 输出

3.2 可以在没有仓库的情况下独立运行。

若已经完成 3.1，可选读取：

```text
repository_summary.json
entrypoints.json
dependency_evidence.json
evaluation_contract_draft.json
environment_context.json
uncertainties.json
repository_source.json
repository_validation_report.json
```

用途：

```text
交叉识别论文方法与代码实现是否一致
将论文仓库链接和已解析仓库进行对齐
向 3.3 输出跨来源冲突
```

3.2 不能修改 3.1 artifacts。

## 3.3 阶段编号勘误

新增路线状态记录：

```yaml
route_revision:
  previous_step_3_2: evidence_based_intent_clarifier
  corrected_step_3_2: paper_intelligence
  clarified_task_stage: "3.3"
  effective_from: <implementation commit>
  history_rewritten: false
```

不修改：

```text
3.0 sealed commit
3.1 sealed plan 历史提交
```

只更新：

```text
当前路线文档
README 当前路线
3.2 plan
未来 3.3 plan
```

## 3.4 三种启动模式

### Full Mode

具备：

```text
MinerU locked provider
Paper Tools
PermissionEngine
Skill Loader
Evidence Middleware
LLM model profiles
optional web_fetch
```

支持：

```text
PDF parse
Agent analysis
optional arXiv/project page fetch
cross-source validation
```

### Local PDF Mode

WebFetch 尚不可用时：

```text
只支持用户本地 PDF
不补充 arXiv HTML / 项目页
外部链接只记录，不访问
```

该模式可以用于 3.2 核心封板。

### Fixture-Only Mode

若以下任一缺失：

```text
PermissionEngine
MinerU locked provider
Evidence Middleware
Paper Validator
```

则只允许：

```text
录制 parser fixtures
手工构造 page/block fixtures
fake LLM
```

不能处理真实用户 PDF，不能用于最终封板。

# 4. 总体架构

## 4.1 分层结构

```text
PaperIntelligenceRequest
        ↓
Paper Source Intake / Attestation
        ↓
DocumentParseTool
        ↓
MinerUProvider
        ↓
Canonical Paper Store
        ↓
Paper Intelligence Agent Harness
        ↓
Skill Loader + Tool Registry + Permission Profile
        ↓
Analysis / Synthesis
        ↓
Paper Evidence Middleware
        ↓
Strict Paper Artifacts
        ↓
Paper Evidence Validator
        ↓
Repair Loop
        ↓
3.1 / 3.3 / 3.4 Handoff
```

## 4.2 组件职责

| 组件 | 负责 | 不负责 |
|---|---|---|
| Source Intake | 接收、哈希、识别 PDF | 解释论文内容 |
| MinerUProvider | 解析版面、文本、公式、表格和图 | 选择科研结论 |
| Canonicalizer | 生成稳定 page/section/block store | 推断语义 |
| Agent | 按需阅读、解释和综合 | 伪造 parser 结果 |
| Skill | 推荐阅读顺序、覆盖和停止条件 | 强制安全 |
| Policy | 文件/网络/工具/预算边界 | 理解论文 |
| Evidence Middleware | 生成证据引用 | 判断 claim 是否正确 |
| Validator | 验证 claim 和 artifacts | 自动创造缺失事实 |
| Repair Loop | 定向回读和修正 | 放宽权限和 parser 约束 |

## 4.3 不采用的架构

3.2 不建设：

```text
固定全文分块 → embedding → vector DB → top-k RAG
一次性将全文塞进单个 LLM prompt
针对论文字段的几十个硬编码 extractor
浏览器自动化抓取 AlphaXiv
MinerU 失败后静默切换另一解析器
```

未来若真实回归证明需要，可以增加：

```text
跨论文检索索引
citation graph
LaTeX source parser
通用 office document adapters
```

但不成为 3.2 MVP 前置条件。

# 5. Parser Provider 决策

## 5.1 MinerU 的正式定位

3.2 P0 唯一正式解析引擎：

```text
MinerUProvider
```

选择理由：

```text
科学 PDF 版面解析
多栏阅读顺序
公式转 LaTeX
表格转 HTML / 结构化表格
图、caption、footnote 提取
扫描 PDF / 乱码 PDF 检测
OCR
结构化 Markdown / JSON / 中间结果
CPU / GPU backend
```

正式实现必须锁定：

```text
MinerU package version
provider implementation version
backend
model ID
model weight SHA / immutable revision
OCR profile
language hints
effort / quality profile
canonicalizer version
container/environment fingerprint
```

禁止只记录：

```text
parser = MinerU
```

## 5.2 MinerU backend 选择

MVP 建议定义两个受控 profile：

### `paper_pdf_pipeline_v1`

```text
目标：可重复、资源要求低、适合原生文本 PDF
backend: pipeline
network: disabled
OCR: auto
image analysis: basic
```

### `paper_pdf_high_accuracy_v1`

```text
目标：复杂布局、扫描件、图表较多论文
backend: hybrid 或团队锁定的高精度 backend
effort: high
network: 按部署方式明确
OCR: auto / configured
```

两个 profile 均必须经过 fixture regression。

Agent 不可以自定义任意 backend 参数。

Profile 选择由确定性 preflight 建议：

```text
原生文本、低乱码、表格普通 → pipeline_v1
扫描页、乱码、复杂图表明显 → high_accuracy_v1
```

用户或策略可以覆盖选择，但必须记录。

## 5.3 MarkItDown 的定位

MarkItDown 不作为 3.2 PDF fallback。

理由：

```text
MinerU 已覆盖 PDF 和主流 Office 格式
3.2 需要高精度 page/layout/formula/table evidence
静默 parser fallback 会改变证据语义
MarkItDown 的通用 convert() 入口权限面较宽
```

保留的借鉴点：

```text
DocumentConverterProvider 的轻量统一接口
按格式拆分 optional dependencies
plugins 默认关闭
使用最窄输入接口
convert_local / convert_stream 的安全思想
低资源普通文档转换模式
```

后续可新增：

```text
GenericDocumentAdapter / MarkItDownProvider
```

用于：

```text
普通 DOCX 技术报告
PPT 演示材料
CSV/JSON/XML 文档
非论文型输入
```

它不参与 3.2 PDF 质量验收。

## 5.4 Parser fallback 原则

```text
MinerU parse success
→ 进入 Analysis

MinerU degraded
→ 只对完整、可验证 blocks 允许 confirmed claim
→ 缺失区域标记 parse uncertainty

MinerU failed
→ parse_failed
→ 可请求另一版 PDF / arXiv HTML / supplementary PDF
→ 不生成 paper_summary.json
```

禁止：

```text
MinerU failed
→ 静默 MarkItDown
→ 继续声称摘要完整
```

## 5.5 Parser license 与部署检查

实施前必须记录：

```text
MinerU 当前 license 文件 SHA
模型及 OCR 组件 license
是否允许比赛部署
是否允许服务器处理用户论文
是否包含额外使用条件
```

License 结论不能只依赖 README 摘要。

# 6. AlphaXiv 与 Web 辅助能力

## 6.1 借鉴 AlphaXiv 的内容

借鉴的是产品交互范式：

```text
按章节导航
论文问答
局部笔记
相关工作入口
代码和数据链接
图表聚焦查看
基于当前位置的讨论
```

AutoAD 对应设计：

```text
paper_list_sections
paper_read(section/page/block)
paper_search
paper_read_figure
paper_read_table
Evidence-backed question answering
局部 observation / notes
repository link handoff
```

## 6.2 不把 AlphaXiv 作为核心依赖

原因：

```text
第三方页面结构可能变化
不保证稳定 API
平台生成摘要不等于论文原文
可能覆盖或改写原始表述
抓取失败不应阻塞 PDF Reader
```

因此：

```text
AlphaXiv = optional deferred WebEvidence
PDF = primary PaperEvidence
```

## 6.3 WebFetch 允许用途

```text
arXiv HTML
官方论文项目页
作者主页
公开 supplementary page
AlphaXiv 页面
```

允许支持：

```text
论文—项目页关联
仓库链接候选
数据集链接候选
supplementary material URL
```

不允许支持：

```text
论文精确公式
精确实验数值

```

沿用 3.1 SSRF Policy：

```text

拒绝 loopback/private/link-local/metadata endpoints
每次重定向重新检查
限制响应体大小
限制 MIME
限制重定向次数
不下载可执行文件
不执行页面脚本
```

## 6.5 不建设浏览器自动化

3.2 不需要：

```text
click_mouse
scroll
screenshot OCR
网页登录
动态页面模拟
```

若普通 `web_fetch` 无法读取 AlphaXiv，则记录：

```text

```

不阻塞 Paper Intelligence。

## 8.5 paper-analysis 推荐策略

```text
1. 检查 parse quality 和缺失页
2. 读取 title / abstract / introduction
3. 找出研究问题和贡献声明
4. 定位 method / approach 章节
5. 分解核心组件、输入输出和训练目标
6. 定位 experiments / implementation details
7. 提取 dataset、baseline、metric mention roles
8. 读取主结果表和关键消融
9. 读取 limitations / appendix 中的约束
10. 定向回读冲突项
11. 判断最低覆盖是否满足
12. 请求进入 synthesis
```

这是推荐策略，不是固定 Python pipeline。

## 8.6 核心禁止项

```text

不得执行代码或 LaTeX
不得根据论文自动选择实验参数
不得把 compared baseline 当作 selected baseline

不得使用 Web 摘要证明论文内部事实
不得生成未有证据的结果数值
不得绕过 parse quality warning
```

# 12. MinerU Parse 与 Canonical Paper Store

## 12.1 Parse Attempt 不可覆盖

```text
runs/<run_id>/paper/parse_attempts/<parse_attempt_id>/
```

已有目录时拒绝覆盖。

Parser 变更、profile 变更或 source 变更：

```text
创建新 parse attempt
```

不修改旧结果。

## 12.2 原始输出与 Canonical 输出分离

```text
raw/
  MinerU 原始 JSON / Markdown / images

canonical/
  pages.jsonl
  blocks.jsonl
  sections.json
  figures.json
  tables.json
  references.json
```

原因：

```text
MinerU 输出格式可能随版本变化
AutoAD 需要稳定内部 contract
```

## 12.3 Canonicalizer

确定性组件负责：

```text
页面编号统一
block ID 生成
section path 生成
reading order 固定
canonical whitespace
content SHA
figure/table/reference identity
header/footer 标记
```

Block ID 建议：

```text
sha256(source_id + parse_attempt_id + page_index + order_index + content_sha)
```

## 12.4 Parse quality gate

进入 Analysis 的最低条件：

```text
Parser status != failed
标题/摘要区域可读取或明确 unknown
方法和实验页未全部失败
canonical output 可校验
page/block 映射完整
```

`accepted_with_warnings` 允许进入，但 warnings 必须注入 Agent context。

## 12.5 空页

空页区分：

```text
真正 blank
全图页但无文本
parser failed to extract
excluded appendix page
```

不得将所有空文本页都标记 `blank`。

## 12.6 乱码检测

指标可包含：

```text
replacement character ratio
control character ratio
invalid unicode ratio
low alphabetic ratio
repeated glyph ratio
```

阈值必须 fixture 校准。

## 12.7 扫描 PDF

扫描 PDF 应触发：

```text
OCR profile
parsed_with_ocr page status
OCR language audit
```

OCR 结果可以支持 confirmed claim，但 parse confidence 和 warning 必须保留。

## 12.8 表格

表格 canonicalization 至少保留：

```text
caption
header rows
cell matrix
merged cell indication
footnotes
page range
```

若表格无法可靠结构化：

```text
parse_confidence=low
cell-level result claim 不允许 confirmed
```

## 12.9 公式

MVP 保存：

```text
LaTeX text
page/block identity
```

不要求建立公式 AST。

## 12.10 图

MVP 主要使用：

```text
caption
邻近正文
figure image artifact
```

Agent 不自动使用视觉模型解读所有图。

对关键结构图，可按需调用后续多模态模型，但必须形成独立 evidence 和预算。

本版先预留，不作为 P0 必需。

# 13. Paper Evidence Middleware 与 Workspace

## 13.1 Evidence Index

```text
runs/<run_id>/paper/evidence_index.jsonl
```

append-only。

## 13.2 Evidence 自动生成

每次 `paper_read`：

```text
验证 source / parse attempt
解析 block IDs
重算 content SHA
生成 PaperTextEvidenceRef
追加 Evidence Index
```

每次 `paper_read_table`：

```text
记录 table ID
记录 cell coordinates
重算 table content SHA
生成 PaperTableEvidenceRef
```

## 13.3 Evidence 不由 Agent 手工填写

Agent structured output 只能引用：

```text
Evidence Index 中已存在的 evidence_id
```

Agent 不能直接提供：

```text
page=5, hash=...
```

由 Middleware 生成。

## 13.4 Evidence scope

Evidence 必须匹配：

```text
source_id
PDF SHA
parse_attempt_id
parser profile SHA
canonical output SHA
```

旧 parser attempt evidence 不可用于新 parser attempt artifact，除非显式声明并通过 cross-attempt equivalence validator。

MVP 不支持跨 attempt 混用。

## 13.5 Web evidence

Web evidence 与 paper evidence 分库存储或通过 source_kind 区分。

规则：

```text
Web evidence 不能支持 paper primary claim
```

## 13.6 Redaction

论文通常是公开内容，但用户可能上传未公开稿件。

必须支持：

```text
source_visibility = public | private_user_provided
```

Private paper：

```text
不上传到未经批准的第三方 parser/API
模型 provider 必须符合用户数据策略
不在日志中输出全文
不在错误信息中输出敏感段落
```

# 14. Agent Harness、模型路由与阶段控制

## 14.1 状态机

```text
request_received
source_intake
source_attested
parser_preflight
parsing
parse_quality_review
analysis_running
synthesis_running
validation_running
repair_running
handoff_ready
success | partial_success | parse_failed | blocked | failed
```

## 14.2 强制状态边界

```text
没有 source attestation → 不得 parse
parse quality rejected → 不得 analysis
没有 Evidence Index → 不得 synthesis
validation 未通过 → 不得 success
```

## 14.3 Agent 自主性

Agent 可以：

```text
调整阅读顺序
搜索关键词
选择是否读取表格、图或 reference
定向回读冲突
在预算内补证据
```

Agent 不能：

```text
跨 stage
选择任意 parser profile
放宽 parse gate
绕过 candidate 语义
修改 selection_status
扩大权限
无限循环
```

## 14.4 AnalysisControlSignal 审计

每轮 Agent 输出 `PaperAnalysisControlSignal`。

Harness 追加写：

```text
paper_analysis_control_signals.jsonl
```

记录：

```text
iteration
raw signal
coverage snapshot
budget remaining
harness decision
accepted/rejected
rejection reasons
```

不保存隐藏思维链。

## 14.5 Agent 请求 Synthesis

只有：

```text
decision=synthesis_ready
```

Harness 再检查最低 coverage。

`checked_unknown` 合法，但必须有搜索/读取记录和 unknown reason。

## 14.6 强制 Synthesis

以下触发：

```text
analysis call budget exhausted
input token budget exhausted
max_no_progress_cycles reached
连续两轮 new_evidence_count=0
```

强制结果：

```text
forced_synthesis=true
未覆盖项 → unknown
status 至少 partial_success
```

若 source identity 或 parse quality 无法证明，则 failed/blocked，不得 partial success。

## 14.7 模型逻辑 Profiles

```yaml
paper_fast_v1:
  purpose:
    - section signal extraction
    - candidate mention extraction
    - light schema repair
  capabilities:
    - structured_output
    - long_context

paper_primary_v1:
  purpose:
    - method understanding
    - conflict resolution
    - synthesis
    - evidence repair
  capabilities:
    - strong_scientific_reasoning
    - structured_output
    - tool_use
    - long_context

paper_fallback_v1:
  purpose:
    - provider outage
    - one structured-output retry
  capabilities:
    - strong_scientific_reasoning
    - structured_output
    - tool_use
```

具体 provider/model ID 由配置映射，不写入 schema 常量。

## 14.8 路由

```text
轻量 section/candidate extraction → paper_fast_v1
核心 method analysis             → paper_primary_v1
Synthesis / Repair                → paper_primary_v1
Primary failure                   → capability gate → fallback 最多一次
```

## 14.9 Resume Fingerprint

```text
request SHA
source PDF SHA
source attestation SHA
parser manifest SHA
canonical output SHA
parse quality SHA
Skill SHAs
permission profile SHA
model config SHA
budget profile SHA
artifact schema versions
```

任一变化：

```text
旧 analysis 不可直接 resume
```

## 15.5 Conflict handling

```text

正文和表格结果不一致
不同 PDF 版本内容不同
论文和项目页描述不同

```

输出：

```text
status=conflicting

不自动选一个为真
```

## 15.6 Unknown handling

```text

未公开的代码实现
图片无法可靠解析的模块
表格结构损坏的数值
```

输出：

```text
unknown
missing information
impact
recommended next source
```

# 17. Paper Evidence Validator

## 17.1 目标

确定性验证：

```text
Agent 输出是否真正由固定 PDF 和固定 parser output 支持
```

## 17.2 建议目录

```text
src/autoad_researcher/paper_intelligence/
├── validate.py
├── evidence.py
├── parser_validation.py
├── consistency.py
├── candidate_validation.py
└── handoff_validation.py
```

## 17.3 Source 与 Parser 验证

```text
PDF 文件存在
PDF SHA 匹配
Parser manifest 匹配 source
provider/profile/model/environment 字段完整
canonical output SHA 可重算
parse quality report 匹配 canonical output
page count 一致
block IDs 唯一
```

## 17.4 Evidence 验证

对 PaperTextEvidenceRef：

```text
source_id 匹配
PDF SHA 匹配
parse attempt 匹配
page 存在
block 存在
section mapping 合法
content SHA 可重算
tool call 存在
```

对 TableEvidence：

```text
table 存在
cell coordinates 合法
content SHA 可重算
```

对 FigureEvidence：

```text
figure 存在
caption/image SHA 合法
```

## 17.5 Claim 状态规则

### confirmed

```text
evidence_ids 非空
至少一个 paper_primary evidence 直接支持
parse quality 不允许该范围为 failed
```

### inferred

```text
至少一个 evidence
rationale_summary 非空
不得使用“论文明确证明”等强措辞
```

### conflicting

```text
至少两个不同 evidence
冲突内容可识别
```

### unknown

```text
不得伪造直接 evidence
必须说明缺失信息或 parse issue
```

## 17.6 Candidate 规则

```text
selection_status 必须等于 paper_mentioned
mention_role 必须合法
compared_baseline 不得出现在 selected baseline
cited_only 不得标记 proposed_method
metric 必须有 experiment/result evidence
repository URL 必须有正文或项目资源证据
```

## 17.7 表格结果规则

数值 claim：

```text
有具体 cell evidence
row/column 可定位
数值可从 canonical table 重算
unit 一致
```

表格 parse confidence=low 时：

```text
不能 confirmed
```

除非正文提供独立证据。

## 17.8 Web 证据规则

自动拒绝：

```text
AlphaXiv 摘要支持精确方法 claim
项目页支持论文结果数值
搜索摘要支持论文 baseline 角色
```

Web 只支持：

```text
association lead
resource link
```

## 17.9 跨 Artifact 一致性

```text
paper_summary identity == PaperSource
all evidence source == current source
all candidate evidence exists
method component claims exist
repository links are unresolved unless 3.1 result supplied
idea sources remain unconfirmed
handoff only references existing artifacts
```

## 17.10 与 3.1 一致性

若输入 3.1 结果：

```text
paper repo URL candidate vs resolved repository URL
paper method name vs repository README/entrypoints
paper dependencies vs repository evidence
paper evaluation description vs evaluation_contract_draft
```

冲突写入：

```text
cross_source_conflicts.json
```

不能由 Validator 自动选择论文或代码为真。

## 17.11 必须失败

```text
critical confirmed claim 无 paper evidence
PDF SHA 不一致
parser profile 不一致
page/block/content SHA 不匹配
paper_mentioned 被改成 selected
cited method 被写成本论文贡献
result 数值无法定位
parse rejected 仍生成 summary
artifact SHA 不一致
```

### I. Compared Baseline

论文比较 PatchCore：

```text
mention_role=compared_baseline
selection_status=paper_mentioned
```

### J. Proposed Method vs Cited Method

确保 related work 方法不成为 main method。

### K. Dataset and Metric

识别实验数据集和报告指标，但不 selected。

### L. Abstract / Body Conflict

输出 conflicting。

### M. Missing Implementation Detail

输出 unknown，不猜超参数。

### N. Explicit Limitation

限制进入 paper_summary 和 Idea source warning。

### O. Repository URL

输出 unresolved repository candidate。

### P. Paper Prompt Injection

正文包含恶意工具指令，Agent 不执行。

## 23.3 Cross-source Fixtures

### Q. Paper and Repository Aligned

论文组件在仓库中有对应实现。

### R. Paper and Repository Diverged

仓库 README / code 与论文方法不同。

### S. Repository Not Available

Paper Reader 独立成功。

## 23.4 Version Fixtures

### T. Same PDF, Same Parser Lock

canonical hashes 稳定。

### U. Same PDF, New Parser Version

resume invalidated，新 parse attempt。

### V. Paper v1 vs v2

source IDs 和 evidence 不可混用。

## 23.5 Web Fixtures

### W. arXiv HTML Available

只作为 supplementary evidence。

### X. AlphaXiv Unavailable

不阻塞 PDF analysis。

### Y. SSRF Redirect

WebFetch deny。

# 36. 最终执行原则

```text
PDF Source 负责固定，不负责解释
Parser 负责提取，不负责科研判断
Agent 负责理解，不负责宣告无证据事实
Skill 负责方法，不负责强制安全
Policy 负责边界，不负责论文语义
Evidence 负责定位，不负责结论正确性
Validator 负责证明，不负责补写事实
Repair 负责定向修正，不允许无限重读
Candidate 负责表达论文提及，不等于用户选择
Paper Reader 负责准备事实，不提前代替 Clarifier、Idea Judge 或 Planner
```
