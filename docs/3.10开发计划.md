## 0. 阶段定义

Step 3.10 是第三阶段（3.1–3.9）的系统验收门，不新增研究功能。第一轮目标是把已完成阶段接入顶层验收：可统一编排、可生成验收报告、可在缺失证据或 SHA 不匹配时 fail-closed。

验收分三层：

| 层 | 内容 | 本轮状态 |
|----|------|----------|
| L1 构件验收 | 各阶段 artifact ref、handoff、SHA 链结构可验证 | 本轮实现 |
| L2 流程验收 | Orchestrator 按固定顺序生成验收 manifest 和报告，异常 fail-closed | 本轮实现 |
| L3 真实任务验收 | 真实 PDF + repo + DeepSeek + GPU/MVTec + patch + metrics + final report | 后续阶段，preflight 通过后才允许运行 |

L1/L2 不调用真实 LLM、不运行 GPU、不读取 MVTec、不应用 patch、不执行外部仓库代码。L3 真实科研流程必须等真实 preflight 条件满足后另行启动。

## 1. 已完成前置

3.1–3.9 的主要 schema、服务、测试和阶段实现已经完成。3.10 不再追踪“3.8/3.9 missing contracts”；当前剩余工作只属于顶层验收基础设施。

固定 Stage 3 顺序为：

```text
intake → repository_intelligence → paper_intelligence → research_context → transfer_design → experiment_planner → patch_planner → patch_applicator → runner_execute → results_analysis → final_report
```

任何阶段必需 artifact 缺失、schema 不合规或 artifact chain SHA 不匹配，都必须返回 `blocked` 或 `failed`，不得 silently continue。

## 2. 本轮 L1/L2 交付

### 2.1 Schema

新增 3.10 acceptance contracts：

```text
Stage3AcceptanceManifest
Stage3AcceptanceStageRecord
EndToEndRunReport
ArtifactChainValidationReport
ArtifactChainBinding
SecurityGateReport
Stage3AcceptanceResult
Stage3AcceptanceRequest
Stage3ProviderConfig
```

约束：

- 所有 schema 使用 `extra="forbid"`。
- artifact ref 只保存 `relative_path`、`sha256`、`artifact_type`，不嵌入裸数据。
- stage 顺序必须与 canonical order 完全一致，重复或未知 stage 必须拒绝。
- successful result 不得包含 `failed_stage` 或 `failure_reason`。

### 2.2 Orchestrator

`pipeline/orchestrator.py` 在 L1/L2 模式执行 deterministic structural orchestration：

- 创建 `runs/{run_id}/stage3_acceptance/`。
- 为每个阶段登记 structural acceptance marker。
- 验证必需 artifact 是否存在。
- 生成 artifact chain validation。
- 生成 acceptance manifest、E2E report、security gate report 和 release candidate markdown。
- 若缺 artifact，返回 `blocked_missing_artifact`。
- 若 SHA chain mismatch，返回 `failed_sha_chain_mismatch`。

### 2.3 CLI

新增：

```bash
autoad stage3-acceptance --run-id <run_id> --runs-root <runs_root> --mode l1-l2 --json
```

L3 暂只支持 preflight-only：

```bash
autoad stage3-acceptance --run-id <run_id> --mode l3-preflight --json
```

preflight 若缺 `base_url` 或 API key 环境变量，返回 `blocked`；即使 preflight 条件齐全，也不执行真实实验。

## 3. DeepSeek 配置边界

DeepSeek v4 flash 只作为后续 L3/provider 配置，不参与 L1/L2 判定。

默认 provider config：

```text
provider = deepseek
model_id = deepseek-v4-flash
api_key_env = DEEPSEEK_API_KEY
structured_output_required = true
```

API key 只能来自环境变量，禁止写入 repo、脚本、日志或 artifact。L1/L2 不读取 Claude Code / VSCode 扩展配置。

## 5. L3 启动前置

进入真实 L3 前需要用户确认或提供：

- DeepSeek `base_url`
- API key 环境变量名和值所在环境
- `papers/patchcore.pdf`
- `workspace/repos/patchcore-inspection` 固定 commit
- MVTec AD bottle dataset 路径
- GPU/env lock 可用性

在这些条件满足前，3.10 只能声明 L1/L2 acceptance foundation ready，不能声明完整科研流程 sealed。
