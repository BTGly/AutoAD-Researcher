# 0. 阶段定义

Step 3.10 不是新功能阶段，而是**第三阶段（3.1–3.9）的系统验收门**。只有通过本步骤的验收，才能声明：

```
Stage 3 Vertical Research Loop accepted and sealed
```

验收分为三层：

| 层 | 内容 | 判定标准 |
|----|------|----------|
| L1 构件验收 | 每个阶段的 artifact、handoff、SHA 链独立可验证 | 所有 handoff 有正式 schema、SHA 双向闭合 |
| L2 流程验收 | PipelineController 或 Orchestrator 按正确顺序调用各阶段，异常时 fail-closed | 一次 `run()` 调用产出完整 artifact 树 |
| L3 真实任务验收 | 一条真实任务（PDF + repo + patch + command + metric + report）走通 | 验收清单全部完成 |

L1 和 L2 可以在任何环境执行。L3 必须在配备 GPU 的 AutoDL 容器内执行。

### 1.2a 顶层编排器（`pipeline/orchestrator.py`）

当前状态：**11 行桩代码**（仅有 `class Orchestrator: def run(): pass`）。需实现一条 `run()` 调用，依次串联：

```
0. Intake (input_task.yaml + source_manifest.json)
    ↓
1. Repository Intelligence  →  repo_summary.json  (7 artifacts)
    ↓
2. Paper Intelligence       →  paper_summary.json (7 artifacts)
    ↓
3. Context Assembly         →  research_context.json
    ↓
4. Idea & Transfer Design   →  idea_transfer_design_handoff.json
    ↓  [用户确认 Variant]
5. Experiment Planner       →  experiment_planner_handoff.json  (9 artifacts)
    ↓  [用户确认实验计划]
6. Patch Planner            →  patch_plan.json + patch.diff
    ↓  [用户审批 Patch]
7. Patch Applicator         →  PatchRunnerHandoff v2
    ↓
8. Runner Intake + Execute  →  ExecutionUnitRecord + AttemptRecord + ResourceUsageReport
    ↓
9. Results Analysis         →  PairedMetricObservation + ValidityReport + BudgetAssessment
    ↓
10. Report Generator        →  FinalReport + comparison_report.md
```

各阶段间通过 `ArtifactStore` 传递，handoff 只包含 SHA 引用，不允许传递裸数据。

### 1.2b CLI 子命令扩展（`cli.py`）

当前 CLI 缺少 3.4–3.9 的直接子命令。需新增：
- `autoad transfer-design` — 从 research_context 运行 3.4 Transfer Design
- `autoad experiment-plan` — 从 3.4 handoff 运行 3.5 Experiment Planner
- `autoad patch-plan` — 从 3.5 handoff 运行 3.6 Patch Planner
- `autoad patch-apply` — 审批后运行 3.7 Patch Applicator
- `autoad runner-execute` — 从 PatchRunnerHandoff v2 运行 3.8 实验执行
- `autoad analyze` — 从 execution 运行 3.9 Results Analysis

每个子命令保持 `PipelineController` / `Orchestrator` 作为内部实现路径，支持 `--json` 输出。

### 1.2c main.py 入口（`main.py`）

当前 `main()` 直接调用 stub Orchestrator。需改为解析 CLI args，根据输入类型路由到对应 pipeline。

# 2. E2E 验收场景

## 2.1 固定任务

验收使用内部 benchmark 任务：

| 参数 | 值 |
|------|-----|
| 论文 | **"Towards Total Recall in Industrial Anomaly Detection"** (PatchCore, CVPR 2022) |
| Baseline 仓库 | `workspace/repos/patchcore-inspection/` (固定 commit `7f5e9a4`) |
| Dataset | MVTec AD **bottle** 类别 |
| 目标 Variant | 修改 backbone 或特征融合层（具体由 3.4 Transfer Design 产出） |
| GPU | 单卡 NVIDIA (H100/L40S/A100) |
| 环境 | `configs/benchmarks/environments/patchcore_linux_gpu/` |

## 2.2 验收路径

```
真实 PDF papers/patchcore.pdf
  → 3.2 Paper Intelligence 解析
  → 与 3.1 Repository Intelligence 结果汇入 Research Context
  → 3.4 Transfer Design 生成 1–3 Variant
  → 用户选择 1 个 Variant
  → 3.5 Experiment Planner 生成实验计划（含 smoke + 1 seed full）
  → 用户确认计划
  → 3.6 Patch Planner 生成代码修改
  → 用户审批 patch
  → 3.7 Patch Applicator 应用 patch 到 workspace
  → 3.8 Runner Intake + 执行 smoke test
  → 3.9 Analysis 解析指标、评估有效性、生成对比报告
  → 最终报告输出到 runs/{run_id}/final_report.json
```

# 3. 验收标准

## 3.1 必须产出物

| # | 产出物 | 格式 | 验证点 |
|---|--------|------|--------|
| 1 | `stage3_acceptance_manifest.json` | JSON | 包含所有 stage + handoff SHA + artifact 列表 |
| 2 | `end_to_end_run_report.json` | JSON | run_id、状态、各阶段结果、失败原因 |
| 3 | `artifact_chain_validation.json` | JSON | 每对 handoff 的 SHA 双向闭合验证 |
| 4 | `security_gate_report.json` | JSON | ProcessTool 白名单、路径逃逸检测、权限审计 |
| 5 | `scientific_validity_report.json` | JSON | ScientificValidityReport 合同 |
| 6 | `comparison_report.md` | Markdown | 人类可读的 baseline-vs-variant 对比 |
| 7 | `final_report.json` | JSON | 结构化最终报告（含结论 + next_run_proposals） |
| 8 | `demo_script.md` | Markdown | 演示脚本（含预期输出和截图指引） |
| 9 | `demo_runbook.md` | Markdown | 从零开始的安装/运行/排错手册 |
| 10 | `release_candidate_report.md` | Markdown | 封板判断 + 已知问题 + P1 清单 |

## 3.2 必须满足的条件

### 不可用 fixture 替代的要素

- [ ] 一个**真实 PDF**（`papers/patchcore.pdf`）
- [ ] 一个**真实 Git 仓库**和固定 commit（`workspace/repos/patchcore-inspection/`）
- [ ] 一次**真实用户确认**（Variant 选择 + 实验计划确认 + Patch 审批）
- [ ] 一个**真实代码修改**（patch.diff 应用到仓库）
- [ ] 一次**真实受控命令执行**（`subprocess.run(shell=False)` 经 PermissionEngine 门控）
- [ ] 一个**真实指标文件**（从 smoke 执行输出的 JSON/CSV 解析）
- [ ] 一个成功、失败或部分成功的**正式报告**（含 conclusion）
- [ ] **完整 artifact SHA 链**（intake → 各阶段 handoff → final report，SHA 双向闭合）
- [ ] 最终 head 的 **GitHub Actions 证据**（verify job success）

### 允许使用 fixture 的要素

- [ ] 论文 PDF 提前下载到 `papers/`（避免网络波动）
- [ ] 仓库提前 clone 到 `workspace/repos/`（固定 commit）
- [ ] 环境锁提前生成（`configs/benchmarks/environments/patchcore_linux_gpu/requirements.lock.txt`）
- [ ] 测试数据提前下载（MVTec AD bottle → `workspace/datasets/`）

## 3.3 验收流程

```
Phase 1: L1 构件验收（~30 min）
  逐个验证每个阶段的 handoff SHA 链 + schema 合规
  可并行执行

Phase 2: L2 流程验收（~15 min）
  Orchestrator.run() 调用全部阶段
  验证 artifact 树完整、异常时 fail-closed

Phase 3: L3 真实任务验收（~60 min）
  准备：确认 PDF + 仓库 + 数据集 + 环境锁
  执行：一条完整 run
  检查：所有验收产出物
  判定：通过 / 有条件通过 / 不通过
```

# 4. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|----------|
| GPU 环境不兼容 | 中 | 高 | 提前生成环境锁 + 使用已有 lockfile |
| PDF 解析精度不足导致 Transfer 失败 | 中 | 高 | 3.2 维护 MinerU 精度 fixture；3.3 Context Repair 可弥补 |
| 真实 patch 导致仓库损坏 | 低 | 高 | 3.7 在 workspace 副本上操作，不从 original 修改 |
| smoke test 执行超时 | 中 | 中 | 3.8 在实验计划中设定 timeout；Pipeline fail-closed |
| Orchestrator 串联顺序错误或遗漏阶段 | 中 | 高 | 编写 L1 手检清单 + L2 自动化测试验证 artifact 树完整性 |
| CLI 命令缺少导致不能独立调试单阶段 | 低 | 中 | 先用 `python -c` 手动调用 service 函数，再补充 CLI 子命令 |
