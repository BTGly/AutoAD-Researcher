# 0. 修订说明

本版本在 Agent-Native 方案基础上进行第四轮一致性修订。第一轮完成架构切换，第二轮关闭真实实现边界，第三轮修复错误码、模型路由与控制信号审计；第四轮统一 ProcessTool 表述、强化阶段级 Git allowlist，并补齐新增行为的 scripted-agent 回归测试。

本轮修正：

1. Git Acquisition 不再混用 partial clone 与 shallow fetch；明确 `shallow_ref` 和 `partial_exact` 两个互斥 profile。
2. 增加 3.0 Tool Foundation 的状态判定、追加封板策略、风险评估与降级运行模式。
3. 明确 Agent 自主性与 Harness 强制状态机的边界，新增 `AnalysisControlSignal` 和强制切换规则。
4. 增加 MVP 模型角色、路由、fallback 和配置固定要求。
5. 明确 Skill 由 Harness 在 stage transition 时自动加载，Agent 不承担必需 Skill 的加载责任。
6. 明确通用 Filesystem Tool 如何通过 `ToolContext.active_repository` 生成仓库证据。
7. Repository State Attestor 的输出统一进入 Evidence Index，使用 `repository_identity` 证据类型。
8. Repair 使用独立保留配额，同时受全局总预算约束。
9. 定义 `analysis_progress.json` 和 append-only `analysis_observations.jsonl`；不保存隐藏思维链。
10. 将 Agent Harness 章节前移到各 Agent 阶段之前。
11. 统一 Git operation、GitHub deferred stage、default branch 获取等不一致项。
12. 给出 small / medium / large 仓库的 MVP 预算基线。
13. 补齐 `SOURCE_REF_MOVED`，区分 ref 漂移与一般 commit mismatch。
14. 将模型 capability/provider smoke test 从 R3 移到 R4。
15. 新增 append-only `analysis_control_signals.jsonl`，保留每轮 Analysis transition 请求与 Harness 判定。
16. 明确 fallback 调用前先做 capability gate；不满足时直接 blocked，不消耗 fallback 配额。
17. 修正 R2、R13 的 section-level 提交映射。
18. 将 `max_no_progress_cycles` 收敛到 `RepositoryAgentBudget` 单一事实源。
19. 最终决策 D2 统一为仅复用 argv-based `ProcessTool`；3.1 不接受 shell 字符串型 Bash Tool。
20. Deferred Tool 表显式区分 acquisition 与 analysis 的 Git 子命令 allowlist，并引用 Section 5.2。
21. Scripted Agent Tests 补齐 control-signal 历史、fallback capability gate 和 ref-moved 错误语义。

最终架构仍为：

```text
Agent Harness
→ Harness 自动加载当前阶段 Skill
→ Agent 调用成熟 Read/Search/Git/Web/GitHub 工具
→ Permission Policy 强制边界
→ Evidence Middleware 自动记录证据
→ Agent 生成严格结构化 Artifacts
→ Evidence Validator 确定性验证
→ 有界 Repair Loop
```

不恢复强制全仓 Snapshot、专用 Extractor、固定 Context Pack Pipeline。

# 1. 执行摘要

## 1.1 阶段目标

3.1 要建立的是一个可处理真实科研仓库的 Repository Intelligence Agent。它能够从以下任一入口开始：

```text
明确 GitHub URL
本地仓库路径
论文标题或项目主页
方法名 / 项目名 / 作者信息
```

必要时发现和验证候选仓库，安全获取固定 commit，在不安装依赖、不执行仓库代码、不修改仓库文件的前提下，形成：

```text
仓库用途与研究任务
安装和运行证据
训练 / 推理 / 评价 / 数据准备入口
依赖和运行时约束
配置来源和默认参数
数据集、外部资产和权重信号
评价逻辑与结果来源
可修改路径与受保护路径建议
不确定性和冲突
EnvironmentPlan 上下文
Intent Clarifier 问题候选
```

所有关键结论必须绑定：

```text
固定 repository commit
相对文件路径
文件 SHA256
起止行号
片段 SHA256
工具调用记录
```

## 1.2 架构公式

```text
Agent 负责理解、选择和编排
Tool 负责执行受控动作
Skill 负责推荐工作方法
Policy 负责阻止危险动作
Attestor 负责证明仓库状态
Evidence Recorder 负责生成引用
Validator 负责验证结论
Artifact 负责跨阶段交付
```

## 1.3 完整执行链路

```text
用户输入 / 论文 / URL / 本地路径
                ↓
      Repository Intelligence Agent
                ↓
       加载对应的 Deferred Skill
                ↓
    ┌────────────┬────────────┬────────────┐
    │            │            │            │
Web Search    GitHub Read   Process/Git   File Read/Search
Web Fetch     Metadata      Read-only     Stat/List
    │            │            │            │
    └────────────┴────────────┴────────────┘
                ↓
        Evidence Workspace / Audit Log
                ↓
         Strict Output Artifacts
                ↓
         Evidence Validator
           ↓              ↓
         通过          定向修复循环
           ↓              ↓
EnvironmentPlan Handoff / Intent Clarifier Handoff
```

## 1.4 3.1 完成后应具备的能力

1. 明确 URL 时直接读取，不进行无意义 WebSearch。
2. 只有论文、方法名或作者时，按需发现候选仓库。
3. 使用 GitHub metadata 和网页关联证据验证候选官方性。
4. 多候选存在实质歧义时要求用户确认。
5. 使用受控 Git 命令固定完整 commit，验证 remote、HEAD、tree、dirty 状态。
6. Agent 自主决定先读哪些文件、搜索哪些符号、何时补读。
7. 不要求所有仓库经过同一套硬编码 extractor。
8. 对 Python、Shell、配置、CI、notebook、混合语言仓库进行静态优先分析。
9. 输出严格 JSON artifacts，关键事实具有 EvidenceRef。
10. 对 unsupported claim、错误行号、错误 SHA 和跨 artifact 冲突进行确定性拒绝。
11. Validator 失败后允许 Agent 在预算内定向修复。
12. 将验证后的环境信息交给 EnvironmentPlan，将未知事项交给 Intent Clarifier。

# 3. 3.0 前置能力、回补策略与 Fallback

3.1 不重新实现通用工具系统，但也不把所有工作阻塞在一个模糊的“3.0 必须全部完成”条件上。本节明确 3.0 的状态、最小依赖和降级模式。

## 3.1 3.0 状态判定

进入实施前必须在 `docs/stages/3.0-final-report.md` 或阶段状态文件中确认：

```text
3.0_overall_status = open | sealed
3.0_tool_foundation_status = missing | partial | ready
3.0_sealed_commit = <sha | null>
```

当前项目中已经封板的 3.0C 子阶段不等于整个 3.0 Tool Foundation 已封板。若 3.0 总阶段尚未最终 seal，则 Tool Foundation 直接作为 3.0 剩余工作完成。

若 3.0 已经整体封板，不重写历史、不修改旧 seal commit，而是新增：

```text
3.0-F Tool Foundation Addendum
```

完成追加提交后生成新的阶段报告并记录：

```text
original_3_0_sealed_commit
addendum_start_commit
addendum_sealed_commit
```

## 3.2 3.0 最小 Tool Foundation

3.1 真实外部仓库流程的硬依赖：

```python
    description: str
    input_schema: dict
    output_schema: dict
    read_only: bool
    destructive: bool
    concurrency_safe: bool
```

```text

filesystem_search
filesystem_stat

github_read         # deferred
web_fetch           # deferred
```

## 3.4 Permission Runtime

必须具备：

```text
全局 deny
stage/profile rules
工具自身参数检查
路径 scope
program/subcommand 规则
session grants
permission_decisions.jsonl
```

决策顺序：

```text
全局 deny
→ 当前 stage profile
→ 工具自身 check_permissions
→ 参数与路径校验
→ 会话授权
→ ask user（仅策略允许时）
→ 执行并审计
```

## 3.5 Skill Runtime

必须具备：

```text
SKILL.md discovery
frontmatter parsing
required_tools 校验
permission_profile 校验
max_context_tokens
skill SHA256
stage-entry 自动加载
加载审计
```

Skill 是工作方法，不是强制安全边界。

## 3.6 Evidence Middleware

通用 Filesystem Tool 不直接依赖 3.1 业务模型。Harness 在工具调用前向 `ToolContext` 注入：

```python
class ActiveRepositoryContext(BaseModel):
    source_id: str
    repository_root: Path
    resolved_commit: str | None
    tree_sha: str
```

Evidence Middleware 根据 `repository_root` 将内部绝对路径映射为 repository-relative path，并校验路径未逃逸。无 active repository 时，Filesystem Tool 只返回通用文件结果；在 repository analysis stage 中缺少 active repository 必须失败。

## 3.7 三种启动模式

### Full Mode

具备完整 Tool Foundation 和 deferred Web/GitHub provider：

```text
论文/方法名发现
→ GitHub 候选验证
→ Acquisition
→ Analysis
```

### Constrained Mode

当 WebSearch/WebFetch/GitHub Provider 尚未完成时，允许 3.1 先支持：

```text

本地 workspace 仓库
录制的 Discovery fixture
```

此模式：

```text

DISCOVERY_PROVIDER_UNAVAILABLE 不影响 direct URL/local flow
```

但仍必须具备 ProcessTool、PermissionEngine、Skill Loader、Attestor 和 Evidence Middleware。

### Fixture-Only Mode

若 PermissionEngine、可审计 ProcessTool 或 Evidence Middleware 任一缺失，禁止读取真实外部仓库，仅允许 fake provider 和临时 Git fixture。此模式不能用于 3.1 最终封板。

# 4. 3.1 总体架构

## 4.1 分层结构

```text
RepositoryIntelligenceRequest
              ↓
Repository Intelligence Agent Harness
              ↓
Skill Loader + Tool Registry + Permission Profile
              ↓

              ↓

              ↓

              ↓

              ↓

              ↓
```

## 4.2 组件职责

| 组件 | 负责 | 不负责 |
|---|---|---|
| Agent | 选择工具、安排读取顺序、解释冲突、综合结论 | 绕过权限、伪造工具结果 |
| Tool | 执行一个受控动作并返回结构化结果 | 决定业务流程 |
| Skill | 提供推荐流程、清单、停止条件 | 强制安全 |
| Permission Policy | allow/ask/deny、路径与程序边界 | 理解科研语义 |
| State Attestor | 证明 remote/commit/tree/dirty | 理解仓库内容 |
| Evidence Recorder | 自动构造文件引用 | 判断 claim 是否正确 |
| Artifact Schema | 固定输出结构 | 决定内容语义 |
| Evidence Validator | 验证证据和跨 artifact 一致性 | 自动猜测仓库语义 |
| Repair Loop | 定向补读和修正 artifact | 放宽安全规则 |

# 5. Tool Reuse Strategy

## 5.1 复用原则

AutoAD 不重新开发通用 shell、文件查看、全文搜索、Web 搜索、HTTP fetch 或 GitHub metadata 能力。3.1 只新增 Repository Skills、Repository Contracts、Repository State Attestor、Evidence Middleware、Evidence Validator 和 Harness wiring。

## 5.2 Git 工具策略

不暴露大量 `GitCloneTool`、`GitFetchTool` 等细粒度工具。统一通过 argv-based ProcessTool 调用 Git CLI，并由 stage-specific policy 约束。

### Acquisition stage 允许

```text
git init
git clone
git remote add
git fetch
git checkout --detach
git status --porcelain
git rev-parse
git remote get-url
git symbolic-ref
```

### Analysis stage 允许

```text
git status --porcelain
git rev-parse
git ls-files
git grep
git show
git log
git diff --no-ext-diff
```

### 默认禁止

```text
push / commit / tag / merge / rebase
reset / clean / config
submodule update
lfs pull
任何未列入当前 stage allowlist 的 Git 子命令
```

Analysis 不允许 clone/fetch/checkout；Acquisition 不需要 grep/show/log 作为主流程。

## 5.3 WebSearch 边界

WebSearch 属于 Repository Discovery，不属于 Reader 内部隐式行为。

```python
resolution = discover_repository(request)
source = acquire_repository(resolution)
result = analyze_repository(source)
```

明确 URL 或 local source 时：

```text
web_search calls = 0
```

## 5.4 Deferred Tools 与 stage

工具 schema 初始不进入模型上下文。Harness 在 stage entry 时决定加载：

| Tool | 允许加载阶段 | 调用约束 |
|---|---|---|
| web_search | discovery | 仅 Discovery stage；无明确 URL/local source 时按需加载 |
| web_fetch | discovery / candidate_verification | 仅公开 HTTP(S)，通过 SSRF 与重定向检查 |
| github_read | discovery / resolution / acquisition | 只读 metadata/ref/commit/file；Analysis/Synthesis 默认不加载 |
| filesystem_* | acquisition / analysis / repair（按权限） | 受 workspace 与 ActiveRepositoryContext 约束 |
| process | acquisition / analysis | acquisition 仅允许 Section 5.2 的 `git_acquisition_allowlist`；analysis 仅允许 Section 5.2 的 `git_analysis_readonly_allowlist`。两阶段子命令集合互斥，均要求 `shell=False`。 |

ProcessTool 的“允许加载”只代表工具 schema 可见，不代表任意 Git 子命令可执行；Permission Engine 必须在每次调用时按 Section 5.2 的阶段级 allowlist 再做参数级校验。

`github_read` 不在 Analysis/Synthesis 默认加载，避免 Agent 在固定源码分析阶段绕回可变远端信息。确需补充 repository identity 时必须返回 resolution/acquisition stage 或走显式 ask policy。

加载审计：

```text
tool_name
stage
trigger_reason
schema_sha256
loaded_at
```

## 5.5 参考实现借鉴

### Claude Code

借鉴通用 Bash/Read/Search、deferred Web tools、工具权限 metadata 和工具调用审计；MVP 不复制完整 Bash AST 与企业级策略系统。

### OpenCode

借鉴简单 Tool interface、长期 Agent loop 和工具结果文件化；3.1 不依赖 PersistentShell，所有 cwd/env 显式传递。

### SWE-agent

借鉴配置化 tool/skill 描述和 bundle 思路。

### mini-swe-agent

借鉴最少工具原则，避免为仓库读取制造大量专用工具。

## 6.3 必需 Skill 的技术触发

必需 Skill 不由 Agent 自己搜索或决定是否加载。Harness 在 stage transition 时自动执行：

```text
进入 discovery      → 自动加载 repository-discovery
进入 acquisition    → 自动加载 repository-acquisition
进入 analysis       → 自动加载 repository-analysis
进入 synthesis      → 自动加载 repository-synthesis
进入 repair         → 重新加载原阶段 Skill + repair addendum
```

流程：

```text
Harness 确认 stage precondition
→ 解析 stage.required_skill
→ 校验 Skill frontmatter
→ 校验 required_tools 已注册
→ 校验 permission profile 存在
→ 将完整 Skill 内容加入该 stage context
→ 写 loaded_skills.json
```

`deferred=true` 的含义是：完整 Skill 内容不占初始 system context，只在进入对应 stage 时加载。`triggers` 字段用于一致性校验、审计和文档，不承担最终调度责任。

Agent 可以主动请求可选辅助 Skill，但不能跳过当前 stage 的 mandatory Skill，也不能加载权限 profile 不兼容的 Skill。

## 6.4 Skill 必须包含

```text
Purpose
Preconditions
Allowed Tools
Forbidden Actions
Recommended Workflow
Evidence Requirements
Output Contract
Stop Conditions
Failure Handling
Examples
```

## 6.5 repository-discovery

```text
先检查 explicit URL/local source
无明确 source 才构造少量查询
优先论文项目页、作者组织和官方链接
用 github_read 获取 default_branch、fork、archived、README 关联和 resolved commit
多候选有歧义时要求用户确认
搜索摘要只作为 association lead
```

## 6.6 repository-acquisition

```text

固定 resolved commit
checkout detached HEAD
验证 remote、HEAD、tree、dirty
不拉 submodule/LFS，不执行 hooks
```

## 6.7 repository-analysis

提供分析清单和停止条件，不规定固定 extractor pipeline。

## 6.8 repository-synthesis

```text
confirmed/inferred/conflicting/unknown 分开
```

# 9. Evidence Middleware 与 Evidence Workspace

## 9.1 目标

让 Agent 的每次读取和来源证明天然生成可验证 Evidence，而不是事后手工补路径、行号或 commit。

## 9.2 ActiveRepositoryContext 注入

Source Attestation 通过后，Harness 设置：

```python
ToolContext.active_repository = ActiveRepositoryContext(
    source_id=...,
    repository_root=...,
    resolved_commit=...,
    tree_sha=...,
)
```

Filesystem Tool 仍是 3.0 通用工具；Repository-specific 字段由 Evidence Middleware 根据 active context 附加。

## 9.3 File Evidence

`filesystem_read` 和 `filesystem_search`：

1. 将请求路径解析到 active repository root；
2. 拒绝 symlink/path escape；
3. 生成 repository-relative path；
4. 计算 file SHA、line range、snippet SHA；
5. 附加 source_id、commit、tool_call_id；
6. 追加写入 Evidence Index。

## 9.4 Identity Evidence

Repository State Attestor 完成后，必须生成 `RepositoryIdentityEvidenceRef` 并写入同一个 Evidence Index：

```text
source_kind=repository_identity
remote / commit / tree / detached / dirty
attestation_sha256
tool_call_ids
```

后续 Agent 可以用该 evidence_id 支持“分析对象是某固定 commit”等 identity claim。

## 9.5 Evidence Index

```text
runs/<run_id>/evidence_index.jsonl
```

不可变追加。任何修复不得覆盖旧 Evidence，只能追加新记录并在 artifact 中引用新 ID。

## 9.6 Evidence Scope

File evidence 必须属于 active source 和 attested tree；Identity evidence 必须匹配 current source fingerprint；Web/Search evidence 只用于关联与仓库身份，不用于源码行为事实。

## 9.7 Redaction

`.env`、私钥、凭证 URL 和高置信 secret value 不进入模型或 Evidence Index。Redacted 内容不能支持依赖值、credential 或 secret claim。

## 9.8 验收

```text

[ ] Harness 正确注入 active repository
[ ] 路径映射与逃逸检查通过
[ ] File/Identity/Web/User evidence 均可进入统一 Index
[ ] 旧 source evidence 不可用于新 source
[ ] secret value 不落盘
```

# 10. Agent Harness、模型路由与阶段控制

## 10.1 目标

Harness 强制控制 stage、Skill、权限、预算和 transition；Agent 在 Analysis stage 内自主选择读什么和如何验证假设。

## 10.2 状态机

```text
request_received
source_needed
discovery_running
resolution_waiting
source_acquiring
source_attested
analysis_running
synthesis_running
validation_running
repair_running
handoff_ready
success | partial_success | failed
```

## 10.3 Mandatory Skill Loading

Harness 在进入每个 stage 前自动加载对应必需 Skill；Skill 加载失败则 stage blocked。Agent 不能跳过或替换 mandatory Skill。

## 10.4 Analysis → Synthesis 协议

每轮 Analysis 结束时，Agent 必须输出 `AnalysisControlSignal`。Harness 在作出 transition 判定前，必须将原始 signal 与判定结果追加写入：

```text
analysis_control_signals.jsonl
```

每条记录至少包含：

```python
class AnalysisControlSignalRecord(BaseModel):
    schema_version: Literal[1]
    iteration: int
    signal: AnalysisControlSignal
    harness_decision: Literal[
        "continue_analysis",
        "enter_synthesis",
        "forced_synthesis",
        "blocked",
    ]
    accepted: bool
    rejection_reasons: list[str]
    coverage_snapshot: dict[str, str]
    stage_budget_remaining: dict[str, int]
    global_budget_remaining: dict[str, int]
    created_at: datetime
```

该文件 append-only，用于 debug、resume 和 transition 审计；不得保存隐藏思维链。`analysis_progress.json` 仍只保存当前快照。

### Agent 请求进入 Synthesis

仅当：

```text
decision=synthesis_ready
```

Harness 再检查最低覆盖：

```text
source_identity      != not_checked
installation         != not_checked
dependencies         != not_checked
entrypoints           != not_checked
evaluation            != not_checked
external_assets       != not_checked
```

某项可以是 `checked_unknown`，但必须有搜索/读取记录和 unknown reason。最低覆盖未满足且仍有预算时，Harness 拒绝 transition 并返回缺口。

### Harness 强制切换

以下任一条件触发：

```text
analysis tool-call budget exhausted
analysis token budget exhausted
max analysis LLM calls reached
max_no_progress_cycles reached
Agent 连续两轮 new_evidence_count=0
```

Harness 强制进入 Synthesis，并设置：

```text
forced_synthesis=true
budget_exhausted=<bool>
unresolved items → unknown/conflicting
final status 至少 partial_success（除非 critical identity/evidence 失败）
```

### Blocked

Agent 返回 `blocked` 时，Harness 根据 blocker 类型进入 user confirmation、failed 或 partial synthesis；不得无限等待。

## 10.5 模型配置

模型不写死在 Contracts 中，但 MVP 必须固定三个逻辑 profile，并在运行配置中映射到具体 provider/model ID：

```yaml
repository_fast_v1:
  purpose: signal extraction, simple discovery, lightweight schema repair
  required_capabilities: [tool_use, structured_output]

repository_primary_v1:
  purpose: repository analysis, conflict resolution, synthesis, repair
  required_capabilities: [strong_code_reasoning, tool_use, structured_output, long_context]
  minimum_context_window: 128000

repository_fallback_v1:
  purpose: provider outage or one failed structured-output retry
  required_capabilities: [strong_code_reasoning, tool_use, structured_output]
```

团队当前可将 fast/primary 候选映射到既定的低成本模型与高能力模型（例如此前讨论的 Flash/Pro 路由），但具体 provider model ID 必须在实现提交中验证并写入：

```text
configs/models/repository_intelligence.yaml
runs/<run_id>/agent_config.json
```

MVP 初始目标映射：

```yaml
repository_fast_v1: deepseek-v4-flash
repository_primary_v1: deepseek-v4-pro
repository_fallback_v1: anthropic:claude-sonnet-4-6
```

这些是项目选定的初始目标 ID，不是 schema 常量。R4 实施时必须分别完成 tool-use、structured-output、context-window 和 provider availability smoke test；任何不可用 ID 必须通过独立配置提交替换。最终封板前不得保留 `latest`、空值、未解析别名或未经 smoke test 的模型 ID。

### 路由

```text
Signal Extraction / simple Discovery → repository_fast_v1
Ambiguous Resolution              → repository_primary_v1
Analysis / Synthesis / Repair      → repository_primary_v1
Primary provider failure           → repository_fallback_v1，最多一次
```

不得在 critical analysis 中静默降级到不满足能力要求的小模型。fallback 的处理顺序固定为：

```text
Primary 调用失败
→ 查询 fallback provider availability
→ 校验 fallback capabilities 和 minimum context requirement
→ capability 不满足：直接 blocked，不发起 fallback 调用，不消耗 fallback 配额
→ capability 满足：允许调用一次 fallback
→ fallback 失败：blocked 或按当前 stage 规则失败
```

因此 fallback 配额只统计实际发起的 fallback LLM 调用。所有 capability gate 和最终路由决定写入 model routing audit。

## 10.6 Skill 设计对模型的约束

Skill 使用短步骤、明确 stop condition、结构化 control signal 和少量正反例，避免依赖模型自行记住长安全规则。安全仍由 Policy 保证。

## 12.1 目标

## 12.2 两个互斥 Acquisition Profiles

### Profile A：shallow_ref

随后必须验证：

```text
git rev-parse FETCH_HEAD == RepositoryResolution.resolved_commit
git rev-parse HEAD       == RepositoryResolution.resolved_commit
```

若 ref 在 resolve 与 fetch 之间移动，必须报 `SOURCE_REF_MOVED`，不得静默接受新 commit。

### Profile B：partial_exact

适用于用户明确指定 40 位 commit 或需要获取历史 exact commit。只使用 partial clone，不使用 shallow depth：

```bash
git clone --filter=blob:none --no-checkout --no-tags \
  --template=<controlled-empty-template> \
  https://github.com/<owner>/<repo> \
  workspace/repos/<source_id>

git -C workspace/repos/<source_id> \
  fetch --no-tags origin <commit_sha>

git -C workspace/repos/<source_id> \
  checkout --detach <commit_sha>
```

若 server 不支持 partial clone 或 exact SHA fetch，返回结构化失败。可在删除失败目标后切换到经过用户/策略允许的 full exact fetch，但必须形成新的 acquisition attempt；不得把 `--filter=blob:none` 与 `--depth` 混用。

## 12.3 Profile 选择

```text
resolved_ref 为 branch/tag/default branch → shallow_ref
用户明确 exact 40-char commit          → partial_exact
本地 source                            → local
```

## 12.4 安全环境

```text

GIT_CONFIG_SYSTEM=/dev/null
GIT_CONFIG_GLOBAL=/dev/null
GIT_TERMINAL_PROMPT=0
GIT_LFS_SKIP_SMUDGE=1
GIT_TEMPLATE_DIR=<controlled-empty-template>

```

## 12.5 Repository State Attestor

非 LLM 组件执行并解析：

```text

git rev-parse HEAD^{tree}
git status --porcelain

直接读取并解析 .gitmodules（不调用 git config，不 update submodule）
```

Attestor 输出 `repository_attestation.json`，并追加 `RepositoryIdentityEvidenceRef` 到 Evidence Index。Agent 不得自行填写 identity 字段。

## 12.6 Local Source

## 12.7 禁止

## 13.4 Analysis Progress

`analysis_progress.json` 是 Harness 每轮覆盖状态，不是模型思维链：

```python
class AnalysisProgress(BaseModel):
    schema_version: Literal[1]
    iteration: int
    stage_status: Literal["running", "synthesis_ready", "blocked", "forced_complete"]
    coverage: dict[str, Literal["confirmed", "checked_unknown", "conflicting", "not_checked"]]
    evidence_count: int
    tool_calls_used: int
    file_reads_used: int
    search_calls_used: int
    llm_calls_used: int
    input_tokens_used: int
    new_evidence_count_last_cycle: int
    no_progress_cycles: int
    budget_exhausted: bool
    unresolved_blockers: list[str]
    next_actions: list[str]
```

`analysis_observations.jsonl` 只记录简洁、证据绑定的工作事实：

```python
class AnalysisObservation(BaseModel):
    observation_id: str
    category: str
    summary: str
    status: Literal["candidate", "confirmed", "conflicting", "discarded"]
    evidence_ids: list[str]
    created_at: datetime
```

`analysis_control_signals.jsonl` 追加保存每轮 Agent control signal 与 Harness transition 判定；它与 `analysis_observations.jsonl` 分工如下：

```text
analysis_progress.json          当前覆盖、预算、阻塞项和下一动作快照
analysis_control_signals.jsonl  每轮 transition 请求、接受/拒绝原因和剩余预算
analysis_observations.jsonl     简洁的 evidence-backed 工作事实
```

禁止保存隐藏 chain-of-thought、长自由推理草稿或未经证据支持的私密 scratchpad。原 `analysis_notes.json` 删除。

## 13.5 Agent 自主性与 Harness 边界

Agent 可调整读取顺序、定向搜索、跳过无关目录和在预算内验证假设；不能自行跨 stage、扩大权限、无限循环或绕过最低覆盖检查。每轮必须输出 `AnalysisControlSignal`，transition 由 Harness 决定。

## 13.6 MVP 预算基线

预算为一次 Repository Intelligence run 的 aggregate 上限；Repair reserve 单独列出。

| Profile | 适用规模 | Analysis tool calls | File reads | Search calls | Analysis LLM calls | Aggregate input tokens | Output tokens | Repair tool/LLM reserve |
|---|---:|---:|---:|---:|---:|---:|---:|---:|
| small | ≤1,000 tracked files 且 ≤20 MB text | 50 | 24 | 12 | 3 | 80k | 10k | 8 / 1 |
| medium | 1,001–10,000 files 或 ≤100 MB text | 90 | 40 | 22 | 4 | 140k | 16k | 12 / 2 |
| large | >10,000 files 或 >100 MB text | 140 | 60 | 32 | 6 | 200k | 24k | 16 / 2 |

共同 hard caps：

```text
max_no_progress_cycles 由 RepositoryAgentBudget.max_no_progress_cycles 定义，MVP 默认值为 2
max_repairs            由 RepositoryAgentBudget.max_repairs 定义，MVP 默认值为 2
single file read 默认 ≤128 KiB
单次搜索返回默认 ≤100 matches
```

`RepositoryAgentBudget` 是 no-progress 和 repair 次数的单一事实源；预算表与 profile 配置不得重复维护独立常量。

规模由 `git ls-files` 数量和轻量 stat 估计；不要求先读取全仓。超过 large hard cap 时强制 synthesis/partial_success，不自动升级无限预算。

## 13.7 Prompt Injection

README 中的运行命令和指令均为 untrusted data，不执行、不扩大权限、不读取 workspace 外内容。

## 13.8 可选工具延后

AST、manifest parser、notebook summarizer、仓库索引器仅在真实回归证明通用工具不足时作为 Agent 可选工具加入。

```text
[ ] small/medium/large 预算 profile 有测试
```

# 16. Artifact Repair Loop

## 16.1 目标

Validator 发现 repairable 的证据缺失或 artifact 冲突后，由 Agent 定向补读或修订，Validator 重新验证。

## 16.2 Budget 归属

Repair 拥有独立 reserve：

```text
max_repair_tool_calls
max_repair_llm_calls
max_repairs
```

同时计入：

```text
max_total_tool_calls
max_total_llm_calls
max_total_input_tokens
max_total_output_tokens
```

Analysis 无权消费 Repair reserve。Repair 调用同时扣减 repair reserve 与 global total budget；二者任一不足即不得发起该调用。若 Analysis 已耗尽其 stage budget但 global budget 和 repair reserve 仍有余额，Repair 仍可执行；若 global hard cap 已耗尽，则 Repair 不执行，直接 failed 或 partial_success。运行中不能临时扩大预算绕过限制。

## 16.3 输入

```text
validation report
failed claim/evidence IDs
受影响 artifact 片段
repair stage remaining budget
total remaining budget
```

## 16.4 允许

补读明确相关文件、重新搜索仓库内符号、修正 path/line、降级 unsupported claim、补充 conflict evidence、重新生成受影响 artifact。

## 16.5 禁止

运行代码、安装依赖、扩大权限、使用搜索摘要替代源码、删除重要 unknown、无依据升级 confirmed、修改仓库。

## 16.6 限制

```text
max_repairs=2
每轮定向补读受 max_repair_tool_calls 约束
每轮最多一次 repair LLM synthesis
每轮写 repair reason 和 consumption
```

## 16.7 验收

```text
[ ] Repair reserve 不被 Analysis 消耗
[ ] Repair 同时受 global hard cap 约束
[ ] budget=0 时行为确定
[ ] 修复只影响相关 artifact
[ ] 安全边界不可降低
```
