# AutoAD-Researcher Step 3.6–3.7 开发计划：Patch Planning, Approval & Controlled Application

> 文档状态：修订稿 v1.3（v1.2 targeted_revision_required，3 P0 + 3 P1 已修复）
> 更新日期：2026-06-17
> 阶段名称：**补丁规划、人工审批与受控应用**
> 上游输入：3.5 Experiment Planning Handoff（`experiment_planning_handoff.json`）
> 下游阶段：Step 3.8 Runner Intake & Experiment Execution Integration → Step 3.9 Results Analysis
> 3.6 本身：**只读、只规划、不修改仓库文件**
> 3.7 本身：**审批后写入、不执行完整实验、不宣称实验成功**

---

# 0. 阶段定义与主流程

```
3.5 Experiment Planning Handoff
        ↓
3.6 Narrow Repository Read（仅读取已确认范围）
        ↓
3.6 RepositoryChangePlan（文件级修改计划）
        ↓
3.6 Conflict Analysis + Workspace Layout
        ↓
3.6 Patch Materialization（生成真实代码载荷）
        ↓
3.6 PatchPayload Manifest + Proposed Patch Diff
        ↓
3.6 Deterministic Payload Validation
        ↓
3.7 Approval Request
  ├── 用户审批 exact change IDs + exact payload SHA
  ├── approve / revise / reject
  └── 审批绑定 patch_payload_manifest_sha256 + proposed_patch_diff_sha256
        ↓
3.7 Application Preflight
  ├── plan SHA / payload manifest SHA / diff SHA 三方绑定
  ├── repository fingerprint 匹配
  ├── workspace 存在
  └── validation report passed
        ↓
3.7 Controlled Patch Application
        ↓
3.7 Local Validation（受控命令执行）
        ↓
patch_applied_and_local_validations_passed
        ↓
eligible_for_runner_intake
        ↓
3.8 Runner Intake & Experiment Execution Integration
```

**合并写一份计划是合理的，但必须保留"只读规划"（3.6）和"审批后写入"（3.7）两道强制边界。**

核心不变量：

```
approved_change_ids 是审批权威粒度
approved_paths 从 approved_change_ids 确定性派生
用户审批绑定的是确切代码 payload（patch_payload_manifest_sha256 + proposed_patch_diff_sha256）
policy allow scope 缺失 → preflight blocked（default-deny，无例外）
deny 永远不能被用户审批覆盖
3.7 成功终态只能是 patch_applied_and_local_validations_passed
Handoff 状态为 eligible_for_runner_intake，不是 runner_ready 或 experiment_ready
Change model 分离 change_kind 和 target_mode
任何影响 canonical SHA 的变化 → 旧审批自动失效 → 重新审批
```

# 3. 核心数据模型

## 3.1 PlannedRepositoryChange

```python
class PlannedRepositoryChange(BaseModel):
    change_id: str
    workspace_id: str
    change_kind: Literal["create", "modify", "delete", "rename", "configuration_only", "test_only"]
    target_mode: Literal["existing_target", "new_target"]
    hook_id: str | None = None
    existing_symbol_id: str | None = None
    proposed_symbol: str | None = None
    payload_id: str | None = None          # 关联 PatchPayload
    repository_path: str                   # validate_workspace_path 已校验
    rename_target_path: str | None = None
    variant_ids: list[str]
    rationale: str
    symbol_delta: SymbolContractDelta | None
    interface_delta: InterfaceContractDelta | None
    target_collision_policy: Literal["must_not_exist", "replace_existing"] = "must_not_exist"
```

**关键规则**：
- `change_kind` 与 `target_mode` 解耦
- `create` → `target_mode=new_target`；文件已存在 + `must_not_exist` → 失败
- `modify/delete/rename` → `target_mode=existing_target`
- `rename_target_path` 必填；目标已存在 + `must_not_exist` → 失败

## 3.2 PatchPayload

```python
class PatchPayload(BaseModel):
    payload_id: str
    change_id: str
    payload_kind: Literal["unified_diff", "full_after_content"]
    before_sha256: str | None
    target_path: str
    payload_artifact_id: str         # ArtifactStore identifier
    payload_sha256: str

class PatchPayloadManifest(BaseModel):
    """所有 payload 的清单，整体计算 SHA 供审批绑定。"""
    manifest_id: str
    run_id: str
    patch_plan_sha256: str
    workspace_id: str
    payloads: list[PatchPayload]
    proposed_diff_artifact_id: str
    proposed_diff_sha256: str
    manifest_sha256: str

class PatchPayloadValidationIssue(BaseModel):
    issue_id: str
    payload_id: str | None
    change_id: str | None
    category: Literal[
        "payload_sha_mismatch", "before_sha_mismatch",
        "undeclared_path", "undeclared_file_creation",
        "file_mode_change", "symlink_change", "submodule_change",
        "unsupported_payload_kind",
    ]
    description: str
    resolution: Literal["regenerate", "blocked"]

class PatchPayloadValidationReport(BaseModel):
    report_id: str
    patch_plan_sha256: str
    payload_manifest_sha256: str
    status: Literal["passed", "failed"]
    issues: list[PatchPayloadValidationIssue]
    validated_at: datetime
```

MVP 只保留 `unified_diff` 和 `full_after_content`。Payload artifact 通过 ArtifactStore 解析，不接受任意本地路径。

## 3.3 PatchPayload 生产者：PatchMaterializer

```
PatchMaterializer / CodeSynthesisAgent
  输入: RepositoryChangePlan + narrow repository context
  输出: PatchPayload + proposed_patch.diff + PatchPayloadManifest
  权限: 只读仓库、只写 artifact，不写工作树
```

`DeterministicPatchPayloadValidator` 检查：
- `payload.change_id` 存在且 `target_path` 与 change 匹配
- `payload_sha256` 匹配
- `before_sha256` 匹配当前文件
- diff 只触碰批准路径
- diff 不新增未声明文件
- diff 不修改文件模式、symlink 或 submodule

## 3.6 ApprovalPatchBundle

`approve_partial` 时，用户选择的 change 子集必须生成独立的 subset diff：

```python
class ApprovalPatchBundle(BaseModel):
    """部分审批时，批准的 change IDs 与对应的 subset diff 一起打包。"""
    bundle_id: str
    patch_plan_sha256: str
    workspace_id: str

    approved_change_ids: list[str]         # 用户选择的子集
    approved_payload_ids: list[str]        # 对应 payload

    approved_diff_artifact_id: str         # subset-specific diff
    approved_diff_sha256: str

    payload_manifest_sha256: str           # 原始全量 manifest（不变）
    bundle_sha256: str                     # 本 bundle 的 canonical SHA
```

流程：

```
用户选择部分 changes
→ 生成 subset-specific diff（只含批准变更）
→ 计算 ApprovalPatchBundle SHA
→ 用户批准该 bundle
→ 3.7 只能应用该 bundle 中的 change IDs + diff
→ 全量 diff SHA 与 部分 change IDs 不再共用一个审批对象
```

## 3.7 审批失效规则

任何影响 canonical SHA 的变化 → 旧审批自动失效：

```
payload 生成后
workspace 重排后
conflict resolution 后
validation command 变化后
→ plan_changed → approval_invalidated → approval_pending
```

# 4. Path 安全模型（严格 default-deny）

```
1. path 或 ancestor 在 deny → DENY
2. change_id 未在 approved_change_ids → DENY
3. path 未从 approved_change_ids 派生 → DENY
4. path 在 ask-list 且未显式批准 → DENY
5. allow scope 缺失 → DENY
6. path 不在 allow scope → DENY
→ ALLOW
```

`allow scope` 必须显式配置。未配置 → preflight blocked，不进入写入。

## 路径逃逸防护

```
模型级: validate_workspace_path（拒绝 ..、绝对路径）
运行时: resolve() + 确认在 repository_root 内
```

# 5. 创建与重命名冲突策略

```
create + target_mode=new_target
  ├── 文件不存在 → 创建
  └── 文件已存在
      ├── target_collision_policy=must_not_exist → patch_application_failed
      └── target_collision_policy=replace_existing → 需单独审批 → 保存 before_blob

rename + 目标路径
  ├── 目标不存在 → 正常 rename
  ├── 目标已存在 → 默认失败
  └── replace_target policy → 单独审批 → 保存 target_before_blob
```

# 8. ValidationCommand 受控执行

## 8.1 命令模板注册表

executable allowlist 不足 — `python` / `pytest` 可执行任意代码。需增加命令模板注册表，限制固定 argv 前缀：

```python
class ValidationCommandTemplate(BaseModel):
    """预注册的命令模板。审批绑定 template_id + resolved argv。"""
    template_id: str
    check_kind: CheckKind
    executable: str
    fixed_prefix_argv: list[str]          # 不可变前缀
    allowed_argument_patterns: list[str]
    working_directory_policy: Literal["workspace_root", "declared_subdir"]
    network_access: Literal["deny"] = "deny"
    repository_write_access: Literal["deny", "temporary_only"] = "deny"

REGISTERED_TEMPLATES = {
    "syntax_python": ValidationCommandTemplate(
        template_id="syntax_python", check_kind="syntax",
        executable="python", fixed_prefix_argv=["python", "-m", "compileall"],
    ),
    "format_ruff": ValidationCommandTemplate(
        template_id="format_ruff", check_kind="format",
        executable="ruff", fixed_prefix_argv=["ruff", "check"],
    ),
}
```

执行时：
- `argv` 必须以 `fixed_prefix_argv` 开头
- 禁止 `-c`、任意脚本路径
- `working_directory` containment
- 执行前后 dirty-state 对比
- 临时文件限定在声明目录

sandbox/隔离环境属于运行时基础设施，列入 3.8 deferred。

## 3.7 可执行命令的范围边界

仅凭 `fixed_prefix_argv` 无法确保网络和写入隔离——`pytest`/`mypy`/`pylint`/`compileall` 都会导入并执行仓库代码。因此 3.7 MVP 的 ValidationCommand 执行能力限定为**不导入仓库代码的静态检查**：

```
3.7 允许:
  - difflib diff 校验（文本比对）
  - AST.parse() 语法检查（不执行）
  - ruff check / ruff format（静态 AST 分析）
  - 路径 containment 检查
  - import 声明存在性检查（不实际导入）

3.7 禁止（移交 3.8 Runner sandbox）:
  - pytest（执行项目测试）
  - mypy（导入模块进行类型推断）
  - pylint（导入模块进行检查）
  - compileall（编译 = 导入顶层代码）
  - 任何 python -c 或 python <script.py>
```

## 8.2 ValidationCommand

```python
class ValidationCommand(BaseModel):
    command_id: str
    label: str
    check_kind: Literal["syntax", "format", "static", "type", "import", "unit_test"]
    required: bool = False
    executable: str                    # 必须在 allowlist
    argv: list[str]                    # shell=False
    working_directory: str             # 必须在 workspace 内
    network_access: Literal["deny"] = "deny"
    timeout_seconds: int = 120

EXECUTABLE_ALLOWLIST = {
    "python", "python3", "ruff", "pylint", "mypy", "pytest",
}
```

命令策略：
- `executable` 必须在 allowlist
- `working_directory` 位于 workspace 内
- `shell=False`，禁止重定向/管道/命令替换
- 禁止网络
- 禁止安装依赖
- `required=True` 且未批准 → 验证失败
- compileall 使用 `PYTHONDONTWRITEBYTECODE=1` 或验证后清理 `__pycache__`

# 11. Preflight（九项校验）

在打开任何文件前强制检查。`approve_all` 与 `approve_partial` 按类型分别验证：

```
1.  compute_canonical_plan_sha256(plan) == plan.plan_sha256
2.  payload_manifest_sha256 == PatchPayloadManifest 计算值
3a. (approve_all)  decision.approved_diff_sha256 == 全量 diff SHA
3b. (approve_partial)  decision.approval_patch_bundle_sha256 == bundle SHA
4.  decision 中引用的 diff artifact 实际 SHA 与声明匹配
5.  _fingerprint(repository_root) == plan.repository_fingerprint
6.  run_id == plan.run_id
7.  workspace_id ∈ plan.workspace_plans
8.  PatchPlanValidationReport: run_id/plan_sha256 匹配, status=passed, issues=empty
9.  PatchPayloadValidationReport: payload_manifest_sha256 匹配, status=passed, issues=empty
```

ApprovalRequest 还应绑定：

```
patch_payload_validation_report_sha256: str
```

# 12. 状态机

```
planning_initialized
→ repository_attested
→ patch_planning
→ patch_payload_generation
→ patch_payload_validating
→ approval_pending
    ├── approve → patch_applying
    ├── revise → patch_planning（或 payload 重生成）
    └── reject → rejected

payload/workspace/conflict 变化 → plan_changed → approval_invalidated → approval_pending

patch_applying
    ├── full success → patch_applied
    ├── partial → patch_application_partial_failure
    ├── failed → patch_application_failed
    └── stale → replan_required

patch_applied → post_patch_validating
    ├── passed → patch_applied_and_local_validations_passed
    └── failed → patch_applied_but_local_validation_failed

patch_applied_and_local_validations_passed → eligible_for_runner_intake

recovery: repair_or_rollback_pending → rollback_running → rolled_back | rollback_failed
```

# 13. 负向测试矩阵

封板前必须覆盖：

```
[ ] payload SHA 不匹配 → blocked
[ ] payload 修改未审批路径 → payload validation 失败
[ ] diff 包含额外文件 → payload validation 失败
[ ] create 覆盖已有文件 → application_failed
[ ] rename 目标冲突 → application_failed
[ ] modify/delete 文件不存在 → application_failed
[ ] 同文件多 change 逆序回滚 → 恢复原始内容
[ ] rollback fingerprint 不匹配 → rollback_failed
[ ] symlink escape → containment blocked
[ ] allow scope 缺失 → preflight blocked
[ ] ask 路径未批准 → write denied
[ ] required command 未批准 → validation failed
[ ] 非法 executable → validation failed
[ ] validation 产生 __pycache__ → dirty state 检测
[ ] workspace shared change 克隆 → 两个 workspace 独立应用
[ ] payload 变化 → 旧审批失效 → reapproval_required
[ ] plan SHA 变化 → preflight blocked
[ ] stale repository fingerprint → preflight blocked
```
