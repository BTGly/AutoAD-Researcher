# AutoAD-Researcher Step 3.6–3.7 开发计划：Patch Planning, Approval & Controlled Application

> 文档状态：修订稿 v1.0
> 更新日期：2026-06-17
> 阶段名称：**补丁规划、人工审批与受控应用**
> 上游输入：`ExperimentPlan`（3.5 产出）+ `ImplementationVariant`（3.4 产出）
> 下游阶段：Step 3.8 Runner Intake → Step 3.9 Experiment Execution
> 3.6 本身：**只读、只规划、不修改仓库文件**
> 3.7 本身：**审批后写入、不执行完整实验、不宣称实验成功**

# 0. 阶段定义

3.6 和 3.7 共享同一组上下文（选中的 Variant、实验计划、固定仓库 commit、baseline architecture contract、modifiable/protected paths），围绕同一份 `RepositoryChangePlan` 完成：生成计划 → 展示风险 → 用户审批 → 应用修改 → 验证修改是否符合计划。

**合并写一份计划是合理的，但必须保留"只读规划"（3.6）和"审批后写入"（3.7）两道强制边界。**

```
3.5 Experiment Planner
        ↓
3.6 Patch Planner（只读）
  ├── 复用 ModificationHook 作为事实源
  ├── narrow repository read
  ├── RepositoryChangePlan（唯一修改事实源）
  ├── multi-variant conflict analysis
  └── deterministic plan validation
        ↓
3.7 Approval
  ├── approval_request.json（展示全部计划变更）
  ├── 用户 approve / revise / reject
  └── approved_change_ids 为权威粒度
        ↓
Controlled Patch Application
  ├── preflight（SHA/fingerprint/workspace/validation report 六项校验）
  ├── 五层路径交集（deny > ask-approved > allowed > planned ∩ approved > default-deny）
  ├── atomic writes with before-blob rollback
  └── rename / delete / modify / create
        ↓
Local Patch Validation
  ├── syntax（compileall）
  ├── approved ValidationCommand 执行（format/static/type/import/unit_test）
  └── CheckResult（passed/failed/not_run/not_required）
        ↓
patch_applied_and_local_validations_passed
        ↓
eligible_for_runner_intake
        ↓
Runner Preflight（独立阶段）
        ↓
experiment_execution_admissible
```

核心不变量：

```
approved_change_ids 是审批权威粒度
approved_paths 从 approved_change_ids 确定性派生，不可独立扩大范围
deny 永远不能被用户审批覆盖
3.7 最终状态只能是 patch_applied_and_local_validations_passed，不能声称 experiment_ready
Handoff 状态为 eligible_for_runner_intake，而不是 runner_ready
Change model 分离 change_kind 和 target_mode，避免 create+existing_target 类歧义
```

# 1. 为什么合并

3.6 和 3.7 使用的是同一组上下文：

```
选中的 Variant
实验计划
固定仓库 commit
baseline architecture contract
modifiable/protected paths
依赖和环境约束
```

并围绕同一份 `RepositoryChangePlan` 完成全部操作。拆成两份大文档会产生大量重复（相同输入、相同路径规则、相同风险模型、相同 PatchPlan contract、相同回滚信息、相同审计链）。

**但安全边界必须分开**：3.6 是纯规划、只读；3.7 是写操作、具有破坏性。

# 3. 核心数据模型

## 3.1 PlannedRepositoryChange

```python
class PlannedRepositoryChange(BaseModel):
    change_id: str
    workspace_id: str
    change_kind: Literal["create", "modify", "delete", "rename", "configuration_only", "test_only"]
    target_mode: Literal["existing_target", "new_target"]

    # existing_target 至少一个非空；new_target 必须 proposed_symbol 非空
    hook_id: str | None = None
    existing_symbol_id: str | None = None
    proposed_symbol: str | None = None

    payload_id: str | None = None  # 关联 PatchPayload
    repository_path: str          # validate_workspace_path 已校验
    rename_target_path: str | None = None

    variant_ids: list[str]
    rationale: str
    symbol_delta: SymbolContractDelta | None
    interface_delta: InterfaceContractDelta | None
```

**关键规则**：
- `change_kind`（操作类型）与 `target_mode`（目标是否存在）解耦
- `create` → `target_mode=new_target`
- `modify/delete/rename` → `target_mode=existing_target`
- `test_only/configuration_only` 两种 target_mode 均允许
- `rename` 必填 `rename_target_path`
- `repository_path` 和 `rename_target_path` 均通过 `validate_workspace_path` 校验（拒绝绝对路径、`..` 遍历）

## 3.2 ApprovalDecision

```python
class ApprovalDecision(BaseModel):
    decision: Literal["approve_all", "approve_partial", "revise", "reject"]

    # 权威审批范围
    approved_change_ids: list[str]
    rejected_change_ids: list[str]

    # 由 change_ids 派生，供 PermissionEngine 执行
    approved_paths: list[str]
    approved_ask_paths: list[str]  # ask 路径显式批准

    # 绑定 plan SHA
    approved_patch_plan_sha256: str
```

**键规则**：
- `approve_all` 等价于 `approved_change_ids = 全部 plan change_ids`
- `approved_paths` 由 `approved_change_ids` 派生，不可独立扩大
- 审批 SHA 必须与 `plan.plan_sha256` 一致；任何变化自动失效

## 3.3 PatchPayload

```python
class PatchPayload(BaseModel):
    payload_id: str
    change_id: str
    payload_kind: Literal["unified_diff", "full_after_content", "ast_edit_script"]
    before_sha256: str | None
    target_path: str
    payload_artifact_ref: str
    payload_sha256: str
```

`RepositoryChangePlan.patch_payloads` 列表存储所有 payload，`PlannedRepositoryChange.payload_id` 关联。

# 4. 关键不变量

## 4.1 Change 目标规则

```
已有对象修改：target_mode=existing_target, hook_id/symbol_id 必须可解析
新建对象：target_mode=new_target, hook_id=None, proposed_symbol 必填
新建 repository_path 及父目录必须属于 modifiable scope
```

## 4.2 Path / Symbol / Interface Validation

```
路径必须通过 validate_workspace_path（拒绝绝对路径、..、反斜杠）
protected_candidate → 默认禁止
generated_or_vendor → 禁止
unknown → 返回 3.1 reanalysis
modifiable_candidate → 可以计划
钩子路径必须等于 change 路径
allowed_for_transfer_design=False → 拒绝
existing_symbol_id 存在但无 symbol table → return_to_3_1
```

## 4.3 审批规则

```
approved_change_ids 是权威事实
approved_paths 由 approved_change_ids 派生
自然语言 "approve" = approve_all
deny 永远不能被用户审批覆盖（policy deny always wins）
```

## 4.4 Workspace 规则

```
3.5 定义需要运行的 Variants
3.6 根据文件级冲突决定 workspace 布局
3.7 在对应 workspace_id 内过滤和验证修改
每个 diff/fingerprint 必须绑定 workspace_id
```

# 5. Path 安全模型

## 五层交集（应用于每个写入操作）

```
1. path 及全部 ancestor 不在 deny → 否则 DENY
2. change_id 在 approved_change_ids → 否则 DENY
3. path 在 approved_paths（从 change_ids 派生）→ 否则 DENY
4. path 不在 ask-list，或已在 approved_ask_paths → 否则 DENY
5. path 在 allowed scope（或 scope 未配置）→ 否则 DENY
→ ALLOW
```

## 双重路径逃逸防护

```
PlannedRepositoryChange 模型级别：validate_workspace_path（拒绝 ..、绝对路径）
Applicator 运行时：resolve() + 确认在 repository_root 内
```

---

# 6. 冲突分析（Path + Symbol 级）

```
相同路径、不同 symbol → parameterizable（可通过配置开关或分离模块解决）
相同路径、相同 symbol → mutually_exclusive → separate worktrees
不同路径 → clean → shared workspace
```

`apply_workspace_layout()` 将 conflict analysis 的 workspace 分配回写到 `RepositoryChangePlan.workspace_plans` 和 `PlannedRepositoryChange.workspace_id`，并重新计算 canonical SHA。

---

# 7. Preflight（应用前置校验）

在打开任何文件进行写入之前强制执行：

```
1. compute_canonical_plan_sha256(plan) == plan.plan_sha256
2. decision.approved_patch_plan_sha256 == plan.plan_sha256
3. request.patch_plan_sha256 == plan.plan_sha256
4. _fingerprint(repository_root) == plan.repository_fingerprint
5. run_id == plan.run_id
6. workspace_id ∈ plan.workspace_plans 各 workspace 的 ID
7. PatchPlanValidationReport: run_id/plan_sha256 匹配, status=passed, issues 为空
```

全部通过 → preflight.ready = True → 开始写入。

# 8. 应用与回滚

## 应用规则

```
modify/delete/rename + 文件不存在 → 失败，不创建新文件
create + 文件已存在 → 覆写
rename: 保存源和目标 before_blob，执行原子 rename
逐 change try/except → 异常收集到 failed_changes
部分写入 → patch_application_partial_failure（可审计）
```

## 回滚规则

```
逆序恢复（reverse_apply_order）
每个 entry 保存 base64 编码的 before_blob
rename 额外记录 target_before_blob
回滚后校验指纹 == repository_before_fingerprint
```

# 9. 本地验证

## CheckResult 状态

```python
class CheckResult(BaseModel):
    status: Literal["passed", "failed", "not_run", "not_required"]
    command_id: str | None
    exit_code: int | None
    stdout_ref: str | None
    stderr_ref: str | None
```

## 验证流程

```
1. syntax_check: python -m compileall（默认执行）
2. 执行 approval 批准的 ValidationCommand（按 check_kind 路由到 format/static/type/import/unit_test）
3. required=True 的命令如果未批准 → 失败
4. 全部通过的 check 为 passed 或 not_required → status=passed
```

## 状态机

```
patch_applied → finalize_with_validation() → patch_applied_and_local_validations_passed
patch_application_failed / partial_failure → finalize 被拒绝
patch_applied_and_local_validations_passed → eligible_for_runner_intake
```

# 10. Canonical SHA

`compute_canonical_plan_sha256()` 包含 plan 的全部内容，仅排除自身 `plan_sha256` 字段。

```
包含：repository_source_id, repository_commit, repository_fingerprint,
      changes（保留顺序）、patch_payloads（保留顺序）、
      dependency_changes、configuration_changes、test_changes、
      workspace_plans（保留顺序）、selected_variant_ids、idea_id
排除：plan_sha256
```

有序列表（changes、patch_payloads、workspace_plans、validation_commands）保留原始顺序；无序列表排序保证确定性。

# 11. 权限划分

## Step 3.6

```yaml
allow:
  - read validated artifacts
  - read repository files within narrow scope
  - search repository symbols
  - write planning artifacts

deny:
  - repository write
  - process execution
  - dependency installation
  - experiment execution
```

## Step 3.7

```yaml
allow:
  - write approved repository paths
  - create approved files
  - run approved validation commands
  - generate diff
  - rollback

deny:
  - modify unapproved paths
  - modify protected paths
  - arbitrary shell
  - git push / commit
  - full experiment
```

# 12. 状态机

```
planning_initialized
→ repository_attested
→ patch_planning
→ patch_plan_validating
→ approval_pending
    ├── approve → patch_applying
    ├── revise → patch_planning
    └── reject → rejected

patch_applying
    ├── full success → patch_applied
    ├── partial → patch_application_partial_failure
    ├── failed → patch_application_failed
    └── stale plan → replan_required

patch_applied → post_patch_validating
    ├── passed → patch_applied_and_local_validations_passed
    └── failed → patch_applied_but_local_validation_failed

patch_applied_and_local_validations_passed → eligible_for_runner_intake

failure recovery: repair_or_rollback_pending → rollback_running → rolled_back
```

# 14. 遗留问题

| 问题 | 优先级 | 说明 |
|---|---|---|
| PatchPayload 未生成真实代码载荷 | P0 | Schema 已定义，但 `_apply_single_change()` 仍是占位注释；需真实 diff/merge 引擎或代码生成器 |
| 真实 diff 应用需外部工具 | P0 | 当前只有 contract + placeholder；unified_diff/ast_edit 模式需外部工具 |
| before_blob 为 base64 文本串 | P1 | 已从原始 bytes 升级为 base64；但未使用外部 artifact 引用存储大文件 |
| workspace_ids: list[str] 多归属 | P1 | 当前 workspace_id 是单值；shared change 无法同时属于多个独立 worktree |
| 真实 ValidationCommand 执行需 runner | P1 | check_kind 已结构化，但实际 lint/typecheck 需依托外部 runner |
| Pydantic model_fields 弃用警告 | P2 | `compute_canonical_plan_sha256` 使用 `__class__.model_fields` 替代已修复 |

# 15. 封板条件

Step 3.6 当前状态：**conditionally_accepted**（核心数据模型和验证器可用）
Step 3.7 当前状态：**changes_required**（安全骨架完成，但真实 patch payload 未实现）

封板前提：

1. PatchPayload 接入 Plan 和 Applicator 的实际代码生成链路
2. 真实 unified diff 应用（不依赖外部模型推理）
3. 字节级精确回滚（含大文件 artifact ref）
4. 多 workspace 共享 change 支持
5. 全链路集成测试（从 Variant → patch → validation → runner handoff）
