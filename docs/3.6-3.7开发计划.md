# 0. 阶段定义与主流程

```
3.5 Experiment Planning Handoff
        ↓
3.6 Narrow Repository Read（仅读取已确认范围）
        ↓
3.6 RepositoryChangePlan（文件级修改计划）
        ↓
3.6 Conflict Analysis + Workspace Layout
        ↓
3.6 Patch Materialization（生成真实代码载荷）
        ↓
3.6 PatchPayload Manifest + Proposed Patch Diff
        ↓
3.6 Deterministic Payload Validation
        ↓
3.7 Approval Request
  ├── 用户审批 exact change IDs + exact payload SHA
  ├── approve / revise / reject
  └── 审批绑定 patch_payload_manifest_sha256 + proposed_patch_diff_sha256
        ↓
3.7 Application Preflight
  ├── plan SHA / payload manifest SHA / diff SHA 三方绑定
  ├── repository fingerprint 匹配
  ├── workspace 存在
  └── validation report passed
        ↓
3.7 Controlled Patch Application
        ↓
3.7 Local Validation（受控命令执行）
        ↓
patch_applied_and_local_validations_passed
        ↓
eligible_for_runner_intake
        ↓
3.8 Runner Intake & Experiment Execution Integration
```

**合并写一份计划是合理的，但必须保留"只读规划"（3.6）和"审批后写入"（3.7）两道强制边界。**

核心不变量：

```
approved_change_ids 是审批权威粒度
approved_paths 从 approved_change_ids 确定性派生
用户审批绑定的是确切代码 payload（patch_payload_manifest_sha256 + proposed_patch_diff_sha256）
policy allow scope 缺失 → preflight blocked（default-deny，无例外）
deny 永远不能被用户审批覆盖
3.7 成功终态只能是 patch_applied_and_local_validations_passed
Handoff 状态为 eligible_for_runner_intake，不是 runner_ready 或 experiment_ready
Change model 分离 change_kind 和 target_mode
任何影响 canonical SHA 的变化 → 旧审批自动失效 → 重新审批
```

## 3.1 PlannedRepositoryChange

```python
class PlannedRepositoryChange(BaseModel):
    change_id: str
    workspace_id: str
    change_kind: Literal["create", "modify", "delete", "rename", "configuration_only", "test_only"]
    target_mode: Literal["existing_target", "new_target"]
```

**关键规则**：
- `change_kind` 与 `target_mode` 解耦
- `create` → `target_mode=new_target`；文件已存在 + `must_not_exist` → 失败
- `modify/delete/rename` → `target_mode=existing_target`
- `rename_target_path` 必填；目标已存在 + `must_not_exist` → 失败

### `before_sha256` 与 `target_before_sha256` 的前态约束

为防止审批后目标内容漂移，每种 `change_kind` 对 `before_sha256` 及 `target_before_sha256`（针对 replace_existing 覆盖目标）有严格约束：

```text
create + must_not_exist
  → before_sha256 必须为 None，target_before_sha256 必须为 None
  → 运行时目标文件必须不存在

create + replace_existing
  → before_sha256 必须为 None
  → target_before_sha256 必填，必须等于目标文件当前 SHA
  → 被覆盖的前态内容通过 target_before_sha256 绑定在 Payload/审批合同中

modify
  → before_sha256 必填，必须等于当前文件 SHA
  → target_before_sha256 必须为 None

delete
  → before_sha256 必填，必须等于当前文件 SHA
  → target_before_sha256 必须为 None

rename
  → source before_sha256 必填，必须等于源文件当前 SHA
  → target_before_sha256 必须为 None（除非目标已存在且有 replace_existing）

rename + replace_existing（目标已存在）
  → source before_sha256 必填，必须等于源文件当前 SHA
  → target_before_sha256 必填，必须等于目标文件当前 SHA

configuration_only / test_only
  → before_sha256 必须为 None，target_before_sha256 必须为 None
  → 仅生成配置/测试骨架，不修改已有文件
```

`BeforeShaValidator` 在 Payload 生成后、审批请求前执行：
- 逐 change 验证 `before_sha256` / `target_before_sha256` 与非 None 约束一致
- 非 None 的 before_sha256 与当前仓库文件 SHA 比对
- 全局 `repository_fingerprint` 作为兜底，但不能替代逐 change 的局部前态合同

`target_before_sha256` 应加入 `PlannedRepositoryChange` 和 `PatchPayload` contract。

## 3.3 PatchPayload 生产者：PatchMaterializer

```
PatchMaterializer / CodeSynthesisAgent
  输入: RepositoryChangePlan + narrow repository context
  输出: PatchPayload + proposed_patch.diff + PatchPayloadManifest
  权限: 只读仓库、只写 artifact，不写工作树
```

`DeterministicPatchPayloadValidator` 检查：
- `payload.change_id` 存在且 `target_path` 与 change 匹配
- `payload_sha256` 匹配
- `before_sha256` 匹配当前文件
- diff 只触碰当前 workspace 中 RepositoryChangePlan 声明的路径
- diff 不新增未声明文件
- diff 不修改文件模式、symlink 或 submodule

> **Payload 与 Plan 的双向分离**：`RepositoryChangePlan` 不包含 `patch_payloads` 实体。Plan 中的 `payload_id` 仅作为前向引用（string ID），在 Materializer 生成 Payload 后**不写回 Plan**，Plan SHA 不因 Payload 生成而变化。`PatchPayloadManifest` 是 PatchPayload 实体的唯一事实源，通过 `payload.change_id` 可反向定位到 Plan 的 `change_id`。PatchPlanValidationReport 验证的是 Plan（change 语义完整性），PatchPayloadValidationReport 验证的是 Manifest 内全部 payload 的正确性，两者独立执行。

## 3.6 ApprovalPatchBundle

`approve_partial` 时，用户选择的 change 子集必须生成独立的 subset diff：

```python
class ApprovalPatchBundle(BaseModel):
    """部分审批时，批准的 change IDs 与对应的 subset diff 一起打包。"""
    bundle_id: str
    approval_request_id: str
    created_at: datetime
    patch_plan_sha256: str
    workspace_id: str

    approved_change_ids: list[str]         # 用户选择的子集
    approved_payload_ids: list[str]        # 对应 payload

    approved_diff_artifact_id: str         # subset-specific diff
    approved_diff_sha256: str

    payload_manifest_sha256: str           # 原始全量 manifest（不变）
    bundle_sha256: str                     # 本 bundle 的 canonical SHA
```

流程：

```
用户选择部分 changes
→ 生成 subset-specific diff（只含批准变更）
→ 计算 ApprovalPatchBundle SHA
→ 用户批准该 bundle
→ 3.7 只能应用该 bundle 中的 change IDs + diff
→ 全量 diff SHA 与 部分 change IDs 不再共用一个审批对象
```

`ApprovalPatchBundleValidator` 确定性验证：

```text
- bundle.approved_payload_ids == manifest 中 change_id ∈ approved_change_ids 的全部 payload IDs
- 无缺失 payload / 无额外 payload
- 无跨 workspace change（bundle.workspace_id == manifest.workspace_id）
- subset diff SHA（approved_diff_sha256）与投影结果一致
- subset diff 不包含未批准路径
- bundle_sha256 == canonical SHA(bundle)
```

## 3.7 审批失效规则

任何影响 canonical SHA 的变化 → 旧审批自动失效：

```
payload 生成后
workspace 重排后
conflict resolution 后
validation command 变化后
→ plan_changed → approval_invalidated → approval_pending
```

# 4. Path 安全模型（严格 default-deny）

```
1. path 或 ancestor 在 deny → DENY
2. change_id 未在 approved_change_ids → DENY
3. path 未从 approved_change_ids 派生 → DENY
4. path 在 ask-list 且未显式批准 → DENY
5. allow scope 缺失 → DENY
6. path 不在 allow scope → DENY
→ ALLOW
```

`allow scope` 必须显式配置。未配置 → preflight blocked，不进入写入。

```
模型级: validate_workspace_path（拒绝 ..、绝对路径）
运行时: resolve() + 确认在 repository_root 内
```

# 5. 创建与重命名冲突策略

```
create + target_mode=new_target
  ├── 文件不存在 → 创建
  └── 文件已存在
      ├── target_collision_policy=must_not_exist → patch_application_failed
      └── target_collision_policy=replace_existing → 需单独审批 → 保存 before_blob

rename + 目标路径
  ├── 目标不存在 → 正常 rename
  ├── 目标已存在 → 默认失败
  └── target_collision_policy=replace_existing → 单独审批 → 保存 target_before_blob
```

## 8.1 内部确定性检查（直接函数调用，非 subprocess）

以下检查由 AutoAD Core 直接执行受信任 Python 函数，不使用 `subprocess.run` 或 `python -c`：

```python
INTERNAL_VALIDATION_STEPS = [
```

对应实现函数：


内部检查不通过 subprocess，不涉及 `argv` 或 `executable`，不存在 `python -c` 风险。

## 8.3 范围边界

```
3.7 允许:
  - AST.parse() 语法检查（内部函数，不执行代码）
  - difflib diff 完整性校验（内部函数）
  - 路径 containment 检查（内部函数）
  - import 声明扫描（内部函数，不实际导入）

  - pytest / mypy / pylint（均导入并执行仓库代码）

  - 任何 shell 或 subprocess 网络操作
```

# 10bis. Canonical Hashing Specification

所有带自引用 SHA 的模型（即模型同时包含自身哈希字段）必须排除自身 SHA 参与 canonical hash 计算，否则计算无法闭合。

## 排除字段

```python
CANONICAL_HASH_EXCLUDED_FIELDS: dict[type[BaseModel], set[str]] = {
    ApprovalRequest: {"approval_request_sha256"},
    PatchPayloadManifest: {"manifest_sha256"},
    ApprovalPatchBundle: {"bundle_sha256"},
    RepositoryChangePlan: {"patch_plan_sha256"},
}
```

## 规范化规则

```
1. 序列化格式：UTF-8 JSON
2. 根对象为 dict，key 按 Unicode 码位升序排序（包括嵌套 dict）
3. 列表保持原序
4. JSON 分隔符固定为紧凑格式 `separators=(",", ":")`，不产生空格
5. None 值字段排除
6. datetime 序列化为 ISO-8601 UTC（如 "2026-06-17T00:00:00Z"），显式 `Z` 后缀而非 `+00:00`
7. ArtifactReference 的 artifact_id 字符串原样参与哈希
8. SHA-256 二进码 hex 摘要（64 个小写十六进制字符）
```

计算函数：

```python
import json, hashlib
from typing import Any

def canonical_sha(model: BaseModel) -> str:
    d = model.model_dump(exclude_none=True, mode="json")
    d = _exclude_fields(type(model), d)
    raw = json.dumps(d, ensure_ascii=False, sort_keys=True, separators=(",", ":")).encode("utf-8")
    return hashlib.sha256(raw).hexdigest()

def _exclude_fields(model_cls: type[BaseModel], d: dict[str, Any]) -> dict[str, Any]:
    excluded = CANONICAL_HASH_EXCLUDED_FIELDS.get(model_cls, set())
    if not excluded:
        return d
    return {k: v for k, v in d.items() if k not in excluded}
```

# 11. Preflight

在打开任何文件前强制检查。按五组验证：

```
A. Common integrity（approve_all 和 approve_partial 均适用）

  1. decision.approval_request_id == request.approval_request_id
  2. decision.workspace_id == request.workspace_id == manifest.workspace_id
  3. canonical_sha(request) == request.approval_request_sha256
       == decision.approved_request_sha256
  4. compute_canonical_plan_sha256(plan) == plan.patch_plan_sha256
  5. decision.patch_plan_sha256 == plan.patch_plan_sha256
       == request.patch_plan_sha256
  6. decision.payload_manifest_sha256 == manifest.manifest_sha256
       == request.patch_payload_manifest_sha256
  7. _fingerprint(repository_root) == plan.repository_fingerprint
  8. run_id == plan.run_id
  9. workspace_id ∈ plan.workspace_plans
 10. PatchPlanValidationReport:
      - sha(report) == request.patch_plan_validation_report_sha256
      - report.run_id == plan.run_id
      - report.patch_plan_sha256 == plan.patch_plan_sha256
      - report.status == passed, report.issues == []
 11. PatchPayloadValidationReport:
      - sha(report) == request.patch_payload_validation_report_sha256
      - report.patch_plan_sha256 == plan.patch_plan_sha256
      - report.payload_manifest_sha256 == manifest.manifest_sha256
      - report.status == passed, report.issues == []
 12. derive_paths(decision.approved_change_ids) == decision.approved_paths
 13. request.workspace_summary.workspace_id == request.workspace_id
 14. set(request.selected_variant_ids) == set(request.workspace_summary.variant_ids)

B. Full approval binding

 15. request.proposed_patch_diff_sha256 == decision.approved_diff_sha256
      == manifest.proposed_diff_sha256
 16. sha256(artifact_store.read(manifest.proposed_diff_artifact_id))
      == decision.approved_diff_sha256
 17. approved_change_ids == workspace 全部 non-blocked changes

C. Partial approval binding

 18. bundle.approval_request_id == request.approval_request_id
 19. decision.approval_patch_bundle_sha256 == canonical_sha(bundle)
 20. bundle.patch_plan_sha256 == plan.patch_plan_sha256
 21. bundle.payload_manifest_sha256 == manifest.manifest_sha256
 22. bundle.workspace_id == request.workspace_id
 23. set(decision.approved_change_ids) == set(bundle.approved_change_ids)
 24. sha256(artifact_store.read(bundle.approved_diff_artifact_id))
      == bundle.approved_diff_sha256
 25. Partial approval 必须运行完整的 ApprovalPatchBundleValidator：
      - approved_payload_ids 精确投影（无额外或缺失 payload）
      - subset diff 不包含未批准路径

D. Validation authorization

 26. approved_internal_step_ids ⊆ request.internal_validation_steps step IDs
 27. approved_external_command_ids ⊆ request.external_validation_commands IDs
 28. 所有 required=True 的 step/command 必须在批准集合中
 29. 所有 required=True 的 InternalValidationStep 的 target_artifact_ids 非空
 30. ast_parse 覆盖所有新增或修改的 Python payload
 31. diff_integrity 覆盖当前 workspace 的 proposed/subset diff
 32. path_containment 覆盖当前 workspace 全部 touched paths
 33. 已批准的 ExternalCommand.resolved_argv 与 Request 中的完全一致
 34. ExternalCommand.working_directory 在当前 workspace 内
 35. ExternalCommand.template_id 必须匹配注册模板，resolved_argv 精确对比：
      - template_id="ruff_check_no_fix"
        → argv 必须精确等于 ["ruff", "check", "--no-fix", "--no-unsafe-fixes"]
      - template_id="ruff_format_check"
        → argv 必须精确等于 ["ruff", "format", "--check"]
      拒绝：--fix、--unsafe-fixes、任意 config/plugin 参数、外部路径、未注册 template_id

E. Collision and path policy

 36. approved_collision_change_ids ⊆ approved_change_ids
 37. 所有 target_collision_policy=replace_existing 且目标当前存在的 change
       ∈ approved_collision_change_ids
 38. 无碰撞策略的 change ∉ approved_collision_change_ids
 39. allow scope 非空，否则 preflight blocked
 40. approved_paths ⊆ allowed scope
 41. approved_paths ∩ denied paths/denied ancestor == ∅
 42. 所有 ask-required approved path ∈ approved_ask_paths
 43. approved_ask_paths ⊆ approved_paths ∩ ask-list
 44. 所有 touched paths 通过 resolve containment
 45. 拒绝 symlink escape
```

# 12. 状态机

```
planning_initialized
→ repository_attested
→ patch_planning
→ patch_payload_generation
→ patch_payload_validating
→ approval_pending
    ├── approve → patch_applying
    ├── revise → patch_planning（或 payload 重生成）
    └── reject → rejected

patch_applying
    ├── full success → patch_applied
    ├── partial → patch_application_partial_failure
    ├── failed → patch_application_failed
    └── stale → replan_required

patch_applied → post_patch_validating
    ├── passed → patch_applied_and_local_validations_passed
    └── failed → patch_applied_but_local_validation_failed

patch_applied_and_local_validations_passed → eligible_for_runner_intake

recovery: repair_or_rollback_pending → rollback_running → rolled_back | rollback_failed
```
