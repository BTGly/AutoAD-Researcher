# 0. 修订说明

## 0.1 本轮修订目的

v1.1 已经将 Step 3.2 从早期“简单 PDF Reader”升级为 Agent-Native Paper Intelligence，并关闭了模型能力、预算字段、artifact 命名和页码索引四项问题。

v1.2 不推翻该技术内核，而是修正产品级阶段关系：

1. 3.1 与 Paper Intelligence 都是统一 Research Context Agent 调用的能力，不是面向用户彼此割裂的 Agent；
2. 3.2 除生成论文 artifacts 外，还必须接入用户任务、3.1 仓库 artifacts 和环境约束；
3. 3.2 必须输出 `research_context_draft.json`、gap/conflict/readiness artifacts；
4. 3.3 不再是固定下游主阶段，而是 `needs_clarification` 时触发的 Context Repair Addendum；
5. 证据缺口必须回到 3.1/3.2 Reader capability，不得让用户代替技术证据；
6. 原 Step 3.4 Single Idea Confirmation 与 Step 3.5 Transferability Judge 合并为 Step 3.4 Idea & Transfer Design；
7. Step 3.5 改为 Multi-variant Experiment Planner；
8. Paper Capability 成功与 Unified Context readiness 必须分开报告。

## 0.2 阶段顺序修订

正式关系调整为：

```text

├── 调用 Step 3.1 Repository Intelligence Capability
```

3.3 可 no-op，不是每个 run 必经的独立大阶段。

# 1. 执行摘要

## 1.1 阶段目标

Step 3.2 的产品级目标是建立统一 Research Context Agent；Paper Intelligence 是其中的核心信息获取能力之一。

Paper Capability 从用户 PDF、可选 arXiv URL、项目页或补充材料出发，在不执行论文内嵌动作、不联网下载任意资源、不引入向量数据库的前提下，生成结构化、可验证、可审计的论文理解结果。

统一 Context 层随后合并：

```text
用户原始任务与历史回答
3.1 Repository Intelligence artifacts
Paper Intelligence artifacts
环境与资源事实
系统 Policy
```

形成：

```text
research_context_draft.json
fact_ledger.json
context_gaps.json
context_conflicts.json
context_readiness_report.json
```

## 1.2 两类 Intelligence Capability 的关系

### Repository Intelligence Capability

回答：

```text
代码仓库是什么？
固定 commit 是什么？
入口、依赖、配置、评价、资产信号和可修改路径在哪里？
```

### Paper Intelligence Capability

回答：

```text
论文研究问题和核心方法是什么？
组件、训练目标、数据与标签假设是什么？
论文提到哪些 baseline/dataset/metric/idea source？
每个事实来自哪一页、哪一节、哪个 block/table/figure？
```

### Unified Research Context Agent

负责：

```text
保留不同来源的证据边界
对齐论文机制与 baseline 架构事实
识别跨来源冲突
区分用户决策缺口与 Reader 证据缺口
决定是否触发 3.3 Context Repair
生成进入 3.4 的稳定上下文
```

二者都不直接选择最终实验参数。

```text
paper_mentioned baseline ≠ selected baseline
paper_mentioned dataset  ≠ selected dataset
paper_mentioned metric   ≠ selected metric
paper_mentioned idea     ≠ confirmed idea
```

## 1.3 架构公式

```text
统一 Research Context Agent 负责上下文组装和 readiness
Repository/Paper Agent 负责按需获取对应来源事实
Parser Tool 负责确定性 PDF 文本/结构提取
Paper Tools 负责 section/page/block/table/figure 读取
Skill 负责阅读与上下文组装方法
Policy 负责只读、网络、资源和数据边界
Evidence Middleware 负责生成来源证据
Validator 负责验证 paper artifacts 与 research context
3.3 负责必要的用户澄清和 append-only ContextPatch
Artifacts 负责跨阶段交付
```

## 1.4 3.2 完成后应具备的能力

### Paper Capability

1. 接收用户 PDF 并计算稳定 source fingerprint；
2. 使用锁定 MinerU profile 解析 PDF；
3. 生成 page/section/block/figure/table/reference 结构；
4. 检测扫描、空页、乱码、OCR 降级和解析失败；
5. Agent 按需阅读而非一次吞全文；
6. 生成研究问题、方法、组件、目标、假设和迁移来源候选；
7. 提取 baseline/dataset/metric/repo/asset 候选；
8. 所有 critical claim 绑定 PaperEvidenceRef；
9. `paper_mentioned` 不自动成为正式参数；
10. 解析失败或证据不足时诚实输出 partial/failed/unknown。

### Unified Context Integration

11. 读取并验证 3.1 Repository artifacts；
12. 接入用户任务、环境和系统策略；
13. 生成统一 Fact Ledger，同时保留 paper/repository/user/policy/derived 类型；
14. 识别 ContextGap 和 ContextConflict；
15. 区分 `user_decision_required` 与 Reader evidence gaps；
16. 计算 ContextReadiness；
17. 无阻塞问题时让 3.3 no-op；
18. 需要用户决策时触发 3.3；
19. 需要技术证据时回到 3.1 或 Paper Capability；
20. 生成稳定 `research_context.json` 与 `idea_transfer_handoff.json`；
21. 直接服务 Step 3.4 Idea & Transfer Design。

# 2. 阶段定位与边界

## 2.1 3.2 解决的问题

```text
如何把用户 PDF 转成可导航、可引用、可验证的结构化文档？
如何从论文中提取科研事实和候选要素？
如何区分本文方法、对比 baseline、引用文献和数据集/指标候选？
如何防止 LLM 把论文提到的元素直接当成系统正式选择？
如何使每个论文 claim 可追溯到页、节、block、表格或图？
如何在 PDF 解析失败时不伪造 paper_summary？
```

## 2.2 3.2 不解决的问题

3.2 严格不做：

```text
不运行论文代码
不 clone 论文仓库
不选择最终 baseline/dataset/metric
不生成最终 Single Idea
不判断迁移是否可行
不生成 experiment_plan
不生成 patch_plan
不下载模型权重或数据集
不执行 PDF 内嵌脚本/附件/远程资源
不编译 LaTeX
不做全量向量数据库
不做多 Agent Idea 发散
不做自动文献综述系统
```

这些分别属于：

```text
Repository Intelligence
Idea & Transfer Design
Multi-variant Experiment Planner
Asset Manager
Future Literature Review System
```

## 2.3 MVP 深度支持范围

MVP 深度支持：

```text
英文 AI / ML / CV / AD 论文 PDF
arXiv 风格双栏 PDF
含公式、表格、图、引用的普通科研论文
10–40 页主文档
用户提供的本地 PDF 文件
```

浅层支持：

```text
中文论文 PDF
扫描 PDF
超长论文或补充材料
包含大量复杂表格的 benchmark paper
```

不在 MVP 正式支持：

```text
Word 论文
PPT
HTML-only 论文
音频/视频材料
完整 arXiv source tar 自动解析
完整综述型百页 PDF
```

## 2.4 输入类型

```text
A. 用户上传 PDF
   → Primary path，进入 MinerU parse

B. arXiv PDF URL
   → 下载策略由上层 Intake 控制；3.2 接收已落盘 PDF

C. arXiv HTML / AlphaXiv / 项目页
   → Web evidence，仅作为可选关联或补充，不作为主事实源

D. LaTeX source
   → 预留，MVP 不实现完整编译或 source analyzer
```

## 2.5 输出责任

Paper Intelligence Capability 输出的是“论文事实”和“候选”，不是“实验决策”：

```text
confirmed paper fact
paper_mentioned candidate
paper_derived_unconfirmed idea source
paper uncertainty
```

Unified Research Context 层输出：

```text
research_context_draft.json
fact_ledger.json
context_gaps.json
context_conflicts.json
context_readiness_report.json
research_context.json（ready 后）
idea_transfer_handoff.json（ready 后）
```

正式 Idea 与迁移设计发生在：

```text
Step 3.4 Idea & Transfer Design
```

实验计划发生在：

```text
Step 3.5 Multi-variant Experiment Planner
```

3.3 只在上下文需要用户补充、确认或修复时触发。

# 3. 与渐进式工程化路线的关系

## 3.1 单一 Idea 纵向闭环

当前 MVP 仍坚持：

```text
一个 run
→ 一个用户确认的科研 Idea
→ 一个 baseline
→ 同一 Idea 下 1–3 个实现 Variant
→ 用户选择一个或多个 Variant
→ 一个共享评价协议下的实验计划包
→ 受控执行与真实指标
```

多个 Variant 不等于多个科研 Idea。

## 3.2 两类信息能力，一个连续研究 Agent

```text
3.1 Repository Intelligence Capability
3.2 Paper Intelligence Capability
```

两者在工程上独立测试和验证，在产品交互上由同一个上层 Research Context Agent 连续调用。

用户不应看到“Reader A 退出、Reader B 退出、Clarifier C 重新开始”的割裂体验。

## 3.3 3.3 是补充机制

```text
上下文完整 → 3.3 no-op
用户偏好/确认缺失 → 3.3 提问并生成 ContextPatch
论文证据缺失 → 回到 Paper Capability
仓库证据缺失 → 回到 3.1 Capability
```

## 3.4 下游阶段

```text
Step 3.4 Idea & Transfer Design
  确认 Idea
  对照 baseline
  检查语义、shape、训练假设、评价和工程边界
  生成 1–3 个实现 Variant
  用户选择

Step 3.5 Multi-variant Experiment Planner
  为所有选中 Variant 生成共享 baseline/dataset/metric/evaluator 下的实验计划包
```

## 3.5 不做开放式多 Agent 发散

3.2 不做：

```text
多 Agent 自由产生不同科研方向
候选 Idea 投票
跨论文自动扩展
多分支实验自动启动
```

这些不属于当前真实纵向闭环的前置条件。

## 5.2 Unified Context 组件职责

| 组件 | 负责 | 不负责 |
|---|---|---|
| Context Assembler | 合并已验证 artifacts 和用户任务 | 重新解析 PDF/仓库 |
| Fact Ledger | 保存来源类型与证据 | 将推断伪装成事实 |
| Context Gap Classifier | 区分用户决策与证据缺口 | 向用户询问技术事实 |
| Context Conflict Detector | 表达 paper/repo/user/policy 冲突 | 静默覆盖来源 |
| Context Readiness Validator | 决定 ready/repair/reanalysis/blocked | 生成 Idea |
| 3.3 Addendum | 用户澄清与 ContextPatch | 重新读论文/仓库 |

## 5.3 不采用的架构

```text

Paper Reader 完成后无条件进入 3.3
用户回答覆盖 paper/repository facts
所有 unknown 都问用户
一次性全文 RAG/vector DB 作为 MVP 前置
```

# 6. 关键设计决策

## D1：3.2 是 Unified Research Context 主阶段

Paper Intelligence 是其核心能力，不是唯一职责。

## D2：3.1 与 Paper Intelligence 是独立能力

证据模型、工具和安全策略分别验证；由统一 Agent 调用。

## D3：3.3 是条件式 Context Repair

无 blocking gap 时 no-op；不得作为固定下游大阶段。

## D4：3.4 合并 Idea Confirmation 与 Transferability

```text
Step 3.4 Idea & Transfer Design
```

## D5：Step 3.5 是 Multi-variant Experiment Planner

一个 Idea 可以产生 1–3 个实现 Variant，并对用户选择的多个 Variant 生成计划包。

## D6：MinerU 是 P0 唯一正式 PDF Parser

MarkItDown 不作为 PDF 主链 fallback。

## D7：AlphaXiv 只作为交互和导航参考

可作为 optional web association source，不是论文正文事实源。

## D8：Paper Evidence 不只记录页码

必须绑定：

```text
PDF SHA
parser manifest
physical page index
section path
block/table/figure/reference ID
content SHA
tool call ID
```

## D9：不引入向量数据库

MVP 使用 section/page/block index、keyword search 和 Agent tool calls。

## D10：候选与正式参数严格分离

```text
PaperMentionedCandidate = 来源事实
CandidateDecision       = 用户后续决策
```

不得原地修改候选的来源语义。

## D11：Capability status 与 Context readiness 分离

```text
paper_capability_status
context_readiness_status
```

## D12：技术证据缺口不问用户

```text
paper_evidence_required      → Paper Capability
repository_evidence_required → 3.1 Capability
user_decision_required       → 3.3
```

# 7. MinerU / MarkItDown / AlphaXiv 策略

## 7.1 MinerU 主路径

MinerU Provider 需要锁定：

```text
package version
backend profile
model revision
model weight SHA
OCR language
parse config SHA
device profile
container or runtime fingerprint
```

输出：

```text
pages.jsonl
blocks.jsonl
sections.json
figures.json
tables.json
references.json
parse_quality_report.json
parser_manifest.json
```

## 7.2 MinerU Provider 接口

```python
class DocumentParseRequest(BaseModel):
    schema_version: Literal[1]
    source_id: str
    source_pdf_path: str
    parser_profile_id: str
    page_range: PageRange | None = None
    ocr_policy: Literal["auto", "disabled", "forced"]
    language_hints: list[str]
    max_pages: int
    max_runtime_seconds: int
```

```python
class DocumentParseResult(BaseModel):
    schema_version: Literal[1]
    parse_attempt_id: str
    source_id: str
    parser_manifest_path: str
    canonical_output_path: str
    parse_quality_report_path: str
    status: Literal["success", "partial_success", "failed"]
    warnings: list[str]
```

## 7.3 MarkItDown 定位

MarkItDown 不进入 3.2 PDF 主链。保留为未来 optional generic document adapter。

可借鉴点：

```text
converter abstraction
optional dependencies
plugin opt-in
narrow input interface
low-resource text conversion
```

不得做：

```text
MinerU failed → silently fallback to MarkItDown → generate normal summary
```

## 7.4 AlphaXiv 定位

AlphaXiv 借鉴：

```text
按章节导航
Assistant / Notes / Comments
相似论文
代码链接
多视图展示
局部讨论
```

不依赖：

```text
不把 AlphaXiv 摘要作为论文事实源
不把页面结构当作稳定 API
不因 AlphaXiv 不可用阻塞 3.2
```

```text
primary reported metric
secondary metric
loss term
evaluation protocol
unrelated metric in citation
```

## 17.3 Conflict handling

Examples:

```text
Abstract says unsupervised, method section uses labels
Table reports AUROC, appendix says AP
Paper text mentions code released, URL missing
Dataset appears only in comparison table, not method
```

Output conflicting claims, not averaged statements.

## 17.4 Unknown handling

Unknown is a valid output. Do not invent:

```text
implementation details
dataset split
training objective
ablation conclusion
code URL
```

# 19. Evidence Validator

## 19.1 Evidence validation

For every evidence:

```text
source exists
PDF SHA matches
parser output SHA matches
page index valid
block/table/figure/reference exists
content SHA recomputable
tool_call exists
trust_level allowed for claim type
```

## 19.2 Claim validation

### confirmed

```text
must have evidence_ids
must use paper_text/table/figure/reference for paper body facts
must not depend only on web_page or alpha_xiv_page
```

### inferred

```text
must have indirect evidence
must have rationale_summary
must not use verified/final/selected wording
```

### conflicting

```text
must have at least two evidence IDs
conflict reason must be explicit
```

### unknown

```text
may have search/read evidence showing absence
must not invent evidence
```

## 19.3 Candidate validation

Reject:

```text
paper_mentioned baseline marked selected
compared_baseline treated as proposed_method
cited_only treated as used_in_experiment
repository URL treated as resolved source
web summary used as paper body evidence
```

## 19.4 Parser validation

Reject:

```text
paper_summary exists when parse status failed
confirmed claim from failed page
physical_page_index out of bounds
printed_page_label used for deterministic page lookup
```

## 19.5 Cross-stage consistency

Check with 3.1 outputs if present:

```text
paper repository candidate URL is only candidate unless resolved by 3.1
repository implementation conflicts are marked conflicting
baseline selected elsewhere must have user/system confirmation
```

## 21.3 Reader Reanalysis

```text

environment_evidence_required → Environment Profiler
```

不得问用户“论文第几页写了什么”或“仓库实际 tensor shape 是多少”来替代证据。

## 21.4 To Step 3.4 Idea & Transfer Design

`idea_transfer_handoff.json` 至少包含：

```text

protected/evaluation constraints
paper idea source candidates
repository architecture facts
cross-source conflicts
known transfer constraints
accepted risks
unresolved nonblocking gaps
```

3.4 完成：

```text
确认一个科研 Idea
比较论文机制和 baseline 架构
检查输入输出 shape、训练假设、评价和工程边界
生成 1–3 个同一 Idea 下的实现 Variant
让用户选择
```

## 21.5 To Step 3.5 Multi-variant Experiment Planner

3.2 不直接输出实验计划。3.5 只消费 3.4 中用户已选 Variant，并在共享 baseline/dataset/metric/evaluator 下生成实验计划包。

## 21.6 To Step 3.1 Repository Intelligence

论文中的 repository URL 继续输出 unresolved candidate，由 3.1 Discovery / Resolution 验证；3.2 不 clone 或自行确认官方性。
