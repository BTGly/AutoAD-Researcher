# AutoAD-Researcher Step 3.3 开发计划：Context Repair & Clarification Addendum

> 文档状态：修订评审稿 v2.1（与 3.2 v1.2 contracts 同步）  
> 更新日期：2026-06-17  
> 阶段名称：**Context Repair & Clarification Addendum / 上下文修复与澄清补充层**  
> 前置能力：Step 3.1 Repository Intelligence Capability、Step 3.2 Unified Research Context Agent / Paper Intelligence  
> 下一主阶段：Step 3.4 **Idea & Transfer Design**（合并原 Single Idea Confirmation 与 Transferability Judge）  
> 架构方案：**统一上层研究 Agent + 能力型 Reader + 补充式 Clarification + Deterministic Context Validation**  
> 核心原则：**3.3 不再是独立主流程阶段，而是 3.2 统一上下文 Agent 的修复、澄清、确认和增量更新机制。**

---

# 0. 修订结论

原 3.3 被设计成一个独立的 `Evidence-based Intent Clarifier` 主阶段。这个设计在技术上可实现，但产品流程上不自然：

```text
3.1 仓库 Reader 完成
→ 3.2 论文 Reader 完成
→ 3.3 Clarifier 换一个 Agent 继续问
→ 3.4 Single Idea Confirmation
→ 3.5 Transferability Judge
```

新的判断是：

```text
3.1 / 3.2 是供统一研究 Agent 调用的两类信息获取能力；
Clarification 应该嵌入统一上下文构建过程；
Single Idea Confirmation 与 Transferability Judge 应合并成一个 Idea & Transfer Design 阶段。
```

因此 3.3 改为：

```text
Context Repair & Clarification Addendum
```

它不是新的 Reader，不是新的科研判断 Agent，也不是必经的大阶段。它是 3.2 统一上下文 Agent 的补充机制，用于：

```text
1. 修复 research_context.json 中的缺口、冲突和低置信字段；
2. 向用户提出少量真正阻塞后续决策的问题；
3. 记录用户原始回答；
4. 将用户回答归一化为 context patch；
5. 验证 patch 不会破坏证据边界；
6. 生成进入 3.4 Idea & Transfer Design 的稳定上下文。
```

# 1. 新阶段关系

## 1.1 内部能力与用户流程分离

内部能力：

```text
Repository Intelligence Capability
  负责仓库来源、commit、入口、依赖、评价、可修改路径和代码证据。

Paper Intelligence Capability
  负责论文来源、PDF、parser、方法、组件、候选 baseline/dataset/metric 和论文证据。
```

用户看到的连续流程：

```text
统一 Research Context Agent
  ↓
调用 3.1 仓库能力
调用 3.2 论文能力
读取用户原始任务
合并事实与证据
发现缺口与冲突
必要时进入 3.3 clarification/repair
  ↓
输出稳定 research_context.json
  ↓
进入 3.4 Idea & Transfer Design
```

## 1.2 3.3 的新定位

3.3 不是：

```text
一个独立会话 Agent
一个固定问卷系统
一个新的信息读取器
一个新的科研 Idea 生成器
一个 Transferability Judge
一个 Experiment Planner
```

3.3 是：

```text
上下文补丁层
用户确认层
缺口修复层
跨来源冲突标注层
进入 3.4 前的上下文稳定器
```

## 1.3 3.4 / 3.5 合并后的下游

原来：

```text
3.4 Single Idea Confirmation
3.5 Transferability Judge
```

合并为：

```text
3.4 Idea & Transfer Design
```

3.4 内部循环：

```text
确认用户真正想迁移的 Idea
→ 对照 baseline 架构检查可迁移性
→ 检查输入输出、训练假设、评价边界和工程风险
→ 生成 1–3 个同一 Idea 下的实现 Variant
→ 用户选择一个或多个 Variant
```

3.3 的输出必须直接服务这个阶段。

# 3. 3.3 核心任务

## 3.1 任务定义

3.3 负责处理以下问题：

```text
3.2 已读材料后仍不能确定用户意图；
论文事实与仓库事实冲突；
用户输入与论文/仓库事实冲突；
baseline、dataset、metric、修改边界或资源预算需要用户偏好；
某些字段需要用户显式确认才能进入 3.4；
3.2 的 research_context_draft 未通过 Context Validator；
用户修改或否认了系统理解；
用户新增约束后需要修复上下文。
```

## 3.2 一句话目标

> **把已读材料和用户回答整理成一个稳定、可验证、可进入 Idea & Transfer Design 的研究上下文。**

## 3.3 完成后的能力

完成后系统应能：

1. 接收 3.2 生成的 `research_context_draft.json`；
2. 读取 3.1 / 3.2 的证据索引；
3. 识别上下文中的 blocking gaps；
4. 区分事实缺失、用户偏好缺失和系统策略缺失；
5. 每轮向用户提出不超过 3 个高价值问题；
6. 避免重复询问已由论文、仓库或用户明确给出的内容；
7. 记录用户原始回答，不覆盖历史；
8. 将回答归一化为 `ContextPatch`；
9. 验证用户回答能覆盖哪些 gap；
10. 对用户回答与证据冲突的内容标记 conflict，不静默覆盖；
11. 允许用户 revise / reject 系统理解；
12. 支持 3.2 context validator 失败后的定向修复；
13. 在无阻塞问题时 no-op 成功；
14. 输出 `research_context.json` 的稳定版本；
15. 输出 `idea_transfer_handoff.json` 给 3.4。

# 8. 统一 Research Context 模型

## 8.1 ResearchContext

3.3 直接复用 3.2 v1.2 的共享 `ResearchContext`，不得复制另一套 schema：

```python
class ResearchContext(BaseModel):
    schema_version: Literal[1]
    run_id: str
    context_id: str
    context_version: int

    task: TaskContext
    sources: SourceContext
    paper: PaperContext | None
    repository: RepositoryContext | None
    baseline: BaselineContext | None
    dataset: DatasetContext | None
    metrics: MetricContext
    constraints: ConstraintContext
    user_preferences: UserPreferenceContext
    idea_sources: list[IdeaSourceContext]

    facts: list[ContextFact]
    gaps: list[InformationGap]
    conflicts: list[ContextConflict]
    readiness: ContextReadiness

    evidence_index_refs: list[str]
    previous_context_id: str | None
    context_sha256: str
```

`research_context_draft.json` 与 `research_context.json` 使用同一 schema；前者允许 readiness 非 ready，后者必须为 `ready_for_idea_transfer_design`。

## 8.2 Fact 类型

3.3 直接复用 3.2 v1.2 的共享 `ContextFact`：

```python
class ContextFact(BaseModel):
    fact_id: str
    fact_type: Literal[
        "paper_fact",
        "repository_fact",
        "user_fact",
        "environment_fact",
        "system_policy_fact",
        "derived_hypothesis",

    subject: str
    predicate: str
    value: Any
    status: Literal["confirmed", "inferred", "conflicting", "unknown"]
```

规则：

```text
paper_fact 必须来自 PaperEvidenceRef
repository_fact 必须来自 Repository EvidenceRef
user_fact 必须来自 UserInputEvidenceRef
environment_fact 必须来自 Environment EvidenceRef
system_policy_fact 来自 policy/config
derived_hypothesis 必须引用至少一个基础 fact
```

## 8.3 用户回答不能覆盖客观事实

用户可以确认：

```text
目标
偏好
预算
是否允许修改
想优先验证哪个模块
```

用户不能直接把以下内容改成客观事实：

```text
论文是否真的提出某方法
仓库是否真的包含某入口
baseline 代码是否真的支持某接口
评价脚本是否真的未修改
```

```text
“这个 repo 就是官方实现”
```

```text

```

```text

```

# 9. Information Gap 模型

## 9.1 Gap 类型

3.3 复用 3.2 v1.2 的共享 `InformationGap`：

```python
class InformationGap(BaseModel):
    gap_id: str
    gap_type: Literal[
        "user_decision_required",
        "paper_evidence_required",
        "repository_evidence_required",
        "environment_evidence_required",
        "system_policy_conflict",
        "unresolvable",
    ]
    category: Literal[
        "task_goal",
        "idea_scope",
        "baseline_selection",
        "dataset_selection",
        "metric_selection",
        "modification_boundary",
        "external_asset_permission",
        "resource_budget",
        "evaluation_protocol",
        "implementation_preference",
        "source_conflict",
        "policy_conflict",
        "reader_reanalysis_needed",
    ]
    severity: Literal["blocking", "high", "medium", "low"]
    question_needed: bool
    reason: str
    downstream_impact: str
    related_fact_ids: list[str]
    related_evidence_ids: list[str]
    resolution_target: Literal[
        "3.3_context_repair",
        "paper_intelligence",
        "repository_intelligence",
        "environment_profiler",
        "stop",
    ]
    resolved_by_patch_id: str | None = None
```

3.3 只能处理：

```text
user_decision_required + resolution_target=3.3_context_repair
```

其他 gap 必须 no-question 并返回对应 capability。

## 9.2 blocking gap

进入 3.4 前必须解决或显式接受风险的 gap：

```text
目标 baseline 不明确
用户想迁移的 Idea 完全不明确
任务类型不明确
评价指标不明确
是否允许修改 protected path 不明确
数据集/类别不明确且影响实验设计
外部资产/权重许可不明确
论文与仓库冲突影响迁移位置
```

## 9.3 high gap

可进入 3.4，但需要在 3.4 中标记风险：

```text
资源预算粗略
实现偏好未完全确认
部分可选 metric 未确认
非关键依赖是否允许新增
```

## 9.4 no-question gap

有些 gap 不应该问用户，而应该回到 Reader：

```text
论文 page evidence 缺失
仓库 entrypoint evidence 缺失
candidate role evidence 不足
parser 质量低导致事实无法确认
```

这类 gap 输出 reanalysis request，而不是向用户问：

```text
“论文第 5 页到底写了什么？”
```

# 10. Clarification Question 模型

## 10.1 ClarificationQuestion

```python
class ClarificationQuestion(BaseModel):
    question_id: str
    gap_ids: list[str]
    prompt: str
    question_type: Literal[
        "single_choice",
        "multi_choice",
        "yes_no",
        "free_text",
        "confirm_summary",
    ]
    options: list[QuestionOption] = []
    default_option_id: str | None = None
    why_this_matters: str
    evidence_ids: list[str]
    priority: Literal["blocking", "high", "medium", "low"]
    max_expected_answer_length: int | None
```

## 10.2 QuestionOption

```python
class QuestionOption(BaseModel):
    option_id: str
    label: str
    normalized_value: Any
    consequence: str
    risks: list[str]
    evidence_ids: list[str]
```

## 10.3 问题生成规则

每轮最多：

```text
3 个问题
```

优先级：

```text
blocking > high > medium > low
```

合并规则：

```text
同一决策影响的多个 gap 合并成一个问题
同一用户回答可解决多个 gap 时合并
已由证据确认的字段不问
已由用户明确回答过的问题不重复问
```

## 10.4 问题措辞

问题必须：

```text
简短
说明为什么需要
给出可选项
说明默认建议但不强迫
标明不确定来源
```

示例：

```text
我已经看到论文模块 M 需要在训练阶段更新参数，但当前 PatchCore baseline 默认冻结 backbone。
进入迁移设计前，需要确认你的约束：

A. 只允许特征后处理，不训练 backbone
B. 允许训练新增模块，但不改 backbone
C. 允许端到端微调
D. 先不确定，保守按 A 设计
```

## 11.2 NormalizedUserDecision

```python
class NormalizedUserDecision(BaseModel):

    source_answer_ids: list[str]

        "task_goal",
```

## 11.3 含糊回答处理

用户回答：

```text

```

处理：

```text

```

## 11.4 用户修正

用户可以说：

```text
“不是这个，我要验证的是另一个模块。”
```

系统动作：

```text
append answer
create revision decision
mark old decision superseded
create context patch
rerun validator
```

不得删除旧回答。

# 12. Context Patch 模型

## 12.1 ContextPatch

```python
class ContextPatch(BaseModel):
    patch_id: str
    base_context_id: str
    base_context_version: int
    patch_type: Literal[
        "add_fact",
        "update_preference",
        "resolve_gap",
        "mark_conflict",
        "supersede_decision",
        "request_reanalysis",
    ]
    target_path: str
    old_value: Any | None
    new_value: Any | None
    reason: str
    evidence_ids: list[str]
    source_decision_ids: list[str]
    created_at: datetime
```

## 12.2 Patch 应用规则

```text
ContextPatch append-only
不得直接原地修改旧 context
每次应用生成 context_version + 1
context_sha256 重算
patch 与新 context 一起验证
```

## 12.3 Patch 限制

不允许 patch：

```text
修改 paper evidence 内容
修改 repository evidence 内容
修改 parser manifest
修改 commit SHA
修改 PDF SHA
删除历史回答
删除旧 context
```

允许 patch：

```text
用户偏好
任务目标
选择的 baseline label
允许修改范围
预算
Idea scope
resolved gap status
context conflict interpretation
```

# 13. Context Validator

## 13.1 目标

验证修复后的 `research_context.json` 是否可进入 3.4。

## 13.2 必须验证

```text
schema 合法
context_sha256 可重算
所有 fact 的 evidence 存在
user decision 引用 raw answer
patch base version 匹配
old context 未被覆盖
blocking gaps 已解决或显式 deferred
candidate selection 没有泄漏
paper_mentioned 没有变成 selected parameter
repository facts 没有被 user assertion 覆盖
protected path 修改需要显式确认
evaluation protocol 修改需要显式确认
```

## 13.3 Readiness

3.3 复用 3.2 v1.2 的共享 `ContextReadiness`：

```python
class ContextReadiness(BaseModel):
    status: Literal[
        "ready_for_idea_transfer_design",
        "needs_clarification",
        "needs_reader_reanalysis",
        "blocked_by_policy",
        "blocked_by_user",

    blocking_gap_ids: list[str]
    unresolved_conflict_ids: list[str]
    accepted_risk_ids: list[str]

        "paper_intelligence",

        "3.3_context_repair",
```

```text
status=ready_for_idea_transfer_design
```

```text
status=needs_reader_reanalysis
next_stage=3.2_reanalysis
```

```text
无用户原始任务
无可用 baseline 或 baseline gap 未处理
Idea source 完全不明确且用户未接受“先由 3.4 生成候选”
没有任何合法 metric 候选
评价协议被用户要求修改但未进入 high-risk confirmation
```

# 15. 用户交互协议

## 15.1 Turn Structure

每次向用户输出：

```text
当前理解摘要
关键证据来源简述
最多 3 个问题
每个问题为什么重要
选项与后果
默认保守建议
```

## 15.2 Confirmation Protocol

高风险字段必须显式确认：

```text
confirm
revise
reject
```

高风险字段：

```text
允许修改 evaluator
允许使用异常标签
允许下载外部权重
允许新增大型依赖
允许改变 metric/post-processing
允许跳过 baseline run
允许使用非官方数据集镜像
```

## 15.3 不骚扰用户

如果系统可以从证据中安全确定，不能问。

例如仓库中 evaluation script 明确存在，3.3 不应问：

```text
“你的 evaluation script 在哪里？”
```

而应只在需要时问：

```text
“是否允许修改该 evaluation script？”
```

## 15.4 用户沉默

用户不回答时：

```text
低风险字段可使用保守默认
高风险字段 blocked
进入 partial context 不进入 3.4
```

# 24. 风险与控制

## 风险 1：3.3 再次变成独立大阶段

控制：

```text
3.3 只处理 context patch
3.3 不读论文正文、不读仓库文件
3.3 不生成 Idea
3.3 可 no-op
```

## 风险 2：用户回答覆盖证据事实

控制：

```text
用户回答只生成 user_fact
paper/repo fact 只能由对应 evidence 支持
冲突标记 conflict
```

## 风险 3：问太多问题

控制：

```text
每轮最多 3 问
只问 blocking/high
合并相关问题
低风险使用保守默认
```

## 风险 4：上下文 patch 不可追踪

控制：

```text
append-only patch
context_version
context_sha
old context 保留
```

## 风险 5：3.4 仍重复澄清

控制：

```text
IdeaTransferHandoff 明确已确认字段
3.4 只针对迁移设计中新发现的技术冲突追问
```

# 26. 更新后的主线

最终建议主线：

```text
3.1 Repository Intelligence Capability
  - 仓库读取能力
  - 已封板

  - 论文读取
  - 仓库能力接入
  - 用户任务接入
```

```text
一个连续的研究助手身份
```
