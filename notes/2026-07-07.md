## Entry 12 — Make need discovery LLM-first

### Goal

纠偏 Entry 11 的定位：`RequirementNeed` / `RequiredNeedSpec` 和 stage-sensitive readiness 应作为 schema、validator 和 deterministic fallback，而不是主意图理解器。主链路在有 API key 时必须先走 LLM NeedSpec，再由 core 做 canonicalization 与安全校验。

### Operations

- 阅读附件 `/root/.codex/attachments/b7cbac52-f4eb-468b-ac45-468348232db7/pasted-text.txt`。
- 修改 `assistant/v2/need_discovery.py`：
  - 新增 `discover_required_needs_with_llm()`。
  - 有 API key 时构造 Need Discovery 专用 system prompt，输入 user input、transcript、existing contract draft、source registry、usable evidence、jobs、answerability、run artifacts summary。
  - LLM 必须输出 `RequiredNeedSpec` JSON。
  - Core 解析 JSON 后执行 `RequiredNeedSpec.model_validate()`、`canonicalize_need_values()`、`validate_need_spec()`。
  - 无 API key、LLM error、非法 JSON 或 schema validation 失败时才回退到 deterministic `discover_required_needs()`。
  - 调用方传入的 `current_stage_goal` 优先级高于 LLM 输出，避免 run 阶段被模型降成 plan 阶段绕过校验。
  - validator 规则加强：
    - `improvement_idea` / `target_module` 永远 optional、non-blocking。
    - plan 阶段不能让 `dataset_path`、`python_env`、`gpu`、`cuda`、`time_budget` blocking。
    - `entrypoint` / `config` / `baseline_entrypoint` / `baseline_config` 是 repo analyzer auto-fillable，不能要求用户手写。
    - run 阶段即使 LLM 漏掉 `dataset_path`、`python_env`、`time_budget`、`human_review_policy`，validator 也会补成 required_now 并计算 blocking。
- 修改 `assistant/v2/intent_contract.py`：
  - `build_contract_from_context()` 接受 `existing_contract_draft`、`api_key`、`provider_url`。
  - contract 构建时调用 LLM-first `discover_required_needs_with_llm()`。
- 修改 `assistant/v2/orchestrator.py`：
  - 调用 contract builder 时传入 API 配置和 existing draft，确保主路径使用 LLM Need Discovery。
- 更新 `tests/test_v2_need_discovery.py`：
  - mock LLM 返回“不需要 dataset”的代码诊断任务，系统不强制 dataset。
  - mock LLM 把 `improvement_idea` 标成 blocking，validator 降为 optional。
  - mock LLM 在 plan 阶段要求 GPU，validator 降为 required_later。
  - mock LLM 在 run 阶段漏掉运行必需品，validator 补齐并 blocking。
  - 无 API key 时 fallback 仍能处理 PatchCore/MVTec smoke case。
