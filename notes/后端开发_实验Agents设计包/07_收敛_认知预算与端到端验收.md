## 3. StuckDetector

### 3.1 两个来源，两层分工

| 检测方式 | 来源 | 成本 | 适用场景 |
|----------|------|------|----------|
| **Step Signature（轻量）** | MiMo `stableStringify()` + `stepSignature()` | 0 token，O(1) | 每轮都跑：Coordinator/Executor 的重复 tool call |
| **5-Mode Detector（深度）** | software-agent-sdk `stuck_detector.py`（320行） | 0 token，O(n) 事件扫描 | 每 N 轮或 stepSig 报警时触发 |

### 3.2 Step Signature（直接复用 MiMo）

> **来源：** `/root/autodl-tmp/repos/mimo-code/packages/opencode/src/session/prompt.ts` lines 109-148。

```python

    """确定性 JSON 序列化（sorted keys）。"""
    return json.dumps({"tool": tool_name, "args": args}, sort_keys=True)

def step_signature(parts: list) -> str:
    """只取 tool name + args，排除 text/reasoning。"""
    # 关键设计（MiMo line 133-137）:
    # "text and reasoning are excluded on purpose...
    #  counting either would mask the repeated action we want to catch."
    ...

# 检查: 最近 3 次 assistant 消息的 stepSignatures 是否全部相同
if len(set(signatures[-REPEATED_STEP_THRESHOLD:])) == 1:

```

**适用：** Coordinator 用同一 tool+参数连续调用 3 次；Executor 用同一 SEARCH/REPLACE block 连续提交 3 次。

### 3.3 5-Mode Detector（直接复用 software-agent-sdk）

> **来源：** `/root/autodl-tmp/repos/software-agent-sdk/openhands-sdk/openhands/sdk/conversation/stuck_detector.py`（320 行，零外部依赖）。直接复制 `StuckDetector` 类和 5 个 `_is_stuck_*` 方法。

| Mode | 方法 (line) | 阈值 | 检测什么 | 关键细节 |
|------|-----------|------|----------|----------|
| 1. Repeating Action-Observation | `_is_stuck_repeating_action_observation` (140-174) | 4 | 同一 tool + 同一返回内容重复 4 次 | 用 `_event_eq()` 去元数据比较 |
| 2. Repeating Action-Error | `_is_stuck_repeating_action_error` (176-198) | 3 | 同一 tool + AgentErrorEvent 重复 3 次 | 更快触发（错误循环比正常循环更危险） |
| 3. Monologue | `_is_stuck_monologue` (200-225) | 3 | 连续 3+ 条 agent 消息无 tool call（空谈） | CondensationSummary 被跳过 |
| 4. Alternating ABAB | `_is_stuck_alternating_action_observation` (227-262) | 6 | A→B→A→B→A→B 交替模式 | Mode 1 和 2 会漏掉 |
| 5. Context Window Error | `_is_stuck_context_window_error` (264-273) | 10 | 重复的 context overflow error | 当前是 stub（line 273: `return False`） |

**关键设计细节（来自 sdk）：**
- **窗口在用户消息处重置**（line 72-82）：Coordinator 的 OBSERVE 后如果有新的工具结果（= 新信息），窗口重置——不误判。
- **`_event_eq()` 忽略 ID 和 metrics**（line 275-320）：两次 tool call 虽然有不同的 call_id 和 timing，但语义相同→判定为相等。
- **不同阈值**：error loop 最快（3），action-observation 中等（4），alternating 最慢（6）。不是全用一个魔法数字。

### 3.4 适配 AutoAD 的额外检测

| 检测 | 触发条件 |
|------|----------|
| repeated same patch | patch.diff SHA256 连续 2 次未变 |
| repeated same failing command | subprocess exit_code 相同且 stderr 内容相同 |
| no artifact progress | 连续 N 轮无新的 metrics.json / checkpoint |
| cyclic tree actions | Coordinator 对同一 node 反复 add→prune→add |

## 7. 全链路测试基准

建立小型 `AD-AgentBench` fixture，不追求模型 SOTA，测试闭环能力。

### Case 1：明确有效的小参数优化

- baseline 可运行；
- 一个参数变化提升超过 noise；
- 预期：candidate→confirm→champion。

### Case 2：实现未生效

- patch 成功；
- 实际路径未触发；
- 预期：UNVERIFIED→repair，不 prune idea。

### Case 3：实现有效但指标下降

- VERIFIED + COMPARABLE；
- 预期：NOT_SUPPORTED；
- 写 failure-WHY；
- 派生或 pivot。

### Case 4：结果在噪声内

- 预期：补 seed 或 NO_EFFECT；
- 不提升 champion。

### Case 5：GPU OOM

- Sentinel 分类；
- 无 LLM 实时监控；
- bounded repair/retry。

### Case 6：训练 hang

- stale heartbeat；
- TERM/KILL；
- Job 有明确状态；
- ResourceLease 释放。

### Case 7：Coordinator 重启

- 删除 checkpoint；
- 从 Session/Tree/Commit 恢复；
- 不重复已完成 Job。

### Case 8：便宜实验批次

- 一次生成 3 variants；
- 3 个 Result Integration；
- 批次结束后一次 Compact Cycle。

### Case 9：停滞

- 重复 idea / low KEEP；
- StrategyDiagnostic；
- overlay；
- 新方向出现；
- prompt change 有审计。

### Case 10：评估作弊

- 修改 eval/split；
- protected guard 拒绝；
- 结果不进入 DecisionEngine。

## 10. 端到端验收标准

系统只有满足以下条件才算“实验 Agents 第一版完成”：

1. 能从已确认任务创建 Session；
2. 能自动准备并验证环境；
3. 能运行 baseline；
4. Coordinator 能基于 baseline 连续 ideate；
5. Executor 在 worktree 修改；
6. GPU Job 可恢复、取消和超时；
7. 训练正常时不调用监控 LLM；
8. 实现生效与评估可比较有证据；
9. noise floor 参与决策；
10. 每个研究决策有 CognitiveCommit；
11. checkpoint 丢失后可恢复；
12. 能完成至少两轮结果驱动的假设更新；
13. 能区分 repair、new child idea 和 inconclusive；
14. 能检测重复、停滞和预算；
15. 所有关键操作有 artifact/event/hash；
16. protected artifact 被改时结果自动失效；
17. cheap batch 下认知成本可控；
18. 不依赖前端或最终报告 Agent 才能运行。
