## 2. ConvergenceMonitor

参考 Arbor 的 `compute_velocity()` + `find_exhausted_parents()` + 滑动窗口设计。

**窗口语义（非重叠滑动窗口）：**

`window_size=5` 指每 5 个 evaluable attempt 为一个窗口。窗口之间不重叠——第 1-5 个 attempt 是窗口 1，第 6-10 个是窗口 2。Velocity 在每个窗口结束时计算一次。

**Stop 判定（显式连续计数）：**

`window_size × stop_windows` 不天然等于 15 次无提升。真正的 stop 信号使用独立的连续计数器，而非滑动窗口叠加：

```python
class ConvergenceConfig(BaseModel):
    """默认值来自 Arbor 参考实现。Session 可配置。"""
    window_size: int = 5              # 每个窗口的可评估 attempt 数
    warn_windows: int = 1             # 1 个窗口 velocity=0 → warn
    paradigm_shift_windows: int = 2   # 2 个连续窗口 velocity=0 → paradigm_shift
    consecutive_no_progress_for_stop: int = 15  # 连续 15 个无提升 → stop
    noise_units_for_progress: float = 1.0
```

分工：

| 信号 | 来源 | 用途 |
|---|---|---|
| window velocity (5-attempt 非重叠固定窗口) | `compute_velocity()` | WARN / PARADIGM_SHIFT 分析 |
| parent exhaustion | `find_exhausted_parents()` | axis 级别停滞诊断 |
| consecutive_no_progress | 独立连续计数器 | **Stop 决策** |

```text
velocity         — 窗口内 champion IMPROVEMENT 次数 / window_size
parent_exhaustion — 已无有效子节点的 research axis IDs
warn             — 最近 1 个窗口 velocity=0
paradigm_shift   — 最近 2 个连续窗口 velocity=0
stop             — 连续 consecutive_no_progress_for_stop 个 evaluable attempt 无 IMPROVEMENT
```

**只统计 `SCIENTIFICALLY_EVALUABLE` + `attempt_purpose == exploration` 的 attempt。** 不统计 baseline、noise_calibration、confirmation 和 repair attempt。

### 3.2 本轮实现：Step Signature（参考 MiMo 算法，自行移植）

> 参考 MiMo `stableStringify()` 和 `stepSignature()` 的算法语义（`[PORT-PENDING-LICENSE]`——mimo-code 仓库许可证确认后方可落地）。

```python
REPEATED_STEP_THRESHOLD = 3  # 连续 3 次相同 → 触发

def stable_stringify(tool_name: str, args: dict) -> str:
    return json.dumps({"tool": tool_name, "args": args}, sort_keys=True)

def step_signature(tool_calls: list[ToolCall]) -> str:
    segments = []
    for call in tool_calls:
        segments.append(f"tool:{call.tool_name}:{stable_stringify(call.args)}")
    return "\n".join(segments) if segments else ""
```

**适用：** Coordinator 用同一 tool+参数连续调用 3 次；Executor 用同一 SEARCH/REPLACE block 连续提交 3 次。

### 3.3 本轮额外实现：AutoAD 特定检测

| 检测 | 触发条件 |
|------|----------|
| repeated same patch | patch.diff SHA256 连续 2 次未变 |
| repeated same failing command | subprocess exit_code 相同且 stderr 内容相同 |

不实现 monologue detection、alternating pattern detection 和 context window error detection。

### 3.4 延后：OpenHands 5-Mode StuckDetector（`[ADAPT-LATER]`）

OpenHands `stuck_detector.py`（320 行）的算法有参考价值，但依赖 OpenHands SDK 的事件类型系统（`ActionEvent`、`ObservationEvent`、`AgentErrorEvent` 等），不能直接复制。

完整 5-mode 检测延后到主闭环完成后适配。

| Mode | 检测什么 | 适配前提 |
|------|----------|---------|
| 1. Repeating Action-Observation | 同一 tool + 同一返回内容重复 N 次 | 需定义 AutoAD 通用 Event 类型 |
| 2. Repeating Action-Error | 同一 tool + 错误重复 N 次 | 同上 |
| 3. Monologue | 连续 N 条消息无 tool call | 需 Agent 框架支持消息流事件 |
| 4. Alternating ABAB | A→B→A→B 交替模式 | 需完整 Agent turn 记录 |
| 5. Context Window Error | 重复 context overflow | DeepAgents 框架抽象后 |

## 5. StrategySelector + SkillsMiddleware

**FINAL DISPOSITION: StrategyPolicy/StrategyOverlay 已删除。** 

ConvergenceAlert
→ StrategySelector.filter_and_rank()（只过滤排序，不决定）
→ eligible_skill_candidates
→ Coordinator 根据 alert + 科研历史选择
→ SkillsMiddleware 加载

StrategySelector 只负责：

```text
skill 是否存在
是否适用当前任务类型
是否违反权限或科研契约
是否已被禁用
是否超预算
是否已连续重复使用
```

它不能根据几个固定数值决定科研策略。

StrategySelector 的确定性过滤：

```text
- 当前激活 skill
- 适用范围（axis/subtree/global）
- 生效周期（attempt/session）
- 审批状态
- 是否影响安全约束
```

不再实现 prompt overlay 管理器。

参考：DeepAgents `SkillsMiddleware` + `SKILL.md` 渐进加载。

## 7. 全链路测试基准

建立小型 `AD-AgentBench` fixture，不追求模型 SOTA，测试闭环能力。

### Case 1：明确有效的小参数优化

- baseline 可运行；
- 一个参数变化提升超过 noise；
- 预期：candidate→confirm→champion。

### Case 2：实现未生效

- patch 成功；
- 实际路径未触发；
- 预期：smoke_passed=False → bounded repair，不 prune idea。

### Case 3：实现有效但指标下降

- patch_applied + smoke_passed + metrics_parsed + protocol_intact；
- 预期：NOT_SUPPORTED；
- 写 failure-WHY；
- 派生或 pivot。

### Case 4：结果在噪声内

- 预期：补 seed 或 NO_EFFECT；
- 不提升 champion。

### Case 5：GPU OOM

- PostRunFailureClassifier 分类；
- 无 LLM 实时监控；
- bounded repair/retry。

### Case 6：训练 hang

- stale heartbeat；
- TERM/KILL；
- Job 有明确状态；
- ResourceLease 释放。

### Case 7：Coordinator 重启

- 删除 checkpoint；
- 从 Session/Tree/Commit 恢复；
- 不重复已完成 Job。

### Case 8：便宜实验批次

- 一次生成 3 variants；
- 3 个 Result Integration；
- 批次结束后一次 Compact Cycle。

### Case 9：停滞

- 重复 idea / low KEEP；
- ConvergenceMonitor → Coordinator → StrategySelector.filter_and_rank → Skills；
- skill activation；
- 新方向出现；
- skill selection event 有审计。

### Case 10：评估作弊

- 修改 eval/split；
- protected guard 拒绝；
- 结果不进入 DecisionEngine。

## 10. 端到端验收标准

系统只有满足以下条件才算“实验 Agents 第一版完成”：

1. 能从已确认任务创建 Session；
2. 能自动准备并验证环境；
3. 能运行 baseline；
4. Coordinator 能基于 baseline 连续 ideate；
5. Executor 在 worktree 修改；
6. GPU Job 可恢复、取消和超时；
7. 训练正常时不调用监控 LLM；
8. 实现生效与评估可比较有证据；
9. noise floor 参与决策；
10. 每个研究决策有 CognitiveCommit；
11. checkpoint 丢失后可恢复；
12. 能完成至少两轮结果驱动的假设更新；
13. 能区分 repair、new child idea 和 inconclusive；
14. 能检测重复、停滞和预算；
15. 所有关键操作有 artifact/event/hash；
16. protected artifact 被改时结果自动失效；
17. cheap batch 下认知成本可控；
18. 不依赖前端或最终报告 Agent 才能运行。
