# 02. 后端只读投影

## 1. 目标

新增一个前端专用的只读投影装配器，将现有 Session、IdeaTree、Attempt、OutcomeCard、CognitiveCommit、科学评价和 Champion Artifact 组织成一次返回的 `ExperimentProjection`。

投影不是新的权威 Store，也不是事件回放器。它每次请求都从当前权威文件读取，构造短生命周期的响应对象。

本文件中的“只读”仅约束投影 `GET` 路径。工作台另有已经明确限定的审批 `POST` 路径时，仍必须由服务端从已冻结工件推导和校验输入；它不改变 GET 的只读性质，也不把控制面状态复制到前端。

## 2. 必须遵守的事实边界

- 当前 `ExperimentSessionStore` 有准确 `session_id` 的 `load()`，没有 `list_sessions()`。
- `ExperimentStarter` 当前固定写入 `task_ref="input_task.yaml"`；这个字段是文件引用，不是用户可读的研究目标。
- `IdeaTreeStore` 的 `experiment.idea_tree.mutated` 事件只保证有 mutation 名称和 `tree_revision`。批量变更可能记录为 `apply_mutations`，不能从事件反推某个历史节点。
- `OutcomeCard` 是执行与协议事实的权威来源。
- `ScientificAssessment` 是科学比较结论的权威来源。
- `AssessmentReconciliation` 记录两者的权威边界。
- `ScientificAssessmentService.assess()` 和 `effective_assessment()` 可能读取并写入评价 Artifact，不能从 GET 投影路径调用。
- `CandidateRegistry.current_summary_for_session()` 返回按 `evaluation_contract_hash` 分组的字典，不是一个固定 Champion。
- `CandidateSnapshot` 没有 `primary_delta` 和 `scientific_effect`；这些字段必须从已有科学评价 Artifact 读取。

## 4. 投影 Schema

以下是本功能新增的 API 设计字段，不是现有 Store 字段。编码前仍需用当前源码和测试核对已有模型字段，禁止凭相似名称猜测。

所有模型使用 Pydantic v2：

```python
from typing import Literal
from pydantic import BaseModel, ConfigDict, Field


class ExperimentProjection(BaseModel):
    model_config = ConfigDict(extra="forbid")

    schema_version: Literal[1] = 1
    selection_status: Literal["no_session", "selected", "ambiguous"]
    session: SessionProjection | None = None
    session_candidates: list[SessionSummary] = Field(default_factory=list)
    input_task: InputTask | None = None
    summary: SessionStats | None = None
    idea_tree: IdeaTreeProjection | None = None
    attempts: list[AttemptProjection] = Field(default_factory=list)
    candidates: list[CandidateProjection] = Field(default_factory=list)
    candidate_inventory_status: Literal["available", "invalid"] = "available"
    actions: ExperimentActionsProjection = Field(default_factory=ExperimentActionsProjection)
    cognitive_commits: list[CognitiveCommit] = Field(default_factory=list)
    champion_status: Literal["absent", "available", "assessment_missing", "assessment_invalid", "control_plane_invalid"] = "absent"
    champion: ChampionProjection | None = None
    activity: list[ActivityCard] = Field(default_factory=list)
    activity_limit: int = 100
    activity_truncated: bool = False
    activity_scan_truncated: bool = False
    developer_refs: DeveloperRefs | None = None
```

这样可以同时表达：

- 没有 Session 时没有 `summary`；
- 多个 Session 尚未选择时不伪造 `summary`；
- 只有选中 Session 后才装配 Idea、Attempt、Champion 和 Activity。

### 4.1 入口函数

```python
def build_projection(
    run_dir: Path,
    session_id: str | None = None,
) -> ExperimentProjection:
    """Build a read-only projection from current durable artifacts."""
```

处理规则：

1. `session_id is None` 且没有 Session 文件：返回 `selection_status="no_session"`；
2. `session_id is None` 且恰好一个 Session 文件：加载该 Session；
3. `session_id is None` 且有多个 Session 文件：返回 `selection_status="ambiguous"` 和候选摘要；
4. 指定 `session_id`：只加载准确的该 Session，不回退到其他 Session；
5. Session 文件损坏：返回明确的服务错误，不静默换用另一个文件。

Session 文件发现只允许在装配器内部读取当前约定目录 `experiments/sessions/`。不要新增 `ExperimentSessionStore.list_sessions()`，也不要使用修改时间或目录顺序做选择。

外部传入的 `session_id` 必须先与只读发现得到的候选文件 stem 做精确相等匹配，再交给 `ExperimentSessionStore.load()`。候选集合为空或不包含该值时返回 404；不能先把未经校验的字符串拼接进 Session 路径。这样拒绝 `/`、`..`、反斜杠、编码后路径逃逸和任意文件名，同时不把当前生成格式硬编码成新的 API 规则。

### 4.2 `SessionProjection`

字段必须来自当前 `ExperimentSession` 模型：

```python
class SessionProjection(BaseModel):
    model_config = ConfigDict(extra="forbid")

    session_id: str
    task_ref: str
    task_hash: str
    status: str
    execution_mode: str
    readiness_status: str
    readiness_blockers: list[str] = Field(default_factory=list)
    environment_status: str
    baseline_status: str
    evaluation_contract_ref: str | None = None
    evaluation_contract_sha256: str | None = None
    budget: dict[str, object] = Field(default_factory=dict)
    created_at: str
    updated_at: str
```

`task_ref` 仅供开发者引用。用户可读的目标由投影顶层的 `input_task` 提供。

### 4.3 `SessionSummary`

候选摘要只放稳定、可验证的 Session 基础信息，例如 `session_id`、`task_hash`、`status`、`created_at`。
不要把 `task_ref` 当作研究目标，也不要把候选摘要扩展成完整 Session。

### 4.4 `InputTask` 读取

使用现有 `InputTask` 模型解析 `session.task_ref` 指向的文件。读取前必须验证：

- 路径是相对于 `run_dir` 的相对路径；
- 路径不是绝对路径；
- 路径不包含 `..`；
- 文件存在；
- 使用当前仓库已有 YAML 读取和 `InputTask.model_validate` 习惯。

前端目标显示规则：

```text
主目标：InputTask.user_idea，如果为空则使用 InputTask.request
辅助信息：baseline、dataset、primary_metrics、constraints
```

解析失败不能把 `input_task.yaml` 当成用户目标；应保留安全错误或空值，并在开发者详情中保留 `task_ref`。

### 4.5 `SessionStats`

```python

    idea_rooted_count: int
```

来源：

- Session：`ExperimentSessionStore.load()`；
- Idea：`IdeaTreeStore.load()`；
- Attempt：`ExperimentAttemptStore.list_for_session()`；

不要把 `CandidateRegistry.current_summary_for_session()` 的完整字典直接冒充 `champion_summary`。

### 4.7 `AttemptProjection`

投影将执行、科学评价和关联信息分层：

```python
class AttemptProjection(BaseModel):
    model_config = ConfigDict(extra="forbid")

    attempt_id: str
    attempt_purpose: str
    runtime_status: str
    job_type: str
    pipeline_job_id: str | None = None
    required_device_count: int
    required_vram_mb: int
    retry_of: str | None = None
    retry_count: int
    max_retries: int
    retry_exhausted: bool
    failure_code: str | None = None
    command_plan_summary: str
    execution_outcome: OutcomeCard | None = None
    scientific_assessment: ScientificAssessment | None = None
    assessment_reconciliation: AssessmentReconciliation | None = None
    scientific_assessment_status: Literal["available", "not_materialized", "invalid"]
    related_idea_ids: list[str] = Field(default_factory=list)
    pid: int | None = None
    heartbeat_at: str | None = None
    resource_lease_id: str | None = None
    created_at: str
    updated_at: str
```

读取规则：

- `execution_outcome` 读取 `attempts/{attempt_id}/outcome_card.json`；
- `scientific_assessment` 只读取已经存在的 `scientific_assessment.json`；
- `assessment_reconciliation` 只读取已经存在的 `assessment_reconciliation.json`；
- Assessment、Reconciliation、OutcomeCard 和评价输入必须属于同一 Attempt，引用必须落在该 Attempt 目录且 SHA-256 与当前文件一致；
- 仅缺少整套评价侧边车时保留执行结果并设置 `not_materialized`；缺少其中一侧、引用或哈希不一致时设置 `invalid`；
- 任何 GET 路径不得调用会写入这些文件的 Service 方法。

如果 Artifact 存在但校验失败，设置 `invalid` 并保留安全错误信息；不能把无效评价降级成科学结论。

`CandidateProjection` 只包含工作台审批区需要的耐久事实：`candidate_id`、`idea_id`、`attempt_id`、`b_test_passed` 和 `guardrails_passed`。候选清单读取失败时，`candidate_inventory_status="invalid"`，不把损坏清单伪装为空清单，也不从浏览器补全 Candidate 事实。

`actions.candidate_confirmations` 和 `actions.candidate_promotions` 是服务端从当前投影事实生成的受限入口。前端只渲染这些目标；噪声阈值提交后，B_test 服务仍会重新执行完整门禁。

```python
class CandidateProjection(BaseModel):
    model_config = ConfigDict(extra="forbid")

    candidate_id: str
    idea_id: str
    attempt_id: str
    b_test_passed: bool
    guardrails_passed: bool
```

### 4.8 `ChampionProjection`

当前 Champion 的选择必须是确定的：

```text
ExperimentSession.evaluation_contract_sha256
        ↓
CandidateRegistry.current_by_contract()[contract_hash]
        ↓
ChampionPointer + CandidateSnapshot
        ↓
CandidateSnapshot.attempt_id
        ↓
已有科学评价 Artifact
```

建议将 Champion 的控制面登记状态与科学评价详情状态分开表达：

```python
class ChampionProjection(BaseModel):
    model_config = ConfigDict(extra="forbid")

    candidate_id: str
    session_id: str
    evaluation_contract_hash: str
    idea_id: str
    attempt_id: str
    scientific_assessment: ScientificAssessment | None = None
    assessment_reconciliation: AssessmentReconciliation | None = None
    assessment_error: str | None = None
```

`ExperimentProjection.champion_status` 和 `SessionStats.champion_status` 使用同一组值：

```text
absent              没有当前 ChampionPointer / Candidate
available           Champion 已登记且科学评价详情有效
assessment_missing  Champion 已登记，但科学评价详情缺失
assessment_invalid  Champion 已登记，但科学评价详情无效
control_plane_invalid  Champion 指针或 Candidate 控制面记录无效
```

因此 Pointer 和 Candidate 已存在但评价 Artifact 缺失或无效时，仍返回 `champion` 的身份和引用，只将其评价字段置空并设置退化状态。只有以下控制面事实不存在时才返回 `champion=None` / `champion_status="absent"`：

- Session 没有绑定评价合同；
- 当前合同没有对应 ChampionPointer；
以下控制面事实无效时返回 `champion_status="control_plane_invalid"`：

- Candidate 清单损坏；
- ChampionPointer 无法校验；
- Pointer 指向的 Candidate 不属于当前 Session 或当前评价合同；
- Pointer 指向的 Candidate 文件不存在或无效。

以上情况返回 `champion_status="control_plane_invalid"`；不能据此声称当前没有 Champion。

`primary_delta`、`scientific_effect` 不能从 `CandidateSnapshot` 直接读取，必须来自当前 Candidate 对应的科学评价文件。

### 4.9 `ActivityCard`

```python
class ActivityCard(BaseModel):
    model_config = ConfigDict(extra="forbid")

    event_id: int
    event_type: str
    created_at: str
    title: str
    summary: str
    card_kind: str
    related_idea_id: str | None = None
    related_attempt_id: str | None = None
    related_commit_id: str | None = None
    related_outcome: dict[str, object] | None = None
    detail: str = ""
    evidence_refs: list[str] = Field(default_factory=list)
```

首版 Activity 返回数量固定受限：装配并过滤出当前 Session 的可靠卡片后，按 `event_id` 从新到旧只返回最近 100 条。响应中的 `activity_limit=100` 表示该上限；可靠卡片超过上限时设置 `activity_truncated=true`，否则为 `false`。首版不增加分页、游标或新索引，也不把完整长期事件日志返回给浏览器。

Candidate 生命周期的现有事件 `experiment.candidate.b_test_queued` 和 `experiment.candidate.registered` 映射为可靠 Activity；前者关联确认 Attempt，后者关联 Candidate 的 B_dev Attempt。

为限制与当前 Session 无关事件的扫描成本，扫描达到当前上限但尚未找到足够可靠卡片时设置 `activity_scan_truncated=true`。这与 `activity_truncated` 不同：前者表示较早事件尚未扫描，后者表示可靠卡片超过返回上限；两者都为 `false` 时才表示已完成当前读取范围。

装配使用确定性映射，不调用 LLM。

允许的最小映射：

| 事件 | 展示 |
|---|---|
| `experiment.session.created` | 实验 Session 已创建 |
| `experiment.idea_tree.created` | Idea 树已初始化 |
| `experiment.idea_tree.mutated` | Idea Tree 已更新；显示 `tree_revision` |
| `experiment.attempt.created` | 实验已创建 |
| `experiment.attempt.queued` | 实验已排队 |
| `experiment.attempt.running` | 实验开始运行 |
| `experiment.attempt.finalized` | 实验已完成；结果按现有 Artifact 分层显示 |
| `experiment.attempt.retry_queued` | 重试已排队 |
| `experiment.cognitive_commit.appended` | 认知提交已记录 |
| `experiment.coordinator.*` | 对应周期或恢复事件的通用说明 |
| `experiment.champion.*` | Champion 变更通知，详情从当前 Registry 读取 |

`experiment.idea_tree.mutated` 不得生成“新增了哪个 Idea”“哪个节点被剪枝”“哪个节点变成 SUPPORTED”等历史断言。

事件若没有足够信息可靠关联当前 Session，则不要靠字段名猜测；应使用已存在的 `attempt_id`、`commit_id` 等引用查询权威 Store，无法确认时忽略或仅放开发者详情。

未知事件类型：研究者视图忽略，开发者详情可保留原始事件 JSON；不能为未知事件编造卡片语义。

## 5. API 路由

新建 `src/autoad_researcher/server/routes/experiment_projection.py`，并在 `server/main.py` 注册：

```text
GET /api/runs/{run_id}/experiment/projection?session_id={session_id}
```

路由沿用当前 `runs.py` 的 run 目录解析方式：

| 情况 | HTTP |
|---|---:|
| run 不存在 | 404 |
| 指定 Session 不存在 | 404 |
| 无 Session | 200，`selection_status="no_session"` |
| 多 Session 且未选择 | 200，`selection_status="ambiguous"` |
| 正常选中 | 200，完整 `ExperimentProjection` |

路由只读取文件，不创建目录、不写 Artifact、不更新 Store、不追加事件。`session_id` 参数必须经过候选文件精确匹配后才能读取；不能直接作为文件名或路径片段使用。

| 组件 | 当前已确认的读取入口 |

| `ExperimentSessionStore` | `load()` |
| `IdeaTreeStore` | `load()` |

| `CognitiveCommitStore` | `load()` |

## 7. 测试

`tests/test_v2_experiment_projection.py` 至少覆盖：

1. Pydantic Schema 可实例化，默认字段无 dataclass 顺序错误；
2. 没有 Session；
3. 一个 Session 自动选中；
4. 多个 Session 返回 `ambiguous`，不自动选中；
5. 指定错误 Session 返回 404；
6. 从 `input_task.yaml` 显示真实研究目标；
7. Session 尚未准备环境；
8. Attempt queued / running / completed；
9. Attempt failed / timed_out / lost；
10. OutcomeCard 缺少 metrics；
11. 科学评价 Artifact 缺失时不写文件；
12. 科学评价和 reconciliation 已存在时正确读取；
13. IdeaTree 含各个现有状态；
14. Idea mutation 只生成 revision 更新卡片；
15. 剪枝原因来自节点 `insights`，不来自 mutation receipt 推断；
16. CognitiveCommit 和 CognitiveBudget 读取；
17. Session 未绑定评价合同；
18. 多合同 Champion 只选择当前 `evaluation_contract_sha256`；
19. Champion Pointer/Candidate 存在但科学评价缺失时保留 Champion 身份并返回 `assessment_missing`；
20. Champion 科学评价无效时保留 Champion 身份并返回 `assessment_invalid`；
21. 无法精确匹配的 `session_id`（包括路径逃逸样式）返回 404 且不会读取候选目录外文件；
22. Activity 默认最多返回最近 100 条，并正确设置 `activity_truncated`；
23. 未知 event type；
24. 评价引用与 SHA-256 绑定正确时为 `available`，错链、跨 Attempt、侧边车不完整时为 `invalid`；
25. 服务端审批动作投影只在确定的 Session/Candidate 事实存在时返回；
26. Candidate B_test 排队和登记事件进入 Activity；
27. 投影接口前后文件集合、内容和修改时间不变。

## 8. 验收

- API 返回 Pydantic 校验通过的 JSON。
- 无 Session、ambiguous、正常 Session 三类响应可以区分。
- 研究目标不显示 `input_task.yaml`。
- Attempt 执行事实和科学评价结论不混淆。
