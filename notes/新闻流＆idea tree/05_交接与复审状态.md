## 2. 已确认的架构边界

工作台不是纯展示页面，也不是浏览器侧实验控制器。当前正确边界为：

```text
只读权威投影
  -
有限、类型化、显式且可审计的审批动作
```

`GET /api/runs/{run_id}/experiment/projection` 只从 Session、IdeaTree、Attempt、OutcomeCard、科学评价、Candidate 与事件工件装配投影；不得物化评价文件、不得从 WebSocket payload 拼接科研状态、不得在前端维护第二套实验状态。

已保留的两个 `POST` 动作不应删除：

| 动作 | 浏览器可提交的事实 | 服务端必须重新验证/推导的事实 |
|---|---|---|
| B_test 确认 | `candidate_attempt_id`、`noise_threshold`、`idempotency_key` | Session、Attempt、科学评价、冻结合同、工作区、protected artifacts 与执行输入 |
| Champion 推广 | `candidate_id`、`approved_by` | 仓库、分支、主 checkout、合并目标和持久化候选工件 |

页面不接受任意命令、仓库路径、补丁哈希或合并目标。审批区只在 `approve_each_step` Session 和服务端已投影出的待确认事实存在时出现；操作成功后仅重新读取投影。

这一设计与本地 Arbor 的“持久化状态到只读 snapshot、原始事件与 UI 分离”思路一致；没有引入第二个状态机或新的 Agent/执行框架。

## 3. 已功能性验收的内容

1. `activity=[]` 且 `activity_scan_truncated=true` 时明确提示较早动态尚未完成扫描，不再伪装成“暂无实验动态”；对应 Playwright 回归已存在。
2. 状态显示按领域拆分为 Session、Environment、Baseline、Idea、Attempt、Execution、Evaluation、Scientific effect、Attempt purpose 和 Cost。`READY`、`INCONCLUSIVE` 不跨领域复用语义，未知值保留原始值。
3. Session 概览显示主指标、约束、预算、已消耗预算、failed/lost Attempt、readiness blockers、baseline、dataset 和领域化状态。
4. Idea 详情显示 grounding、最小干预、关系、证据、Attempt/CognitiveCommit 关联、父子节点与 insights；Attempt 详情显示 job type、retry、资源请求、lease、heartbeat、指标解析、guardrail、patch、smoke 与评价输入；Activity 详情显示事件类型及已有关联引用。
5. 选中 Idea 后，Attempt 列表只保留 `related_idea_ids` 精确包含该 Idea 的 Attempt；无关联时显示明确空状态。浏览器测试同时覆盖无关 Attempt 不出现。
6. `ExperimentPage` 保持挂载，`SettingsPage` 使用覆盖层。保存会关闭覆盖层并保留 Session、详情选择、Idea 展开、开发者详情和已加载投影；浏览器测试已覆盖保存后已选 Idea 仍存在。
7. Idea Tree 对 `READY` 和 `MERGED` 分别显示等待与合并标记。
8. 页面与计划文本明确为“只读观测 + 受限显式审批动作”，未扩大审批输入边界。

## 4. 待闭环事项

### P1 — 后端投影测试矩阵

`02_后端只读投影.md` 约定的 `tests/test_v2_experiment_projection.py` 测试矩阵尚未逐项闭环。当前专用测试已覆盖无/多 Session、InputTask、路径逃逸、坏 Session、坏 Champion pointer、坏 Candidate inventory、Activity 扫描、缺失评价不物化、Candidate facts、100 条 Activity 和路由错误等关键退化路径，但不足以证明下列合同均被测试：

- Attempt 的完整状态矩阵；
- OutcomeCard 缺 metrics；
- 有效 ScientificAssessment 与 AssessmentReconciliation；
- IdeaTree 全状态与 insights 来源；
- CognitiveCommit 与 CognitiveBudget；
- 多评价合同 Champion；
- `assessment_missing`、`assessment_invalid`；
- 未知事件；
- GET 前后相关文件内容与 mtime 完全不变。

处理方式：先以当前投影 Schema 和测试 fixture 为准，逐项补测试或在计划中明确归档为已由其他测试覆盖；不要为了凑数量引入虚构状态、前端状态机或新持久化层。测试应继续遵守纯只读 GET 边界。

### P1 — `02_后端只读投影.md` 与实际 Schema 同步

计划中的顶层 `ExperimentProjection` 示例仍缺少以下实际字段：

```text
candidates
candidate_inventory_status
activity_scan_truncated
```

实际 `ChampionStatus` 还包含：

```text
control_plane_invalid
```

文档中 Pointer/Candidate 无效时回退 `absent` 的旧描述也与当前实现不符：控制面记录无效时应为 `control_plane_invalid`，不能据此声称不存在 Champion。下次修订必须同时更新 `ExperimentProjection`、`SessionStats` 和 Champion 选择/退化语义，避免前端类型和故障展示重新偏离。

### P2 — 普通 Attempt 详情的科学评价工件引用

详情已显示 `outcome_card_ref` 与 `inputs_ref`，但未显示 `AssessmentReconciliation.scientific_assessment_ref`。计划要求“评价 Artifact 引用”，建议在普通详情的权威边界区域以用户可理解的引用信息展示；原始路径、hash 与其他开发者细节仍应遵循现有隐藏/展开边界。
