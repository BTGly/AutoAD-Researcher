# 开发计划 R8：转交、版本与审阅闭环（对应 PR-R5）

## 1. 目标

把报告讨论结果转换为用户可确认的 Proposal，把用户审阅决定记录为独立 ReviewDecision，并通过现有 Experiment/Task 控制面完成后续转交。

## 2. 参考依据

| 来源 | 可复用机制 | 本项目限制 |
|---|---|---|
| Claw-AI-Lab | PIVOT/REFINE rollback 前版本化、decision history | AutoAD 不复制其阶段树；只吸收 lineage 思想 |
| Arbor | 用户显式 Ask/Steer 边界 | 报告侧只保留只读讨论和显式确认 |
| AutoAD `ExperimentAttemptService` | confirmatory Attempt、retry lineage、幂等 Job | 直接作为轻量后续实验入口 |
| AutoAD `TaskBridge` | 待确认任务草案和合同确认 | 仅用于新任务/新 Session；受 `input_task.yaml` 单次物化约束 |
| ARIS | 执行完成与接受分离 | ReviewDecision 单独持久化 |

## 3. 数据模型

### 3.1 `FollowUpProposal`

计划新增的 Proposal 至少包含：

```text
proposal_id
source_report_id
source_report_version
source_snapshot_content_sha256
proposal_type
rationale
evidence_refs: list[ArtifactReferenceV2]
requested_changes
required_experiments
estimated_budget
unresolved_questions
risk_level
status
created_at
```

Proposal 类型：

```text
ADD_CONFIRMATION
RETRY_FAILED
REFINE_CURRENT
PIVOT
REQUEST_HUMAN
```

不包含 `END_AND_ACCEPT`。接受结论是 ReviewDecision。

Proposal 状态：

```text
DRAFT
READY_FOR_CONFIRMATION
CONFIRMED
REJECTED
SUPERSEDED
HANDED_OFF
```

所有 evidence、合同和预算引用必须使用当前仓库中已存在的类型化引用或精确的 ref+sha 组合；不要使用裸 `dict`。

### 3.2 `ReviewDecision`

```text
decision_id
request_id
report_id
version
source_snapshot_content_sha256
decision: accept / suspend / needs_more / needs_repair / needs_pivot / disputed
user_comment
accepted_claims
disputed_claims
requested_follow_up
created_at
```

ReviewDecision 只记录用户审阅，不创建实验 Job。`accept` 不能被表示成 Proposal，也不能把报告生成完成误写成用户接受。

ReviewDecision 不是单个可覆盖 JSON，而是 append-only 决定序列；当前 `review_status` 由最后一条有效决定确定性投影。`accepted_claims` 和 `disputed_claims` 只引用 Narrative 的稳定 `claim_id`，不保存无法校验的自由文本。

## 4. Proposal 校验

确定性校验至少包括：

```text
source report 存在且 hash 匹配
evidence_refs 已登记且 SHA 匹配
proposal_type 与 requested_changes 一致
预算不超过当前可用预算
RETRY_FAILED 绑定原 Attempt
ADD_CONFIRMATION 不修改 EvaluationContract
REFINE_CURRENT 保持合同、主指标和比较协议不变，并要求 Idea/Intervention/patch evidence
PIVOT 需要新 Run、新任务和新 Session 的合同输入
```

Proposal 不通过时保留错误列表，不交给 Worker。

## 5. Handoff 映射

| Proposal | 首版实现 |
|---|---|
| `ADD_CONFIRMATION` | 通过现有 `ExperimentAttemptService` 创建 confirmatory Attempt/Job，绑定 proposal_id |
| `RETRY_FAILED` | 使用现有 retry 创建逻辑，保存 `retry_of` 和 proposal lineage |
| `REFINE_CURRENT` | 先经过 Coordinator/IdeaTree 创建 Idea 或 child、生成 InterventionContract，再由 Executor 产生 patch/worktree/实现证据，最后调用 AttemptService |
| `PIVOT` | 创建新的 Run、新任务和新 Session，再调用 `TaskBridge`；记录 parent report/session |
| `REQUEST_HUMAN` | 只写待人工处理状态，不创建 Job |

当前没有正式 `Iteration` 领域模型，因此首版不承诺“同一 Session 新建 Iteration”。`ADD_CONFIRMATION` 和满足闭环的 `REFINE_CURRENT` 才能在同一 Session 下形成新的 Attempt；方向变化进入新的 Run、任务和 Session。

当前 `TaskBridge.build_experiment_task()` 在 `input_task.yaml` 已存在时会拒绝创建新任务，所以 PIVOT 必须进入新 Run，不能在同一已物化任务中强行复用。实现前必须通过实际测试确认新 Run、任务上下文和 Session 的创建边界。

## 6. 用户确认流程

```text
Agent 生成 Proposal
→ ProposalService.validate
→ READY_FOR_CONFIRMATION
→ 前端展示预算、证据、改动和风险
→ 用户修改 / 拒绝 / 确认

拒绝 → REJECTED，不创建 Job
确认 → CONFIRMED
      → 确定性 handoff service
      → 创建 Attempt/Task/Session
      → HANDED_OFF
```

确认 API 必须幂等：重复确认返回已有 handoff 结果，不能重复创建 Attempt 或 Job。

## 7. 报告版本与 lineage

```text
Report v001（不可变）
└── Proposal #1
    └── 用户确认
        └── 新 Attempt 或新 Session
            └── 新 Snapshot
                └── Report v002（新目录）
```

新报告必须拥有新的 Snapshot identity 和 report_id。`previous_report_id`、`parent_report_id`、proposal_id 和 parent session ref 用于追踪 lineage；旧报告的 Facts、Markdown、Evidence 和 validation 文件不得覆盖。

## 8. 审阅结束

```text
READY_FOR_REVIEW ≠ ACCEPTED
```

只有用户提交 `ReviewDecision(decision="accept")` 后，才可把审阅状态投影为 accepted，并在需要时更新 Session 的审阅字段。不得使用当前没有定义的 Session 状态或 `ARCHIVED` 代替该事实。

审阅 API 只追加新的 ReviewDecision，并从序列投影当前状态；不能通过更新单个 current review JSON 丢失历史。

## 9. 验收

- [ ] 未确认 Proposal 不创建 Job、Attempt 或 Session。
- [ ] 重复确认幂等。
- [ ] 用户拒绝后不得 handoff。
- [ ] RETRY_FAILED 绑定正确的原 Attempt。
- [ ] ADD_CONFIRMATION 不修改 EvaluationContract。
- [ ] `REFINE_CURRENT` 在产生 Idea/Intervention/patch/实现 evidence 前不能创建 Attempt。
- [ ] PIVOT 不覆盖旧合同，进入新任务/Session lineage。
- [ ] PIVOT 进入新 Run，不能在已有 `input_task.yaml` 的 run 内复用。
- [ ] Report v002 不覆盖 Report v001。
- [ ] 多次 ReviewDecision 保留完整历史，并能投影 needs_more/disputed/accepted 等当前状态。
- [ ] `accepted_claims` / `disputed_claims` 中的 claim_id 都能在 Narrative 中解析。
- [ ] `accept` 只写 ReviewDecision，不创建 Proposal 或实验。
- [ ] 所有 handoff 事件可在 EventStore 中追踪。

## 10. 不做什么

- 不让 Report Agent 直接创建 Job、Attempt、Session 或修改合同。
- 不首版添加 `IterationStore`、分支树和跨 Session Proposal 平台。
- 不自动执行用户没有确认的 Proposal。
- 不删除或覆盖旧报告。
