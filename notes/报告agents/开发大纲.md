# AutoAD 实验报告生成—下载—讨论—再实验闭环开发大纲 v0.3

## 一、总体目标

把当前只读取一份 Markdown 的 `ReportPage`，升级为现有实验控制面的可信只读投影：

```text
实验控制面产生可报告条件
→ 锁内同步冻结 Snapshot
→ immutable Manifest + mutable State
→ 持久化 Report Job
→ 纯代码装配 Facts/Evidence
→ LLM 生成解释性段落
→ 确定性 Validator
→ Markdown content_ready
→ HTML / Bundle / 可选 PDF
→ 报告工作区阅读和查证
→ 只读 Discussion
→ 用户确认 Proposal
→ 现有 Attempt/Task/Session 控制面接管
```

报告模块不新建实验调度、Session、Attempt、事件总线或数据库。

## 二、核心原则

### 1. 事实先行

权威事实来自当前 AutoAD 的 `ExperimentSession`、`ExperimentAttempt`、`OutcomeCard`、`EffectiveScientificAssessment`、`IdeaTreeStore`、`CandidateRegistry`、`CognitiveCostSummary`、`ResourceUsageReport`、`StopDecision` 和 `ArtifactReferenceV2`。执行/协议事实与科学比较事实不能由报告层自行重新拼接。

### 2. 报告和格式状态分离

Markdown/HTML 可审阅不依赖 TeX/PDF。PDF 是可选 capability，失败只影响 PDF 制品。

Bundle 在打包时固定交付状态快照；v1 Bundle 不包含 PDF，PDF 作为独立制品下载，避免后续 PDF 到达导致原 Bundle hash 改变。

### 3. 版本不可覆盖

Snapshot、Facts、Evidence、Validation、Markdown 和每个版本制品不可覆盖；可变的生成状态、格式状态、审阅状态和 Proposal 状态必须原子更新并留事件。

### 4. Report Agent 只读

Report Agent 只能使用受限 typed tools，不能使用通用文件系统、shell、`execute`、写入工具或实验控制面入口。

### 5. Proposal 与 Review 分离

“接受结论”是 `ReviewDecision`；补实验、重试、refine、pivot 才是 `FollowUpProposal`。所有后续实验必须经过用户确认。

### 6. 事实数字由确定性 Renderer 填充

LLM 可以生成解释性模板和结构化 claim，但数字、状态、Attempt、Champion 和预算通过 Facts placeholder 由 Renderer 填充。这里不做逐句事实图谱，也不使用全文数字正则作为主门禁。

## 三、目标架构

```text
Session / Attempt / Outcome / EffectiveAssessment / IdeaTree / Champion / Cost / Resource / Stop
                              │
                              ▼
                    ReportRequestService
                    - readiness
                    - lock-in snapshot freeze
                    - optimistic double-read consistency check
                    - snapshot identity
                    - report recipe identity
                    - version allocation
                    - durable jobs
                              │
                              ▼
                PipelineJobStore + Worker + EventStore
                              │
              ┌───────────────┴───────────────┐
              ▼                               ▼
       SnapshotBuilder (sync)            ReportStore
              │                     manifest/state/artifacts
              ▼
       ExperimentReportFactsV1
              │
       EvidenceIndex + Digest
              │
       NarrativeSectionsV1
              │
       Validator → Markdown content_ready
              │
       HTML / Bundle / optional PDF
              │

       用户确认
       ├── confirmatory/retry/refine Attempt
       └── new Task + new Session（PIVOT）
```

## 四、阶段计划

### PR-R0A：Source Adapters + Immutable Snapshot

包含：

- Session、IdeaTree、Champion、StopDecision 的冻结适配器；
- 大型不可变 artifact 的 SHA 引用；
- Snapshot canonical 内容和 hash；
- 各来源 revision/hash 双读一致性和有限重试；
- Snapshot 修改后仍可复现的测试。

不创建异步 `report_snapshot_build` Job。

### PR-R0B：Reporting Integration Contract

包含：

- immutable `ReportManifest`、revisioned `ReportState` 和 `ReportSnapshot`；
- `ArtifactReferenceV2` 的引用约束；
- Report Job 类型、幂等键、Worker dispatch 和恢复 lease/heartbeat；
- readiness projection；
- 原子 Store、版本锁、事件；
- 旧 `/report` 兼容测试。

不调 LLM，不渲染 PDF，不改前端。

### PR-R1：Facts + Evidence

包含：

- `ExperimentReportFactsV1`；
- `OutcomeCard` + `EffectiveScientificAssessment` 权威适配；
- `IdeaTreeStore`、Champion、认知成本和计算资源适配；
- Evidence Index 和 Digest；
- partial/failure/inconclusive fixtures；
- 不可比较、失败、OOM、timeout、Champion baseline 测试。

### PR-R2：Narrative + Validation + Markdown

包含：

- 确定性标题、表格和状态块；
- `NarrativeSectionsV1`；
- StructuredClaim 和 fact placeholder；
- 段落到 claim 的绑定、声明模板和 evidence 引用；
- schema/evidence/status Validator；
- `report.md` 和失败重试。

不使用“正则提取所有数字”作为事实主门禁。

### PR-R3：API + 最小前端 + HTML

包含：

- 版本列表、latest-created、latest-content-ready、manifest、content、evidence、download；
- 当前 `frontend/src/components/ReportPage.tsx` 原地升级；
- 状态摘要、版本选择、Markdown/HTML、EvidenceCard；
- 自包含 HTML；
- HTML 下载/新窗口安全边界；
- 旧 `/report` 路由兼容。

此阶段完成后可进行真人测试“看报告、看状态、查证据、下载”。

### PR-R4：Read-only Report Discussion

包含：

- 应用层 `discussion/messages.jsonl`；
- `DiscussionTurn` 状态、锁内 append、request 幂等、pending 恢复、尾行恢复和 transcript 上限；
- 固定 `report_id` 和 snapshot hash；
- narrow typed tools；
- Discussion budget 和结构化 response；
- 工具注册、路径、大小、Evidence allow-list 测试；
- Discuss 模式。

DeepAgents 只按当前 AutoAD 锁定版本的公开 API 使用；不引用旧的内部 SQLite 路径。

### PR-R5：Proposal + Review + Handoff

包含：

- `ReviewDecision`；
- `FollowUpProposal`；
- Proposal validate、用户确认/拒绝和幂等；
- confirmatory/retry/refine 的同 Session Attempt；
- REFINE 经 Coordinator/IdeaTree/Executor/patch evidence 闭环；
- PIVOT 的新 Run、新任务、新 Session 和 lineage；
- append-only ReviewDecision 及状态投影。

### PR-R6：可选 PDF + Bundle

包含：

- TeX capability preflight；
- timeout、return code、输出文件检查；
- `report.pdf`；
- `report_bundle.zip`、evidence allow-list 和 checksums；
- 打包时的 `delivery_state_snapshot.json`；v1 Bundle 不含 PDF，PDF 单独下载；
- 确定性 ZIP 排序/timestamp、symlink 排除和 exclusion manifest；
- PDF 失败不阻断内容审阅的集成测试。

## 六、测试矩阵

### Store / Job / 生命周期

- 并发报告请求只创建一个版本和一组 Jobs；
- 同身份重放幂等，异身份冲突；
- 原子写失败不留半截 JSON；
- 非法状态转换失败；
- Worker 崩溃后可恢复；
- 旧版本字节不变。
- Session、IdeaTree、Champion 修改后旧 Snapshot 不变；
- 报告 Job 超过普通 300 秒恢复边界不重复 claim。

### Facts / Evidence

- baseline-only、NO_EFFECT、IMPROVEMENT、REGRESSION、INCONCLUSIVE；
- NON_COMPARABLE、OOM、timeout、metrics 缺失；
- 部分 seed、定性 artifact、Champion baseline；
- 未登记 Evidence、SHA 错误、路径逃逸和超大 artifact。
- OutcomeCard 与 ScientificAssessment 冲突时使用 EffectiveScientificAssessment；
- CognitiveCostSummary 与 ResourceUsageReport 分开。

### Narrative / Validator

- 不存在的 Attempt/evidence 被拒绝；
- failed Attempt 不遗漏；
- NON_COMPARABLE 不写成提升；
- baseline/champion/status 一致；
- LLM 超时和结构化输出错误可重试；
- 数字表格由 Facts 生成；自由文本 lint 不替代主校验。
- 错误数字或未解析 fact placeholder 阻断 `content_ready`。

### 安全

- `run_id`、`report_id` 和 artifact allow-list；
- `../`、绝对路径、symlink、前缀碰撞；
- Agent 无 execute、shell、write/edit；
- Evidence 不能读取未登记文件；
- HTML 不直接注入 DOM；
- 日志读取有行数、字节和结果数量上限。

### Handoff

- 未确认 Proposal 不创建 Job；
- 重复确认幂等；
- 拒绝后不得 handoff；
- retry 绑定原 Attempt；
- PIVOT 不覆盖旧合同，并在新 Run 中建立新任务/Session lineage；
- REFINE 没有 Idea/Intervention/patch evidence 时不得创建 Attempt；
- ReviewDecision 历史 append-only 并能投影当前状态；
- accept 只写 ReviewDecision。

## 七、关键验收标准

```text
1. 报告由持久化 Job 生成，不依赖请求进程存活。
2. Markdown/HTML/Facts/Evidence 属于同一 report_id 和 snapshot hash。
3. 所有关键事实可追溯到 ArtifactReferenceV2/evidence_id。
4. 失败和不可比较实验不会被隐藏。
5. PDF 失败不阻断 Markdown/HTML 审阅。
6. Report Agent 不能写文件、运行命令或创建实验。
7. Discussion transcript 独立且不污染主 Chat。
8. Proposal 必须用户确认后才进入现有控制面。
9. ReviewDecision 与 FollowUpProposal 不混用。
10. 新报告不会覆盖旧报告和旧结论。
```

## 八、明确暂缓

暂不优先：

```text
通用报告数据库
同 Session Iteration 平台
复杂版本分支树
拖拽式 Report Workspace
PDF 内嵌浏览器
自动 LaTeX 修复
多 Agent 报告分析编排
自动执行 Follow-up
```

这些能力只有在核心闭环和实际使用暴露出需求后再增加。
