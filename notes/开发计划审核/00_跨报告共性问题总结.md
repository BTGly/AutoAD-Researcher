## 🔴 P0 — 必须立即解决的问题

### Q1：DeepAgents 框架就绪状态未知

| 报告来源 | 原文 |
|----------|------|
| A-H3 | 「核心依赖但未定义来源与就绪状态」 |
| C-H.TECH-04 | 「框架核心依赖无备选方案」 |
| D-H1 | 「整系统核心但来源完全未定义」 |
| E-H2 | 「最大的技术依赖黑盒」 |

**出现报告数：** 5/5

**问题：** Coordinator、ExecutorAgent、IdeaExplorerAgent、ReflectionAgent 全部依赖 `create_deep_agent()`。框架来源（内部库/第三方/待开发）、版本、成熟度、定制能力全部未定义。无备选方案评估。

**影响：** 如果 DeepAgents 不可用，整套 Agent 系统无运行基础。

**建议行动：**
1. 确认 DeepAgents 当前状态：可用版本？API 文档？是否已有生产用例？
2. 评估备选方案：LangGraph / AutoGen / 自建 Agent loop
3. 准备 MockTaskRunner 支持离线开发（不依赖 DeepAgents 即可测试业务逻辑）
4. 若为自研框架，需出独立排期，不能视为零成本基础设施

## 🟡 P1 — 第一版前必须明确方向的问题

### Q2：AI 生成代码质量无管控

| 报告来源 | 原文 |
|----------|------|
| C-H.TECH-03 | 「AI 生成代码质量管控机制完全缺失」 |
| D-H4 | 「AI 编程助手生成 Agent 代码的质量管控机制完全缺失」 |
| E-H6 | 「AI 生成代码的质量管控机制完全缺失」 |

**出现报告数：** 4/5

**问题：** 无代码规范校验、无安全审查清单、无架构一致性检查。ExecutorAgent 本身由 AI 生成，它的职责又是修改其他代码——"AI 套娃"模式下 bug 排查链极长。

**建议行动：**
1. 在 `verify.sh` 中强制 ruff + mypy + pytest + 安全扫描
2. 建立 AI 生成代码的强制审核清单（输入校验、异常处理、权限控制、幂等性、日志）
3. 核心 Store 操作（Idea Tree mutation、ResourceLease、PreApplyPatchGate）由开发者手写主体架构，AI 仅填充辅助和测试代码
4. 在 AGENTS.md 中固化架构约束规则

### Q3：无 Agent 效果评测指标

| 报告来源 | 原文 |
|----------|------|
| C-H.TECH-05 | 「Agent 效果评测体系完全缺失」 |
| D-M3 | 「评测体系不可量化，效果无验收标准」 |
| E-L2 | 「缺少 Agent 效果退化的回归检测」 |

**出现报告数：** 4/5

**问题：** 所有验收标准聚焦于确定性功能正确性（"能不能 X"），无一条关于"X 做得好不好"。无人定义任务完成率、决策正确率、无效尝试率。

**建议行动：**
1. 定义核心 Agent 评测指标（task_success_rate、decision_quality_score、invalid_attempt_rate、token/per_cycle）
2. 建立固定测试输入 + 预期输出 + 自动化比对的评测集
3. 每次 prompt / 配置修改后单命令跑全量评测
4. 评测纳入 CI，阻止 Agent 效果退化合并

### Q4：认知成本（Token 消耗）黑盒

| 报告来源 | 原文 |
|----------|------|
| A-M4 | 「CognitiveBudget 层级关系与传递机制未定义」 |
| B-H4 | 「LLM Agent 认知成本缺乏总量估算」 |
| D-M4 | 「Token 成本模型与预算控制不可落地」 |
| E-H5 | 「认知成本黑盒，单人无法承受的 API 账单风险」 |

**出现报告数：** 5/5

**问题：** 全文档无对单轮实验 Token 消耗的估算。Compact/Exploratory Cycle 每次调用消耗多少 token、子 Agent 调用平均花费——全部空白。CognitiveBudget 的 `max_calls` / `max_cost` / `max_tokens` 设太小频繁打断，设太大成本失控。

**建议行动：**
1. 基于参考项目（AI-Scientist、Arbor、AutoSOTA）的经验，给出典型认知消耗估算表
2. CognitiveBudget 默认值建议：单 Session `max_cost = $50`，调试模式用便宜模型（Haiku）
3. 实现 Token 消耗的斜率检测（2× 基线时预警）
4. 每次 LLM 调用强制写入 `llm_usage.jsonl`，空行校验纳入 verify.sh
5. 评估哪些 Agent 可以降级使用便宜模型（ReviewerAgent、HealthDiagnosisAgent）

### Q5：排期与里程碑完全缺失

| 报告来源 | 原文 |
|----------|------|
| A-H1 | 「全文档无任何排期与里程碑日期」 |
| B-H1 | 「排期与人力配置完全缺失」 |
| C-H.TECH-07 | 「单人全链路负载远超可行范围」 |
| E-H1 | 「系统规模与单人开发能力严重不匹配」 |

**出现报告数：** 4/5

**问题：** 30+ PR、7 Agent、6 子系统、20+ 确定性组件。无任何 PR 的工期估算、无人力配置、无关键路径分析。单人全栈（后端 + GPU + Agent 调优 + 安全 + 部署）不可能按当前范围交付。

**建议行动：**
1. 每 PR 标注 T 恤尺寸工时估算（S/M/L/XL）
2. 绘制 PR 依赖 DAG，标注关键路径
3. 将 Agent 效果调优单独拆分为独立排期项（建议不低于总排期的 40%）
4. 缩窄 MVP——目标：Session → Environment → baseline → Coordinator 单次决策 → Executor → Job → 结果判断

### Q6：外部代码 License 与版本锁定

| 报告来源 | 原文 |
|----------|------|
| A-H4 | 「外部代码复用存在 License 风险」 |
| A-M5 | 「第三方仓库无版本锁定」 |
| E-H4 | 「2+ 个未开源的外部依赖构成严重断供风险」 |

**出现报告数：** 3/5

**问题：** aider (AGPL)、SWE-Together、MiMo、software-agent-sdk 的 License 未审计。引用路径为绝对路径（`/root/autodl-tmp/repos/`），环境迁移和 CI 执行时路径失效。

**建议行动：**
1. 逐项确认外部仓库 License（aider AGPL 需特别评估——是否可内联复用？还是需独立实现？）
2. 锁定各依赖的 commit SHA
3. 明确复用方式：内联复制 / submodule / pip install / 重写兼容实现

### Q7：MVP 第一版范围偏大

| 报告来源 | 原文 |
|----------|------|
| A-M2 | 「第一版边界存在模糊地带」 |
| C-H.TECH-07 (附件) | 「30 PR 对于单人保守估计 3-4 个月」 |
| D-L5 | 「第一版范围可能仍偏大」 |

**出现报告数：** 4/5

**问题：** 18 项「必须」能力 + 7 Agent + 多 seed noise floor + Exploratory Cycle + ReflectionAgent + StrategyDiagnosticAgent 对于第一版过重。

**建议行动：**

2. 推迟到第二版：Exploratory Cycle、ReflectionAgent、StrategyDiagnostic、多 seed noise floor、B_test
3. 明确单 GPU 单进程的硬边界（当前已确定），第一版不碰多 GPU / torchrun
