## 整体技术风险评级

**风险等级：高**

本计划在架构层面的设计深度和模式吸收质量较高，但从**单人开发 + 重度依赖 AI 编程助手**的落地视角审视，存在以下系统性风险：

> **核心判断：这是一份优秀的架构设计文档，但作为单人可执行的开发计划，严重低估了 AI Agent 系统的调优周期、AI 编程助手的不可靠性，以及单人维护多 Agent 系统的认知负载。**

## 高优先级风险

### H-TECH-01：Agent 记忆机制设计严重缺失（单人调试地狱风险）

**问题定位：** 全文无任何 Agent 记忆管理的完整设计。

- 短期上下文管理：Plan 02 §4 只提到 "裁剪大 tool outputs" 和 "超限时使用 SummarizationMiddleware"，但 SummarizationMiddleware 是 DeepAgents 框架的内部能力，其裁剪策略（按 token 阈值 / 按消息轮数 / 按引用权重）、裁剪后的信息损失对决策质量的影响——全部未定义。
- 长期记忆：Idea Tree 和 CognitiveCommit 是"科研状态"，不是"Agent 行为记忆"。系统没有设计 Agent 如何记住"之前的某个工具调用返回了什么"、"哪个参数组合之前试过失败了"——这些信息随着每次 cycle 的上下文裁剪而丢失。
- 记忆检索与淘汰策略：零设计。

**潜在风险（单人 + AI Agent）：**
- 单开发者在调试 Agent 行为时，会发现 Agent 反复问同样的问题、重复同样的失败操作，因为上下文裁剪丢掉了关键信息。这是 AI Agent 系统最常见的隐性问题。
- 定位一个因记忆丢失导致的错误决策，需要跟踪 10+ 轮 LLM 调用——在无记忆链路追踪的情况下，单人开发者需逐条阅读 JSONL 日志，排查效率极低。
- 非结构化裁剪策略（完全交给 DeepAgents 的 SummarizationMiddleware）导致记忆行为不可预测，调试时难以复现。

**需核对/补充：**
- 明确每种 Agent（Coordinator / Executor / ReflectionAgent）在每轮 cycle 保留/丢弃的具体信息类别，定义保留优先级规则
- 设计跨 cycle 的"失败经验缓存"——至少记住最近 3-5 次重复失败的工具调用及其结果，不被摘要裁剪
- 补充 Agent 调试辅助工具（记忆回放、上下文状态可视化）的建设计划

### H-TECH-02：Coordinator 决策逻辑完全依赖 LLM 黑盒，无确定性保底

**问题定位：** Plan 02 §3.4–3.5 Compact / Exploratory 两级循环。

- Compact Cycle 的描述是"一次结构化 LLM 调用，完成 compare + reflect + hypothesis update + ideate + next action"。这相当于把一整套科研推理任务打包给一次 LLM 调用，未定义：
  - 输出 schema 的验证规则（如果 LLM 输出了不完整的 CycleDecision 怎么办？）
  - 前后决策的一致性校验（第二轮的 decision 是否与第一轮的 CognitiveCommit 矛盾？）
  - 当 LLM 连续输出低质量决策时的降级策略
  - 多步推理的 Chain-of-Thought 质量如何保证

**潜在风险（AI Agent 特有）：**
- LLM 在科研决策场景下极易产生"幻觉式自信"——输出看起来结构完整但逻辑错误的决策。单人开发者需要逐条验证每个 CycleDecision 的合理性，而一个 Session 可能产生 20+ 个 cycle。
- 没有"决策质量分数"或"决策自检"机制，系统无法自动识别低质量决策，Debug 完全依赖开发者手动审查 LLM 输出。
- Compact Cycle 试图一次调用完成 5 个子任务（compare / reflect / hypothesis update / ideate / next action），违反"单个 LLM 调用职责单一"的最佳实践。5 合 1 调用中任何一步输出错误，整个 cycle 需要重来。

**需核对/补充：**
- 每一决策步骤是否应设计独立的 schema 验证器（如 hypothesis 不能为空、next action 必须属于枚举集合）
- 设计决策质量的自检机制（如交叉验证：要求 LLM 同时输出 confidence + 引用的 evidence refs，证据不匹配时触发重试）
- 定义 Compact Cycle 失败 N 次后自动降级为 Exploratory Cycle 的规则

### H-TECH-03：AI 生成代码质量管控机制完全缺失

**问题定位：** 全文无任何关于 AI 生成代码的质量管控描述。

- 无代码规范校验规则定义
- 无逻辑复核范围与方法论（开发者审核 AI 代码时重点关注什么）
- 无安全漏洞排查流程（prompt 注入、命令注入、路径穿越——在 Agent 系统中是高频风险）
- 无代码可维护性标准

**潜在风险（单人 + AI 编程助手核心风险）：**
- 本项目的 ExecutorAgent（Plan 03）本身就由 AI 代码生成工具开发，它的核心职责之一是对其他代码做 SEARCH/REPLACE 修改。这意味着：**用 AI 写一个修改代码的 Agent，再用这个 Agent 去修改其他代码。** 这种"AI 套娃"模式下的 bug 排查链路极长——一个 bug 可能源于：(a) AI 编程助手生成的 ExecutorAgent 代码有缺陷，(b) 训练脚本本身有问题，(c) AI Agent 决策错误。三者交织在一起，单人开发者几乎无法高效定位。
- 无代码规范校验 → AI 逐段生成的代码风格不一致、命名混乱，长期维护成本指数级上升
- 无安全漏洞排查流程 → Agent 的 shell 工具可能被 prompt 注入，导致任意命令执行。这是 AI Agent 系统的头号安全风险，文档中无任何安全测试设计
- 单人模式下缺少"同伴代码审查"环节，AI 代码中的逻辑错误只能靠开发者自己逐行审查

**需核对/补充：**
- 建立 AI 代码审查清单：每次 AI 生成代码后开发者必须检查的安全项（命令构造、路径处理、权限校验）
- 定义代码规范工具链（ruff / mypy / pyright）的配置与执行节点
- 针对 ExecutorAgent 设计专项安全测试：prompt 注入尝试修改 forbidden_paths、尝试越权调用 shell
- 建立"AI 生成代码 - 开发者审查 - 自动化测试"的强制流程

### H-TECH-04：框架核心依赖 DeepAgents 无备选方案 — 单人开发的高风险押注

**问题定位：** `01_实验Agents大框架.md` §6.1 CognitiveTaskRunner、§5 Coordinator。

所有 Agent 依赖 `create_deep_agent()`，但：
- DeepAgents 未定义（版本、成熟度、可定制性未知）
- CognitiveTaskRunner 是 Protocol 定义，但未展示 DeepAgents 本身能否实现此 Protocol 的所有行为
- 无任何备选框架评估

**潜在风险（单人开发）：**
- 开发过程中发现 DeepAgents 无法满足需求（例如无法强制 Agent 的 output 匹配 `CycleDecision` schema、无法控制 SummarizationMiddleware 的裁剪粒度），单人开发者需要临时切换框架或在现有框架上加 hack。在单人模式下，这可能导致 2-4 周的进度停滞。
- DeepAgents 作为核心依赖，其框架变更、API 不兼容、长期未维护等风险完全由单人承担。
- 无备选方案意味着：如果 DeepAgents 走不通，整个架构需要重新设计。

**需核对/补充：**
- 明确 DeepAgents 的来源、版本、定制能力验证计划
- 评估至少 1 个备选方案（如直接基于 OpenAI Assistants API / Anthropic Claude API + 自建 Agent loop）的最简可行性
- 为"DeepAgents 不可用"场景设计架构降级路径——例如将 Coordinator 降级为纯确定性调度 + 单次 LLM 调用

### H-TECH-05：Agent 效果评测体系完全缺失（AI Agent 系统的致命缺口）

**问题定位：** 全文无任何可量化的 Agent 效果评测指标。

- 各 Plan 的"检验方案"覆盖了单元测试和集成测试，但全部聚焦于**确定性功能正确性**，而非**Agent 行为质量**
- 无关键指标定义：任务完成率、准确率、幻觉率、平均 LLM 调用步数、Token 消耗、决策正确率、无效尝试率
- 无评测数据集建设计划——系统自称面向"异常检测研究"，但没有定义评测用的基准任务集
- 无回归测试方案——功能迭代后如何保障已有 Agent 能力不退化？

**潜在风险（单人 + AI Agent 至关重要）：**
- AI Agent 系统的核心挑战不是"代码能不能跑"，而是"Agent 的决策质量能不能接受"。没有评测体系，开发者无法知道：
  - 一次代码修改是让 Agent 表现得更好还是更差
  - 当前的 prompt 版本是提升还是降低了决策质量
  - 系统是否真的"收敛"到了更好的实验方案
- 单人开发者面对 10+ 次 prompt 迭代，没有量化指标只能凭"感觉"判断效果——这在大模型时代已被反复证明不可靠
- Plan 06 的 end-to-end 验收标准中 18 条都是功能性的"能不能 X"，没有一条是"X 做得好不好"

**需核对/补充：**
- 定义核心评测指标（至少：task_success_rate、avg_llm_calls_per_cycle、decision_quality_score、invalid_attempt_rate）
- 建立基线评测数据集（至少 5-10 个标准化异常检测实验场景）
- 建设自动化评测脚本（`benchmark/run.py`），保证每次 prompt/代码变更后可以单命令跑全量评测
- 规划回归测试的自动化执行频率（每次 PR 合并前？每晚？）

### H-TECH-06：架构一致性管控缺失 — AI 逐段生成的代码必然架构漂移

**问题定位：** 全文无架构一致性校验机制。

- 8 个功能模块（Session / Environment / Coordinator / IdeaTree / Executor / Job / Validity / Convergence）之间的接口定义未成文
- 分层原则明确，但无架构 compliance test（如"所有 Agent 必须通过 CognitiveTaskRunner 创建"这样的规则无自动化检查）
- 无模块间的契约测试

**潜在风险（单人 + AI 编程助手特有）：**
- AI 编程助手在生成代码时，每次只关注当前 prompt 的上下文。当项目 > 20 个文件时，AI 会频繁产生"新函数调用旧接口"、"重复实现已有功能"、"使用已废弃的 import 路径"等问题。
- 单人开发者同时维护 <session / coordinator / executor / job / evaluation / convergence 六个子系统> + <外部依赖适配器> + <测试> + <部署>，当 AI 代码引入架构不兼容时，单人排查和修复的成本极高。
- 多次修复同一类问题（AI 反复犯同样的架构错误）会严重消耗开发者精力。

**需核对/补充：**
- 建立关键接口的契约测试（例如 `CognitiveTaskRunner` 的 mock 测试确保所有 Agent 创建路径一致）
- 定义架构规则集（如 `arch_rules.yaml`），引入 `pyarchitect` 或基于 AST 的自动化检查
- 在 AI 编程助手的指令模板中固化架构约束：每次生成代码前自动注入当前项目的关键接口定义文件
- 为每个子系统的 `__init__.py` 设计明确的 public API 边界，超出边界的 import 自动告警

### H-TECH-07：单人全链路负载远超可行范围

**问题定位：** 开发计划包含约 30 个 PR，横跨 6 个子系统 + 前置 PR-001A，覆盖：

- 后端架构设计 & 编码
- LLM 集成 & Agent prompt 工程
- GPU 资源管理与调度
- 代码安全检查
- 全套测试（单元 + 集成 + 故障注入 + 端到端）
- 部署 & 运维 & 监控
- 文档 & 知识沉淀

**潜在风险：**
- 单人开发者需要同时掌握：Python 后端开发 / Agent 框架 / LLM 工程 / GPU 运维 / 安全测试 / CI/CD——全栈大师级的技能要求，不现实
- AI 编程助手可以在编码环节提供 2-3x 效率提升，但 Agent 系统的**效果调优**无法被 AI 替代——调 prompt、测决策质量、做回归对比，这些工作需要大量人工干预。按行业经验，Agent 系统的调优周期通常占整体工期的 40-60%。
- 文档计划中将 Agent 调优与功能开发混合在同一 PR 中（如 Plan 02 的 PR 02C Compact Cycle），未单独划分调优时间窗口
- 没有任一项任务标注了"工作量估计"，无法判断 30 个 PR 是否超过单人 1-2 个月的合理负荷

**需核对/补充：**
- 对 30 个 PR 标注工作量估算（即使是 T 恤尺寸 S/M/L/XL），评估总工作量是否匹配单人可用时间
- 将 Agent 效果调优（prompt 迭代、决策质量验证）单独拆分为独立的 PR 或迭代，与功能开发 PR 分离
- 明确开发者当前的技术栈覆盖范围，识别需要前置学习的技术短板
- 考虑项目范围裁剪方案：例如第一版只实现 Compact Cycle（不实现 Exploratory / ReflectionAgent / StrategyDiagnosticAgent）

### H-TECH-08：核心技术难点无预研计划，效果不确定性无兜底

**问题定位：** 文档涉及多个"研究级"技术挑战，但无任何预研（spike）计划：

- LLM 能否稳定输出结构化 CycleDecision？未经实验验证
- SEARCH/REPLACE 四层策略栈在目标仓库的准确率？未经实验验证
- GPU 训练 NaN/Inf 的确定性检测覆盖了多少场景？未经实验验证
- Sentinl 的 error pattern detector chain 对真实 DL 训练日志的分类准确率？未经实验验证

**潜在风险（AI Agent 系统核心风险）：**
- AI Agent 系统的开发流程不同于传统软件开发：先验证"模型能做 X"，再开发"系统支持 X"。本项目顺序颠倒——先设计了完整架构，再假设 LLM 能完成所有认知任务。
- 最典型的失败模式：开发 4 周后发现 LLM 无法稳定输出 CycleDecision，需要重新设计决策逻辑，推倒 3 周的架构代码。
- 单人开发者若在架构完成后才发现模型能力不足，返工成本极高。

**需核对/补充：**
- 在正式开发前安排 1-2 周的技术预研（spike），验证最高风险假设：
  - LLM 能否稳定输出结构化的科研决策（CycleDecision schema 保持率 > 90%）
  - SEARCH/REPLACE 在目标 repo 中的准确率
  - GPU 训练异常的确定性检测召回率
- 为每项高风险假设设定"Go/No-Go"判断标准，预研不通过时启动备选方案
- 将预研结果文档化，作为架构调整的依据

### H-TECH-09：Prompt 工程 / Agent 配置的版本管理与回滚方案缺失

**问题定位：** 全文仅在 Plan 06 §5 提到 prompt overlay 的版本要求，但：

- 不含 prompt 模板本身的版本管理
- 不含 Agent 系统提示词（system prompt）的版本管理
- 不含不同 prompt 版本对应的评测结果的关联记录
- 不含 prompt 变更的 A/B 测试方案

**潜在风险（AI Agent 特有）：**
- AI Agent 系统中最频繁的变更就是 prompt 调优。没有版本管理，单人开发者无法做到：
  - "回滚到上周那个效果更好的 prompt 版本"
  - "这次是改 prompt 还是改代码导致的效果提升/下降"
  - "Session A 用的 prompt 版本和 Session B 不一样"
- 在单人开发模式下，缺乏同伴提醒，prompt 迭代日志极容易被忽略，造成"改了但忘了改了啥"的经典困境

**需核对/补充：**
- 建立 prompt 版本管理方案：每个 prompt 变更 → 记录版本号、变更内容、关联评测结果、变更日期
- 建议方案：prompt 以 YAML 文件形式存放在版本控制中（非 DB），每个系统提示词独立文件，变更走 PR
- 为每个 prompt 版本关联基线评测结果，支持"调 prompt → 跑评测 → 对比结果"的闭环

### M-TECH-01：Agent 工具调用全链路缺少关键环节设计

**问题定位：** Plan 03 §3.3–3.4 定义了 Executor 的 SEARCH/REPLACE 策略和 edit gate，但：

- 函数注册机制：工具如何在 Agent 运行时注册和发现？未定义
- 参数校验：Agent 传递的参数在调用前是否校验 schema？未定义
- 失败重试：工具调用失败后的重试策略（退避 / 幂等 / 最多重试次数）？Plan 03 只在 §3.4.4 提到修复最多 3 次，但这是针对"代码修复"层面的，不是"工具调用"层面的
- 异常兜底：工具调用抛出异常时的统一处理机制？未定义
- 超时处理：每个工具调用的 timeout 设置？未定义

**潜在风险（AI Agent 系统）：**
- 缺少工具调用层的统一 failover 机制，每个工具的异常处理逻辑需要单独实现，单人开发者需要为 10+ 工具逐个编写异常处理
- 参数不校验可能导致 Agent 传递恶意或错误参数——这在 Agent 系统的实际运行中频繁发生

**需核对/补充：**
- 设计统一的 ToolCallHandler：参数校验（pydantic schema）→ 权限验证 → 调用 → 超时/重试 → 异常分类 → 格式化返回
- 明确每个工具的超时阈值和重试策略

### M-TECH-02：多 Agent 协作的竞态条件与死锁风险未考虑

**问题定位：** 未定义多个 Agent 同时操作共享状态时的并发控制。

- Coordinator 读取 Idea Tree 时，IdeaExplorerAgent 可能在写入
- BatchSupervisor（Plan 04 §5.6）处理紧急事件时，Coordinator 可能在处理同一条 event
- 崩溃恢复规则（Plan 02 §4.1）与 Job claim 机制可能同时触发

**潜在风险（AI Agent 系统）：**
- 缺少并发控制时，系统在高负载下会出现数据不一致、状态丢失等问题
- 单人开发环境下，此类竞态条件极难复现和调试（依赖时序的 bug 是最难定位的问题类型之一）

**需核对/补充：**
- 明确共享状态的访问模式（乐观锁 / 悲观锁 / 单线程 event loop）
- Idea Tree 的并发修改使用 revision + compare-and-swap（已在 Plan 02 §3.1 提及，但其他组件未对齐）
- 确定 Coordinator pipeline 是单线程 event loop 还是多线程并发

---

### M-TECH-03：部署架构与运维方案缺失 — 单人运维成本被低估

**问题定位：** 全文无部署架构设计，无运维方案。所有关于 Worker / Job / Sentinel 的设计均假设"系统已经在运行"，未回答：

- 系统部署形态（单机服务？Docker Compose？K8s？）
- 各组件（ExperimentStarter / Worker / Sentinel / Coordinator）的进程模型
- 日志聚合与检索方案
- 告警方案（Sentinel 发现问题后如何通知开发者）
- GPU 训练状态的中心化看板

**潜在风险（单人开发）：**
- 单人开发者从第一天起就需要确定部署架构，否则无法做本地开发和调试。部署架构的决策会影响代码结构（如 Worker 是否使用 subprocess / thread / async）
- 没有运维方案，开发者需要手动 SSH 上服务器排查所有问题。对于偶发的 GPU 死锁、Sentinel 误报，单人排查一次可能消耗半天
- 缺乏 GPU 使用看板，开发者无法快速了解"当前集群 GPU 被谁占用"

**需核对/补充：**
- 明确第一版的部署架构（单机 Docker Compose 可接受）
- 设计日志聚合方案（建议简单的 file-based + `tail -f` 级别，不引入 ELK）
- 设计关键指标的告警方案（至少：GPU 泄漏、Job 失败率过高、Coordinator 连续停滞）

---

### M-TECH-04：Agent 行为退化的回归测试机制缺失

**问题定位：** 各 Plan 的"检验方案"包含单元测试和 fixture 测试，但都是针对**确定性功能**的静态测试。

- 无 Agent 行为退化检测：修改一个 prompt 后，Agent 在 5 个已验证场景中的表现是否不变？
- 无 Agent 决策一致性测试：同样的输入，两次调用 LLM 是否得到合理一致的决策？
- 无非确定性行为的中位数/方差统计

**潜在风险（AI Agent 特有）：**
- LLM 的非确定性会导致：同样的输入在不同时间调用可能得到不同的结果。修改一个 prompt 可能无意中破坏了 3 个已验证的 edge case。
- 没有回归测试，开发者无法在数分钟内验证"这次改的东西是否没搞坏已有的能力"
- 单人开发者每次改 prompt 都需要手动验证 5-10 个场景，一天只能做 2-3 轮迭代

**需核对/补充：**
- 构建回归测试套件：每个已验证的场景→固化输入 + 预期行为描述 + 自动化校验
- 每次 prompt / 配置变更后，单命令运行回归套件并输出差异报告
- 对非确定性场景，至少运行 3 次取多数结果或中位数

---

### M-TECH-05：AI 编程助手服务不可用 / 输出退化的兜底方案缺失

**问题定位：** 全文未提及 AI 编程助手在开发过程中的定位和风险。

**潜在风险：**
- 项目重度依赖 AI 编程助手（如 Codex / Claude Code），但没有定义：
  - 如果 AI 编程助手服务中断（API 故障、网络问题），开发如何继续
  - 如果 AI 编程助手的输出质量下降（模型更新后行为变化），如何降级
  - 哪些代码必须是开发者手写（核心架构 / 安全逻辑），哪些可以交给 AI
- 根据多个团队的经验，Claude Code / Codex 等工具的 output quality 在模型版本更新后可能出现显著波动

**需核对/补充：**
- 明确 AI 编程助手的使用边界：核心架构（Session / Coordinator / Idea Tree / EvaluationContract）的骨架代码建议手写或强校验，CRUD / 工具函数 / 测试代码可委托 AI
- 建立"AI 编程助手不可用"时的手写优先级排序（哪些功能可推迟、哪些必须手写）
- 评估本地模型作为 AI 编程助手的备选方案（如 vllm + Qwen-Coder）

## 低优先级待确认项

### L-TECH-01：CognitiveBudget 的三重限制未匹配单人调试场景

**定位：** Plan 06 §2 的 CognitiveBudget 设计（call_count / cost / step_count 三重限制）。

**待确认：** 单人调试 Agent 时，经常需要临时突破 budget 限制来观察 Agent 行为。当前设计未提供"调试模式"与"生产模式"的 budget 切换机制。建议在 CognitiveBudget 中加入 `allow_override_for_debug` 标志。

---

### L-TECH-02：AttemptStore 的 schema 缺少 `failure_ref` 详细定义

**定位：** `01_实验Agents大框架.md` §4.4 AttemptStore。

`failure_ref` 指向失败详情，但未定义失败详情的数据结构。单人调试时，标准化的失败分类（失败原因枚举 + 上下文 + 可复现步骤）能大幅加速问题定位。建议明确 `FailureDetail` 的 schema。

---

### L-TECH-03：项目文件组织与模块边界风险评估

**定位：** Artifact 目录建议（01 §14）给出了详细的文件结构，但开发期代码的模块组织未定义。

单人 + AI 编程助手场景下，模块组织决定了 AI 生成代码时参考上下文的准确性。文件结构过于扁平（单目录 20+ 文件）会导致 AI 频繁"发明"新的文件路径。建议提前定义模块目录结构并固化到 AI 编程助手的项目上下文中。

---

### L-TECH-04：Agent 行为日志的可读性与可查询性

**定位：** 各 Plan 中 Event / trajectory 数据模型以 JSONL 形式存储。

纯 JSONL 格式在单人调试场景下的查询效率低——开发者需要手动 grep / jq 来过滤特定事件。建议评估是否需要引入简单的结构化查询方案（SQLite 或 DuckDB），或在工具链中加入 log viewer 辅助脚本。

---

### L-TECH-05：热加载 / 热更新机制缺失

**定位：** 未定义开发期如何快速迭代。

Agent 系统开发中，prompt / tool / 配置文件的修改频率极高。如果每次修改需要重启整个系统，单人开发者每天重启 10+ 次，效率极低。建议评估是否支持 prompt 和 Agent 配置的热加载（reload without restart）。

## 核心改进方向总结

**第一优先：建立 AI Agent 系统的基础设施——评测与调试。** 没有评测体系和调试工具链，单人无法在 LLM 的非确定性输出中高效定位问题、验证改进。建议在"第一版边界"中加入：评测基准数据集 + 自动化评测脚本 + Agent 行为日志回放工具。

**第二优先：承认"AI 套娃"的风险，建立代码质量防线。** AI 编程助手生成的 Agent 代码再去修改其他代码——这个递归的可靠性必须是最高优先级的关注点。建议：手动实现 ExecutorAgent 的核心安全逻辑，AI 只协助完成周边辅助代码。对 AI 生成的代码执行强制安全审查流程。

**第三优先：为所有"高估 LLM 能力"的设计点增加确定性保底。** Coordinator 的决策、SEARCH/REPLACE 的准确率、SSHA 防护的可靠性——在这些关键节点上，假设 LLM 会出错，并为错误设计降级路径。

**第四优先：调低项目的"功能范围"，调高"调优周期"。** 30 个 PR 对于单人开发者 + 重度依赖 AI 编程助手来说，保守估计 3-4 个月。建议第一版裁剪到核心闭环（Session → Environment → baseline → Coordinator 单次决策 → Executor → Job → 结果判定），去掉 Exploratory Cycle / ReflectionAgent / StrategyDiagnosticAgent，先跑通再扩展。
