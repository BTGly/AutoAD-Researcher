# 2026-07-19

## Plan 08B — Typed Task Parameters and Local Dataset Intake

### Goal

Carry only user-provided or user-confirmed task parameters into the existing
pending experiment task, preserve a local dataset directory as a distinct
source reference, and retain the existing evaluation/promotion boundaries.

### Operations

- Added `confirmed_task_parameters` to the V2 summary using the existing
  `ConfirmedDecision` provenance contract. It carries baseline, dataset,
  compute budget, primary metrics, and evaluation constraints without parsing
  `confirmed_facts` or inferring values from materials.
- Projected those confirmed values into the existing `InputTask` fields; added
  the already-established `primary_metrics` vocabulary to that intake model.
  Evaluation constraints remain constraints, rather than being reinterpreted
  as authority to alter the evaluator, B_test protocol, or promotion gates.
- Added explicit local-dataset directory registration with configured-root and
  directory checks. Dataset identity stays in `InputTask.dataset`; its server
  location is stored separately as the source registry's `original_reference`.
  Registration neither reads/copies data nor creates a Job, Session, lease, or
  experiment attempt.
- Extended the decision/reply contracts only for explicit local dataset source
  intake and user-confirmed values. A failed dataset registration blocks draft
  preparation for that turn rather than silently omitting the supplied source.
- Closed the confirmation recovery boundary: a confirmed task cannot be
  replayed with a different execution mode, the route uses the persisted mode,
  and the UI offers materialization recovery without showing a new mode picker.

### Result

- Focused Python regressions passed: `/root/.local/bin/uv run --extra dev

  tests/test_ui_sources.py tests/test_experiment_starter.py` — `81 passed`.
- Frontend type-check and production build passed with `npm run build`.
