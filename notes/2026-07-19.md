## Plan 08B — Typed Task Parameters and Local Dataset Intake

### Goal

Carry only user-provided or user-confirmed task parameters into the existing
pending experiment task, preserve a local dataset directory as a distinct
source reference, and retain the existing evaluation/promotion boundaries.

### Operations

- Added `confirmed_task_parameters` to the V2 summary using the existing
  `ConfirmedDecision` provenance contract. It carries baseline, dataset,
  compute budget, primary metrics, and evaluation constraints without parsing
  `confirmed_facts` or inferring values from materials.
- Projected those confirmed values into the existing `InputTask` fields; added
  the already-established `primary_metrics` vocabulary to that intake model.
  Evaluation constraints remain constraints, rather than being reinterpreted
  as authority to alter the evaluator, B_test protocol, or promotion gates.
- Added explicit local-dataset directory registration with configured-root and
  directory checks. Dataset identity stays in `InputTask.dataset`; its server
  location is stored separately as the source registry's `original_reference`.
  Registration neither reads/copies data nor creates a Job, Session, lease, or
  experiment attempt.
- Extended the decision/reply contracts only for explicit local dataset source
  intake and user-confirmed values. A failed dataset registration blocks draft
  preparation for that turn rather than silently omitting the supplied source.
- Closed the confirmation recovery boundary: a confirmed task cannot be
  replayed with a different execution mode, the route uses the persisted mode,
  and the UI offers materialization recovery without showing a new mode picker.

## Plan 08C — Outcome Projection and Schema-Repair Reliability

### Goal

Close the verified V2 reliability gaps without adding a second state machine:
make material-source job failures terminal in the existing registry, give both
schema-bound dialogue calls one bounded diagnostic repair, and use the
configured run root consistently across V2 routes.

### Operations

- Reused the existing `SourceStatus.failed`, `intake_status`, and
  `intake_error` fields to project a source-linked worker failure before its
  job becomes terminal. Successful parsed sources and an already-recorded
  primary failure are preserved; existing parse-error artifacts remain the
  detailed diagnostic record.
- Extracted the established Decision-Agent parse/validate/one-repair flow into
  one internal schema helper and applied it to Reply. The repair prompt carries
  compact Pydantic diagnostics, demands one JSON object, and asks the model to
  preserve the prior semantic answer. Event records contain only output length,
  hash, failure class, and compact field diagnostics, never raw model output.
- Added a small server run-directory resolver backed by the existing core
  `run_id` validation, and migrated V2 HTTP routes to their configured
  `RUNS_ROOT`; the WebSocket route uses the same core resolver because it must
  return a WebSocket close rather than an HTTP exception.
- Added regressions for malformed Reply recovery/fail-closed behavior, unsafe
  archive failure projection to Source, and cross-route configured-root use.

## P0 — Explicit Executable Repository Admission

### Goal

Prevent an acquired reference repository from becoming an experiment target by
default, while retaining support for an explicitly user-selected GitHub or
local repository that has completed acquisition and declares an existing
Executor adapter.

### Operations

- Added a small V2 execution-repository resolver rather than a parallel
  execution framework. It consumes the existing source registry,
  `RepositoryAttestation`, SHA-256 helper, and `ExecutorAdapter.inspect()`.
- Introduced a typed, user-confirmed source role assignment with three roles:
  `reference_only`, `candidate_source_only`, and `executable`. No source kind,
  checkout directory, filename, or adapter manifest implicitly assigns the
  executable role.
- Resolver admission requires exactly one explicitly executable repository,
  an acquired source (`intake_status=ok`), a matching attestation, and a
  supported adapter manifest. It freezes these into a binding containing the
  repository tree fingerprint, attestation hash, and adapter-manifest hash.
- Corrected successful Git acquisition to record `intake_status=ok`, matching
  the existing local-repository acquisition path. Without this, a legitimate
  explicit GitHub execution target could never satisfy the same admission
  contract.
- Added focused coverage for reference exclusion, no implicit promotion,
  unsupported adapters, multiple explicit targets, non-repository role
  rejection, and Git acquisition status projection.

### Design basis

- Arbor persists a confirmed contract before coordinator execution and isolates
  experiments in individual worktrees; it does not treat a discovered checkout
  as run authority.
- DeepAgents keeps approval as a typed human decision at the action boundary.
  The V2 role assignment follows that authority pattern while reusing AutoAD's
  acquisition and adapter evidence instead of adding transcript or keyword
  parsing.

### Result

- Focused tests passed before the full gate:
  `81 passed, 2 skipped` for execution repository, source registry, and
  evidence-route regressions.

### Next

  materializing `input_task.yaml`, and make Session/environment preparation
  consume only that binding.
