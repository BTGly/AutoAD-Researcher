## Entry 1 — Research Chat parse artifact UX alignment

### Goal

修复 Research Chat 在 PDF parse 后的用户可见行为：`paper_artifact_quality != "usable"` 不应在已有 `paper_summary.json` 或 `blocks.jsonl` 等正文内容时直接被标记为失败；LLM 回复上下文应知道真实可用 artifact；GitHub repo source 能力应在自然回复 prompt 中明确；parse 成功/失败结果应进入自然回复通道并保留 transcript 历史。

### Operations

- 阅读并复核 `src/autoad_researcher/ui/research_chat.py` 中 `_execute_or_report_pdf_parse_action`、`_natural_reply_for_decision` 和调用点。
- 阅读并复核 `src/autoad_researcher/assistant/intent_action.py` 中 `ResearchContextSnapshot`、`build_response_context_for_decision`、`evaluate_paper_artifact_quality`。
- 阅读相关测试：`tests/test_assistant_intent_action.py`、`tests/test_natural_reply_for_decision.py`、`tests/test_ui_research_chat_safety.py`、`tests/test_legacy_error_path_hardening.py`。
- 在 `ResearchContextSnapshot` / ResponseContext facts 中加入 `available_artifacts`，从 `paper/artifacts/` 与 `paper/parse/` 递归列出真实文件。
- 新增 `has_readable_paper_artifact_content`：当 `paper_summary.json` 有摘要/方法/贡献等内容，或 `paper/parse` 下 JSON/JSONL/Markdown/Text 文件有非乱码文本时，判定为有可读解析内容。
- 调整 parse 分支：
  - artifacts metadata 不完整但有可读内容时，source 状态更新为 `parsed`，ActionDecision 记录 `executed_success` + `PAPER_ARTIFACTS_PARTIAL_METADATA`，回复标注 metadata 限制。
  - artifacts 既不可用又无可读内容时，仍按失败处理。
  - parse 成功、partial、失败均通过 `_natural_reply_for_decision` 生成自然回复；无 LLM 参数时回落到结构化 fallback。
- 在自然回复 system prompt 中明确：可以登记 GitHub 仓库链接作为 source，后续实验 agents 可以 clone 并分析；这不等于当前聊天会执行 runner、patch 或 benchmark。
- 补充回归测试覆盖：
  - `available_artifacts` 包含 `paper_summary.json`、`blocks.jsonl`、`sections.json`。
  - 可读正文 block 不要求 artifact quality 为 usable。
  - GitHub source/clone 能力提示词存在。
  - parse partial 不把 source 标记为 failed，并走自然回复。
  - parse failure 也走自然回复。

## Entry 2 — Research Chat UX follow-up: readable failed artifacts, transcript window, prompt identity, upload flow

### Goal

继续修复 Research Chat 用户可见体验：旧 run 即使 source 状态为 `failed`，只要已有可读 `paper/parse/blocks.jsonl` 或 `paper/artifacts/paper_summary.json`，也不能继续回答“没有可读正文”；聊天区不能把全部历史铺满页面；开发者信息不应在主页面展示；LLM 需要明确自己的 AutoAD Research Assistant 身份、任务、能力边界和后续审批职责；资料上传主路径应恢复为文件选择按钮，服务器路径仅保留为高级备用。

### Operations

- 在 `ResearchContextSnapshot` 增加：
  - `has_readable_paper_artifact_content`
  - `paper_artifact_content_preview`
- 新增 `build_paper_artifact_content_preview`，从 `paper_summary.json` 和 `blocks.jsonl` 提取紧凑 preview，供 LLM 在 ResponseContext facts 中使用。
- 调整 intent decision：
  - 当 source registry 状态为 `failed` 但已有可读 paper artifacts，且用户要求读论文时，返回 `summarize_parsed_artifacts` + `parsed_artifact_insufficient`，不再返回 `parsing_failed_status`。
  - deterministic fallback 在 partial metadata 且可读内容存在时，不再说“不能基于论文正文”。
- 去掉 `_execute_or_report_pdf_parse_action` 内部的 `st.warning/st.error/st.success/st.chat_message` 重复输出；由外层统一写 chat message，避免同一回复显示两遍。
- 新增 `BASE_RESEARCH_ASSISTANT_PROMPT`，并注入普通聊天与 parse 自然回复，明确：
  - 助手身份是 AutoAD Research Assistant。
  - 任务是读清论文/资料/仓库/run artifacts，整理研究目标、证据边界和 pipeline 输入。
  - 后续 pipeline 到人工确认阶段时在这里请求审批。
  - 系统有 source 登记、web_search/web_fetch/git_clone 等后续 agent/tool 能力边界，不能把工具说成不存在。
- 聊天 UI 改为默认只显示最近 8 条，较早历史进入折叠区。
- 移除 `render_research_chat()` 主页面中的开发者信息渲染调用；纯 helper 仍保留给测试/调试。
- Sources 面板恢复主文件上传体验：
  - `st.file_uploader("添加资料", accept_multiple_files=True)` + “添加到当前任务”按钮。
  - 服务器本地路径入口折叠到“高级：从服务器路径添加”。
  - 底部 `st.chat_input` 新版本下启用 `accept_file="multiple"`；当前 Streamlit 1.38.0 不支持该参数，因此增加签名兼容包装，旧版本不传不支持参数。
- 补充回归测试覆盖 readable failed artifacts、prompt identity/tool boundary、transcript visible tail、upload button priority、developer info 不在主页面渲染、chat_input 文件参数兼容。
